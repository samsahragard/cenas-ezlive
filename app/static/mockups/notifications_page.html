Page-first design notes (per Sam #2531):
  • Source data is the existing ribbon data path. ribbon_render_context() in app/web/ribbon_routes.py already returns the 7 fixed-order categories with per-item entries (text, sub_text, severity styling_class, can_dismiss, can_check). New notifications_page() view passes the same context to a new template; no data shape change.
  • Tabs use the V2 .lg-nav pill pattern Sam locked in at #2485 — pure white inactive text on transparent + gold-soft border, dark text on gold-bright pill for active. Same shape as Drivers v3 tabs. Counts inside the pill; counts with alert items get red tint so the eye lands.
  • Cards use V2 glass primitive (rgba(40,22,14,0.55) bg, 0.5px gold-soft border, 8px radius) with a 3px left border in the severity color. Border-left = severity is a standard alerts/log convention; aligns with the existing ribbon styling tokens.
  • Severity tokens identical to ribbon: --ck-status-warn (alert/red) → --ck-status-active (warn/gold) → --ck-status-open (info/green). Zero retraining for operators.
  • Action buttons (X / check) mirror the existing ribbon's per-item controls. X is universal; check renders only when the item has can_check (To-do items, mostly). Same controls, new chrome.
  • Sidebar entry under Dashboard in the Today section. Bell icon (ti ti-bell) + red unread-count badge so the operator sees pending notifications at a glance from any page. Same role-gating as Dashboard (no driver/expo restrictions in v1; matches the ribbon's audience).
  • Empty state per tab with ti ti-bell-off + "No notifications in <Category> · you're clear here." Friendly affordance.
Implementation notes for ck:
  • Route: /<store_slug>/notifications as the store-scoped version (matches the per-store data model); plus /partner/notifications for partner-tier rollup if needed. v1 = store-scoped only.
  • View: notifications_page() in app/web/store_routes.py (or new app/web/notifications.py module). Reuse ribbon_render_context() as the data source; just render into the new template.
  • Tab filter: client-side first (all categories in DOM, JS toggles a .is-active class on the matching .notif-cat-pane). If categories grow large enough that this slows page load, switch to server-side ?cat=todo querystring + URL-shareable tab state.
  • Dismiss endpoint: reuse existing POST /partner/ribbon/dismiss/<item_id> (ribbon already has this per Block 1D). No new endpoint.
  • Sidebar badge count: total non-dismissed notifications, fetched on page load + refreshed on dismiss. v1 = no live updates (page reload refreshes); v2 = SSE or poll if Sam wants real-time.
  • Ribbon removal — two commits, behavior-parity gate between them (per cena #2624):
    • Commit 1: ship Notifications page (route + view + template + sidebar entry + base_dashboard.html still includes _ribbon.html). Notifications and ribbon coexist briefly.
    • Behavior-parity gate (Sam on live, not a smoke check): Sam uses the Notifications page on production for at least one real operational beat — an actual alert lands in a category tab, Sam dismisses it via the X, the unread sidebar badge decrements. The ribbon has been the single delivery channel for operational signals; do not pull it until the replacement has earned trust under real load.
    • Commit 2 (separate PR/commit, after gate clears): remove {% include 'partials/_ribbon.html' %} from base_dashboard.html, delete partials/_ribbon.html, retire ribbon.css + ribbon.js, remove ribbon-related routes/context from app/web/ribbon_routes.py (keep ribbon_render_context() if Notifications still reuses it — rename if cleaner).
    • Critical: "while we're in there" ribbon removal in the Notifications-ship commit is the exact failure mode that takes the only delivery channel dark mid-deploy. ck + aick — strictly two commits.
Mobile responsive (v1 expectations):
  • Tabs row uses overflow-x: auto — 7 tabs scroll horizontally on narrow widths.
  • Cards stay full-width single-column.
  • Page sits below the hamburger + page-title in the existing topbar (no special mobile treatment needed).
Accessibility:
  • Tabs are <button> with role="tab" + aria-selected; the list below is role="tabpanel". Keyboard arrow-left/right cycles tabs.
  • Severity color is reinforced by the icon (ti-alert-triangle / ti-flag / ti-circle-check) so color isn't the sole signal.
  • X/check buttons have aria-label and 28×28 touch target. Larger on mobile if testing shows it (44×44 a11y minimum).
