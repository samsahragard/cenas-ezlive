Page-first design notes (per Sam #2531 / charter §4):
  • Source page: app/templates/produce/order_guide.html + app/static/produce/styles.css. Current page renders a 2-column .item-grid at ≥900px (two .item-blocks each with their own 6-col table: Item / Date / Size / Price / Vendor / Qty). At ≤900px the grid collapses to 1 column; the table still tries to fit 6 columns side-by-side. At ~440px the QTY column gets clipped off the viewport's right edge — Sam's reported failure mode.
  • Sticky cart already exists. .sticky-cart with J. Luna + Alvarado + Total trackers + Place Order is already in production (lines 38-55 of order_guide.html, styled at .sticky-cart in styles.css). Above 900px it sits at top: 12px. Above 500px it stays flex-row; below 500px it goes column. Reuse this DOM — just re-pin to bottom + gate visibility on cart-has-items + restyle as a compact summary on narrow.
  • QTY input already has inputmode="numeric". No change needed for keypad behavior. What's new: wrap it in stepper buttons (44×44pt tap target each side).
  • Markers (★ / ○) preserved. Star = "what we want", circle = "if price is better" — same semantics as production.
  • Vendor badges preserved. .badge-jluna / .badge-alvarado color/copy unchanged. Card layout shows badge inline in meta row.
  • Per-row line subtotal is NEW. Currently only the cart totals display computed sums; the card shows $qty × $price per row. Spec asks for it — adding it. Above 640px the table doesn't add a Subtotal column (keeps existing 6-col shape); subtotal lives only in the cards.
Implementation notes for ck:
  • Breakpoint additions to styles.css:
    • New @media (max-width: 640px) block: hide .item-table-wrap, render .pcard list instead. (Or single template that conditionally renders card vs table — see template strategy below.)
    • Existing 900px and 500px breakpoints stay but become redundant for narrow widths (cards take over below 640px). Don't remove them — they're still relevant at the 640-900px band (single-column table) and inside the sticky-cart layout collapse.
  • Template strategy — single source, two visual modes:
    • Recommended: render BOTH card list and table in the template (CSS toggles visibility per breakpoint). Both consume the same {% for it in col_items %} data — no Python branching. Data-binding from a single source means no risk of "cards show qty 3, table shows qty 0" desync.
    • Alternative: keep table-only at ≥640, render cards-only at <640 via JS-driven swap on resize. More fragile.
    • The existing 2-column .item-grid split (left_items / right_items) only matters above ~900px. Below 640 the card list is a single column of all items.
  • Stepper component:
    • .stepper wraps two buttons + the existing .qty-input. Buttons fire refreshCart() (existing JS) on click after incrementing the input value.
    • Tap targets: 44×44px minimum per spec (a11y minimum, also matches WCAG 2.5.5).
    • − button disables when value = 0; + button stays enabled (no max for now — production input has max="999", keep that).
    • Long-press +/− to increment by 10 (spec marks this OPTIONAL — defer unless Sam asks).
  • Sticky bottom bar (narrow widths only):
    • The existing .sticky-cart moves from position: sticky; top: 12px to position: sticky; bottom: 0 below 640px (a sticky bottom inside the scroll container works for in-page scroll; if container scroll vs body scroll causes issues, fall back to position: fixed; bottom: 0; left: 0; right: 0).
    • Gate visibility on cart.length > 0 in the existing refreshCart() JS — add/remove .is-hidden class on the sticky element. Currently the cart is always visible but the Place Order button is disabled when empty; on narrow we want the whole bar to disappear when empty so it doesn't eat 80px of scroll real estate.
    • Layout collapses from "row of full trackers + Place Order" to "row of compact chips above + full-width Place Order below."
  • Above-640 table fixes:
    • Add .item-table-wrap { overflow-x: auto; -webkit-overflow-scrolling: touch; } — currently has overflow: hidden which is the actual cause of the clip. Changing to auto means scrollable rather than clipped.
    • Sticky ITEM column: .item-table th:first-child, .item-table td:first-child { position: sticky; left: 0; z-index: 2; background: ... }. Background must match row state (resting + hover) — see mockup CSS for the two-bg-color approach.
    • Right-edge gradient fade: .item-table-wrap::after pseudo-element with linear-gradient. Pure CSS, no JS.
    • QTY column min-width: 120px (currently 80-90px). The 64px .qty-input + padding fits comfortably at 120px without clipping.
  • Viewport meta tag: base.html for the produce page ({% extends "base_dashboard.html" %}) needs <meta name="viewport" content="width=device-width, initial-scale=1.0">. Spec says "confirm — Fold devices can render as desktop without it." Quick grep before implementing: if missing, this is the load-bearing fix; if present, no change.
Accessibility:
  • Stepper buttons have aria-label="Decrease quantity" / "Increase quantity".
  • Qty input has aria-label="Quantity" (in the stepper) / visible label (in the table column header).
  • 44×44pt stepper tap target meets WCAG 2.5.5 minimum.
  • Sticky bottom bar uses role="region" aria-label="Order summary" when ck wires it.
  • Stepper input is type="text" + inputmode="numeric" + pattern="[0-9]*" per spec. Production currently uses type="number" + inputmode="numeric". type="text" avoids iOS Safari quirks where number inputs lose the numeric keypad in some PWA contexts — but if production hasn't seen that issue, keep type="number" and only the stepper UI changes. ck's call based on telemetry / test results.
