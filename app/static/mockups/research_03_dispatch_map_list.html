Source pattern

DoorDash Drive Portal. The portal pairs a live map of active deliveries with a sortable list of order tiles, each tile showing Dasher name, ETA, and current state (assigned / en route to pickup / picked up / en route to drop / arrived). The dual map+list layout means dispatch can scan visually or scan textually depending on need.

Why it works

  • Two scanning modalities, one source of truth. Some operators are visual ("where are my drivers right now") and some are list-oriented ("who's late, in order"). Same data, two surfaces — no mode switching cognitive cost.
  • Map = situational awareness; list = action queue. Map shows clustering, route overlap, dead zones. List shows priorities, urgency, what-to-do-next. Operators flip between them constantly.
  • Single status pill vocabulary. Pickup / enroute / arrived / late / idle — same labels on the tile AND the map pin color. Reduces "wait, what does yellow mean again" friction.

How it adapts to Cenas

  • Primary surface: /dos/drivers + /uno/drivers admin pages. Today these are list-only. Add a map pane on the left (desktop ≥768px) showing active drivers + their destinations relative to the store.
  • Map data source: driver GPS pings (if you have them) OR last-known-from-route-API geocode. For v1, even a static map with pins placed by destination address would beat no map at all.
  • Status vocabulary. Five states match our driver workflow: pickup-en-route / drop-en-route / arrived / late / idle. Token bindings: --ck-gold (pickup) / --ck-gold-bright (drop) / --ck-status-open (arrived) / --ck-status-warn (late) / --ck-text-faint (idle).
  • Mobile collapse: default to LIST, "Map" toggle in header. 440px phone can't show both meaningfully; managers can opt-in to map mode for triage. Don't waste vertical real estate by default.
  • Single-driver multi-order: if a driver has multiple deliveries pinned to same route, stack the tiles + show a single pin on the map with the order count badge.

Implementation cost

The map is the load-bearing piece — pick a library (Leaflet + OSM tiles is the cheap path; Mapbox GL JS is the rich path) and budget a real chunk of time for it. The list view + status pill vocabulary is the easy half (~1 day). Map integration: ~3-5 days depending on whether you have driver GPS or have to geocode destinations. Mobile drawer/toggle UX: ~1 day.
