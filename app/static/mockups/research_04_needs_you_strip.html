Source pattern

7shifts. Time-off, shift-pool, and availability requests appear as count badges at the very top of the manager home — selecting one jumps straight to the approval screen for that type. Approvals never get lost in a generic inbox; each chip is typed by what kind of action it represents and clicking it deep-links to that exact surface.

Why it works

  • Typed > generic. Generic "12 notifications" buries the urgent inside the routine. Typed chips ("Driver no-show: 1 / EzCater unconfirmed: 3") tell the operator exactly what kind of attention is needed, so they triage at a glance.
  • Top of fold. Most operators open the dashboard, glance at the top, and react. The strip is the first thing they see. KPIs are secondary — what needs ACTION beats what shows STATUS.
  • Click = action, not list. Each chip deep-links to the screen where the operator can dispatch the action immediately. No "view all" → "select item" → "click action" three-step funnel.
  • Auto-hides when empty. When count drops to 0 the strip collapses to a quiet "all caught up" or hides entirely. Dashboard isn't permanently apologizing for nothing being wrong.

How it adapts to Cenas

  • Primary surface: per-store dashboard home (/dos/ + /uno/). Strip sits above the KPI grid, below the page title.
  • Chip types for Cenas v1:
    • Driver no-show / late-acknowledge (deep-link → /dos/drivers with filter)
    • EzCater unconfirmed orders (deep-link → /dos/orders/<date> with status=pending)
    • Produce order awaiting send (deep-link → /produce/ with cart loaded)
    • Vendor invoice flagged (deep-link → flagged invoices)
    • (future) Maintenance ticket open / Catering bid awaiting response / etc.
  • Distinct from /partner/notifications. Notifications page is the inbox of EVERYTHING; the strip is the JUST-NEEDS-ACTION subset hoisted to dashboard. Notifications has 30 items; the strip has at most 4-6.
  • Urgent variant. Driver no-show chip uses border-color: status-warn + label color: status-warn — escalates visually because it's time-critical. EzCater unconfirmed is gold (warm but not panic).
  • Empty state is a feature. The reassuring "all caught up" message becomes a small but real morale boost when an operator clears their queue. Don't skip it.

Implementation cost

Backend: define an "operator-action-queue" aggregator query that hits 4-6 tables (driver_runs, ezcater_orders, produce_carts, vendor_invoices) and returns typed counts. ~half day. Frontend: ~half day for the strip component + per-chip deep-link routes. Refresh: polls every 30-60s OR taps existing notification poll if /partner/notifications event stream is wired (which it now is via samples_watch.py + the approval-events endpoint pattern).
