Source pattern

Olo Dashboard. Olo's value proposition is "stop juggling tablets" — they aggregate every digital order channel into one list with a filter chip row at the top to narrow by source. Filter state persists per session so operators don't re-pick on every page load.

Why it works

  • One pane > many panes. Operators stop context-switching between EzCater portal / DoorDash portal / phone-notepad / Direct-CRM. Everything lives in one list.
  • Per-channel color identity. EzCater orange / Direct green / Phone gold / DoorDash red / UberEats green — operators recognize source at a glance without reading the channel name.
  • Forward-compatible by construction. Adding a new channel (e.g. a future Cenas direct-catering web form) is one row in a channel-config table + a new chip. No restructuring of the order pipeline.
  • Filter persists per session. Operator who lives in EzCater filter all morning doesn't get reset on page reload. localStorage or query-string persist.

How it adapts to Cenas

  • Primary surface: /dos/orders + /uno/orders catering pipeline pages. Today these are probably EzCater-only. Build the page as if multiple channels exist even if only EzCater is active right now — one channel works fine + you avoid a rewrite when direct/phone channels are added.
  • Channel taxonomy for Cenas v1: EzCater (live) / Direct (web form, future) / Phone (manager-entered, today probably an off-system notepad — formalize) / DoorDash (if you add) / UberEats (if you add).
  • Color palette per channel: use the actual brand colors of the source service (#FFAA33 EzCater, #FF3B3B DoorDash, #00C868 UberEats). For internal-source channels (Direct, Phone) use semantic tokens — green for direct (no platform fee), gold for phone (warmth).
  • Filter state persistence: URL query string (?source=ezcater) is preferred over localStorage — shareable, bookmarkable, survives device switches.
  • "All" chip is the default + carries the total. Don't make operators click a chip to see anything; show everything by default.
  • Zero-count chips dim, don't hide. Empty DoorDash chip stays visible at 40% opacity so operator knows the channel exists + sees if it activates. Hiding zero-counts breaks discoverability.

Implementation cost

Backend: add a channel column to the orders table (default 'ezcater' for existing rows). ~10 min migration. Frontend: chip row component (~30 lines), URL query param wiring (~10 lines), per-channel color dot (~5 lines CSS). Total: ~2-4 hours. Trivial cost, large future-proofing value.
