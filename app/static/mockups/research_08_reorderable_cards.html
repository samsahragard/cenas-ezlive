Source pattern

Square for Restaurants. Square's POS layout editor lets staff drag tiles to reorder and the grid auto-sizes to fit. Principle: layout is owned by the person using it, not the person who built it. Tomball's GM ranks "driver runs" first because that's where their attention goes; Copperfield's BOH lead ranks "produce" first for the same reason. Same app, different default views per role.

Why it works

  • Roles imply different priorities. A GM cares about covers + tickets + drivers. A BOH lead cares about produce + inventory + tomorrow's prep. A partner cares about both-stores aggregate + week-over-week trends. One default ordering can't serve all three well.
  • Edit-mode toggle keeps the regular dashboard clean. Reorder controls only appear when the user is actively editing. Otherwise the cards just render — no grip handles, no arrows, no chrome noise.
  • Up/down arrows beat drag-drop on cost. Real drag-drop has touch-event quirks, accessibility implications, edge cases (dragging past viewport, etc.). Up/down arrows are 5 lines of JS, work on every device, accessible by keyboard. 80% of the value at 10% of the cost.
  • Hide ≠ delete. Hidden cards land in a "tap to re-add" section. Discoverability preserved; the manager just removed them from their default scroll.

How it adapts to Cenas

  • Primary surface: per-store dashboard. Each card on the dashboard becomes one item in a per-user ordered list. Order stored in user_dashboard_layout table (user_id, store_slug, card_slug, position, is_hidden).
  • Default order ships sensibly for the GM role. Most managers will never touch it — that's fine. The reordering serves the ~20% who care to customize.
  • Cards inventoryable. Each renderable dashboard component declares itself as a card with {slug, default_position, role_visibility}. Adding a new card to the app = adding a row to the inventory, not editing every user's layout.
  • Persistence model: localStorage for instant feel + server-sync on edit-mode exit. Survives browser switches once synced.
  • Don't ship full drag-drop yet. Up/down arrows + hide/show + reset-to-default button gets you the win without the engineering investment. Drag-drop is a future polish item if reorder usage proves significant.
  • "Reset to default" escape hatch. If a manager creates a layout they regret, one button restores the role's default. Cheap to ship, prevents support tickets.

Implementation cost

Backend: small new table + 2 endpoints (get layout, save layout). ~half day. Frontend: refactor dashboard render to iterate over a user-ordered card list rather than hardcoded sections (~1 day if dashboard is component-based today, ~2 days if it's a single template), edit-mode toggle + arrows + hide chips (~half day). Total: ~2-3 days. Defer drag-drop entirely; arrows + hide cover 90% of the value.
