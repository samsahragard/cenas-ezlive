Source pattern

ChowNow Business Manager + general restaurant-ops convention. ChowNow specifically calls out dark mode + text size as top-level preferences (not buried under "Advanced → Accessibility → ..."). The principle: display preferences are working conditions in restaurant ops, not aesthetic preferences. A line cook on the 5pm rush trying to read an order on a flour-dusted phone in dim kitchen lighting isn't going to dig through 3 menus to find text size.

Why it works

  • Operators use phones in extreme conditions. Walk-ins (dim, cold). Patios (bright sun glare). Kitchens (wet, dusty hands). Late-shift fatigue (smaller text = eye strain). Display flexibility isn't accessibility — it's basic ergonomic respect.
  • Three density tiers > continuous scale. Compact / Regular / Large covers the use cases (max info density / balanced / max readability) without offering 47 knobs. Decision fatigue stays low.
  • Density scales SPACING + FONT together. Just changing font-size leaves cramped padding; just changing padding leaves tiny text. Single density scalar drives both via CSS var arithmetic.
  • System theme respect. If the OS is set to light mode and the user hasn't explicitly picked dark in the app, honor the OS. Don't override unless explicitly requested.
  • Reduce motion + bigger tap targets are separate. Some users want low-density + bigger taps (small text but easy tap). Decoupled controls let them mix.

How it adapts to Cenas

  • Add a Display section to user settings. Three controls v1: Density (Compact/Regular/Large), High contrast (on/off), Bigger tap targets (on/off). Reduce motion can come later. Light theme is honest "coming soon" until you build the V2 light tokens.
  • Density via CSS variable: --ck-density: 1 at :root, all type/spacing tokens derive: --ck-text-size-md: calc(13px * var(--ck-density)). User preference flips the var; entire app rescales without touching component CSS.
  • Persistence: server-side per-user (so it follows the user across devices). localStorage as the fast-cache.
  • Highest-impact surfaces for density: orders list, drivers list, produce order page, notifications page. KPI cards already breathe; they don't need to shrink.
  • Bigger-tap-targets toggle = 48px minimum on every button + 24px+ spacing between tap zones. Independent of density (a manager might want Compact rows + Large tap targets).
  • High contrast toggle: WCAG AAA on text/background. Boost --ck-text to pure white, deepen --ck-bg-page. Some accent shifts. ~30 minutes of work to add the toggle once tokens are set up.

Implementation cost

Refactor to density-scaled tokens: ~2-3 days IF the app already uses tokens consistently. If not, longer — but the V2 token system you have is most of the prep. Settings UI: ~1 day for the panel + persistence + per-user storage. Live preview in settings: ~half day. Total: ~3-5 days for the full pattern. Phased shippable: Density first (highest impact), High Contrast + Tap Targets second, Motion later.
