Source pattern

Toast KDS. Toast's KDS briefly flashes ticket headers when a new ticket arrives or an existing ticket is modified. The flash is pre-attentive — kitchen doesn't have to be looking at the screen to register that something changed; the motion catches peripheral vision. Combined with the persistent ticket-age color (research #2), it's a two-channel attention system: color says "this ticket needs to move soon," flash says "this ticket just changed (look at it)."

Why it works

  • Pre-attentive motion processing. The human visual system detects movement faster than it parses text or color. A 1.4s flash on a previously-static page is impossible to miss with peripheral vision.
  • One-shot, not loop. The flash plays ONCE on the triggering event, then settles. A looping animation would become wallpaper after 5 seconds — operators tune it out.
  • Three flash flavors, three meanings. NEW (gold burst + scale-up) / CHANGED (gold soft) / URGENT (red burst + scale-up). Operators learn the semantics quickly: gold = something new arrived, red = something needs immediate attention.
  • Tied to data events, not to UI events. The flash fires when a row enters/changes in the data — not when the operator clicks something. Operator-driven changes don't need to flash because the operator already knows about them.

How it adapts to Cenas

  • Primary surfaces:
    • EzCater pipeline cards — flash NEW on order-arrival, flash CHANGED on order-modify, flash URGENT when an order crosses the late threshold
    • Driver dispatch tiles — flash NEW on dispatch, flash CHANGED on status-update (en-route→arrived), flash URGENT when ETA slips past late
    • KPI cards on the dashboard — flash CHANGED when the number updates from the live poll (the demo button above demos this)
    • Notifications page cards — flash NEW on incoming notification (gold burst draws eye to the new card without needing a separate "new!" badge)
  • Trigger source: the polling layer. chat_tail / samples_watch / future order-watch already poll for new state. When a poll surfaces a delta, add an is-new / is-changed / is-urgent class to the affected DOM node for 1.5s, then JS removes it. CSS handles the animation.
  • Don't flash on initial page load. Only flash when something changes WHILE the user is viewing. Initial render with 14 EzCater orders doesn't need 14 simultaneous flashes — that's noise, not signal.
  • Respect "Reduce motion" preference. If research #9's reduce-motion toggle is on, swap the keyframe for a subtler background-color-only fade (no scale, no glow). The signal stays; the motion goes.
  • Coexists with ticket-age color (research #2). A late ticket can flash URGENT once when it crosses the threshold, then settle into the persistent red border. Two-channel attention.

Implementation cost

CSS: ~30 lines (3 keyframes + 3 modifier classes). JS: ~10 lines per polling layer to detect delta + set/remove class. Total: ~half day across all surfaces. Cheap pattern, high signal-to-cost ratio.
