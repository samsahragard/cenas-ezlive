Page-first build per Sam #2531:
  • Visual vocabulary copied from existing sidebar.html partial (the V2 .ck-sb-* family). Right-side bar uses the same V2 --ck-* tokens, the same Tabler icon vocabulary, the same gold uppercase section labels with cream-glow text-shadow, the same hover-row treatment. No new design language invented — the right bar reads as a sibling of the left bar, not a foreign element.
  • Sourced from plan.md §5 (origin/main 9e34176). Each section header references its §5 subsection so a reader can trace any item back to the plan.
  • Per-section completion counts in the header (X/Y). At-a-glance scan: where the gold-green numbers diverge from total, that's the work remaining. Block 2 (0/9) + Block 3 (0/2) + the "soon" line items in Operations (13/16) + Forecasts (3/4) are the live "still need to do" list.
  • Status legend at top. Live (gold-green dot), partial/soon (gold dot), planned (faint cream dot). Item text dims for soon/planned states so the eye lands first on what's live.
  • "new" pills on items shipped in the last ~24h (Drivers admin Phase A, Samples page, Build Plan viewer). Drops to nothing after a few sessions.
Structural decisions worth flagging:
  • Width 260px — slightly wider than the left sidebar's 220px because the per-item status dot + soon-pill add horizontal weight. Could squeeze to 240px if main content needs the room.
  • Sticky scroll independent of main content — vertical position scrolls within the right bar itself, not the page. Otherwise the bar disappears when the user scrolls deep on a long page.
  • This is the FULL page list, not role-gated. A real implementation should role-gate (drivers don't need to see "Manager Tools" todo) — but for a build-status mockup, showing everything to everyone is the right call.
  • Right bar lives at desktop only. On mobile (≤1024px per --ck-bp-mobile) it collapses to a "Build progress" entry in the existing hamburger menu — adding a second sidebar on phone would crowd the surface. Defer the mobile pattern to a Phase B mockup if Sam wants it.
  • Two-sidebar layout has a tradeoff. The main content column shrinks. On 1440px desktop you get ~920px main column — fine for most surfaces but tight for wide tables (orders list, sales reports). Could be a toggle ("show build progress") rather than always-on, if main-column real estate matters more than always-visible status.
Status data freshness:
  • v1 is hand-coded from my knowledge of what's live as of 2026-05-17. For a production implementation, the status per route should be derived from a registry — e.g., scan DOC_PAGES + the route registrations + a small FEATURE_STATUS map per route — so the right bar updates automatically as new pages ship.
  • "new" pill would come from a per-route shipped_at field, with a 7-day window.
  • Counts in section headers would compute server-side from the same status registry.
