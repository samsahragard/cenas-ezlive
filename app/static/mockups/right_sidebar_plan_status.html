v2 design decisions (page-first per Sam #2531 + cena #2557 reshape):
  • Source of truth: plan.md §9 phases drive the structure. Per-phase items pulled from §5 surface inventory + §6 features + §7 Block 3. Standing "Design system" track separated per §9 framing.
  • Visual vocabulary copied from existing sidebar.html partial (V2 .ck-sb-* family + --ck-* tokens). New .ck-roadmap-* classes mirror the same idioms — gold accent on current state, status-open green for done, cream-glow on headers, Tabler icons.
  • Phase number badges work as visual anchors — at-a-glance you see how many phases exist + which are done (green) vs current (gold-glow) vs planned (faint).
  • Progress bars per phase at 4px height. Subtle, but reads instantly. Gradient on current-phase bar (green→gold-bright) signals "actively in progress."
  • Overall progress in header = single % across all phases. v1 had per-section counts; v2 surfaces a roll-up so operators know the global state without scanning.
  • Standing design system track rendered distinct (left gold accent border + warm gradient bg) since it doesn't slot into a single phase per plan.md §9 ("parallel to all phases").
v1 vs v2 — what changed and why:
  • v1 was a navigation menu with status icons (grouped by §5 audience tier). Read as "where can I go" first, "what's done" second.
  • v2 is a roadmap visualization (grouped by §9 build phase with progress bars). Reads as "where the product stands" first, "what's where" second.
  • Cena #2557 directed this reshape. Sam wants an at-a-glance build state, not a second nav menu.
Implementation notes for ck when Sam approves direction:
  • Phase + per-item status data: currently hand-coded. Production should derive from a registry — e.g., one entry per plan.md surface mapping to {phase, status, route, shipped_at}. The Build Plan viewer (/partner/developer/plan) could double as the source-of-truth surface if it parses plan.md headings into a structured map.
  • Overall % computation: simple — count items built across all phases, divide by total. Weighted-by-phase is possible but adds complexity for marginal benefit.
  • Auto-update on ship: when a route registers in DOC_PAGES or SAMPLES, the registry entry flips to built. "new" pill via 7-day window on shipped_at.
  • Mobile: top-right small icon (40×40) with completion badge ("36%") opens roadmap as right-slide drawer. Same pattern I proposed for v1 — applies cleanly here.
  • Wide-table routes: right-bar collapses to icon-rail showing only phase number badges + their bars (a vertical mini-timeline). One click expands a phase inline. Same toggle as the left bar's collapsed state.
Accuracy caveat: per-item built/in-progress/coming markers and the % numbers in v1 are best-effort estimates from what I've seen tonight. Production registry-derived numbers will differ; the mockup demonstrates the SHAPE, not the exact state. Verify-before-promote when ck wires it.
