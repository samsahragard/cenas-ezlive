Production integration spec (per cena #2693)

Class convention: .ck-rsb-* namespace (parallel to .ck-sb-* for left sidebar). The internal class family from v2's .ck-roadmap-* is renamed to .ck-rsb-* verbatim — same battle-tested CSS, just namespaced.

1 · base_dashboard.html integration

Mount the partial in base_dashboard.html as a sibling to the existing left sidebar + main content:

<div class="ck-app-shell">
  {% include 'partials/_left_sidebar.html' %}     {# existing #}
  <main class="ck-main">
    {% include 'partials/_topbar.html' %}         {# add .rsb-chip on mobile here #}
    {% block content %}{% endblock %}
  </main>
  {% include 'partials/_right_sidebar.html' %}    {# NEW #}
</div>

Layout: CSS Grid grid-template-columns: 264px 1fr 280px on .ck-app-shell at desktop. Below --ck-bp-mobile (1024px), drop to 1fr single-column and the right sidebar becomes a fixed-position drawer.

2 · Mobile drawer toggle

Topbar gets a .rsb-chip button (gold-soft bg + gold-bright border, parallels the left hamburger affordance). Tap fires:

document.querySelector('.ck-rsb-drawer').classList.toggle('is-open');
document.querySelector('.ck-rsb-backdrop').classList.toggle('is-open');

Backdrop click closes. ESC key closes. Drawer width: 320px max-width: 86vw — never edge-to-edge so a tap-strip remains for closing.

3 · Collapse persistence — reuse /partner/ribbon/collapse/<slug> (cena #2693)

Per-phase collapse state persists per-user. Reuse the existing endpoint with namespaced slugs:

  • Phase 1: POST /partner/ribbon/collapse/rsb_phase_1
  • Phase 2: rsb_phase_2 · etc.
  • The right-sidebar-itself open/closed (mobile drawer): rsb_drawer_open (ephemeral; don't persist drawer state across sessions — restart fresh-collapsed each load)

The ribbon_collapse endpoint handler doesn't care about the slug prefix; the rsb_* namespace just keeps it cleanly grep-able. No new endpoint, no new table, no new migration.

4 · Data source: parse plan.md at request time (cena #2693)

Single source of truth: plan.md at repo root. View function reads + parses on each render:

def right_sidebar_context():
    phases = parse_plan_md(Path('plan.md'))  # new helper
    return {
        'phases': phases,
        'overall_pct': sum(p.pct for p in phases) / len(phases),
        'standing': parse_standing_track(Path('plan.md')),
    }

No pre-built JSON file to keep in sync. plan.md edits are immediately reflected. Partner-tier pages have low traffic, parsing cost is negligible. If perf becomes an issue later, cache to g._rsb_context per-request (already in-process for that view).

5 · Audience eligibility

Right-sidebar renders only for partner-tier authenticated users (same scope as the existing samples + plan-doc surfaces). Driver-tier + manager-tier users don't see it — they don't read plan.md and shouldn't see internal roadmap state. Gate via {% if g.current_user.tier == 'partner' %} in base_dashboard.html.
