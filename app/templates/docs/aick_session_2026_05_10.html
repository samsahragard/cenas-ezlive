Driver Phase A — self-signup + login + admin (commit 5a30e36)

Sam wanted drivers self-registering from the main store picker, with per-location admin tools. Auto-approve sign-ups (no manager gate), admin-clicks-reset (no SMS/Twilio infra), lockout after 5 failed attempts in 10 min.

  • Migration 8_drivers_auth: extends the existing drivers table with email (unique), phone, address, password_hash, active, failed_attempts, lockout_until. Existing rows keep password_hash NULL until a manager runs Reset PW.
  • Idempotent column backfill in create_app(): Render's preDeployCommand doesn't run alembic and create_all() doesn't ALTER existing tables, so the migration's columns wouldn't otherwise reach prod. Each ALTER is gated on column-absence via inspect() — safe to run on every boot. Same self-healing pattern later applied to migration 10's orders.total_amount.
  • 5th "Drivers" card on the main store picker (store_picker.html) — full-width below the 4 store cards, with Sign Up + Log In buttons.
  • Routes in app/web/driver_routes.py:
    • GET /driver — redirects to login
    • GET/POST /driver/login — email + password, werkzeug check_password_hash
    • GET/POST /driver/signup — location dropdown + name + email + phone + address + password (8+ chars, confirm), werkzeug generate_password_hash, auto-active
    • GET /driver/logs — driver's own delivery log (existing, unchanged structurally)
    • POST /driver/logout — clears session
    • issue_temp_password(driver) helper — sets a new password_hash from secrets.token_urlsafe(6), returns plaintext for one-time display
  • Lockout: failed_attempts increments on bad password; at 5+ in 10 min the account is locked until lockout_until. Successful login resets both counters.
  • Per-store admin at /<store>/drivers (drivers_admin route in store_routes.py):
    • Lists drivers — Tomball/Copperfield see only theirs; Corporate/Partner see all locations.
    • Reset PW button — calls issue_temp_password(), redirects with the temp shown once via querystring (manager reads it to the driver).
    • Deactivate / Reactivate toggle.
  • Sidebar entry under each store's Ezcater group: "Drivers (Admin)".

Driver Phase B — live GPS + manager map (commit df61bc1)

After the auth foundation, the live tracking. Drivers tap "Start shift" in the portal — the browser opens a Wake Lock and starts streaming GPS fixes to the server, which renders a live map for managers.

  • Migration 9_driver_live_tracking: two new tables, both purely additive so create_all() handles them on Render with no extra backfill needed.
    • driver_shift: id, driver_id (FK), started_at, ended_at. Open shifts have ended_at IS NULL.
    • driver_location: id, shift_id (FK), driver_id (FK), captured_at, lat, lng, accuracy_m, speed_mps, heading_deg.
  • POST routes in driver_routes.py:
    • /driver/shift/start — closes any prior open shifts (defensive against orphans), opens new, returns shift_id.
    • /driver/shift/end — closes the open shift.
    • /driver/track — JSON body {lat, lng, accuracy_m?, speed_mps?, heading_deg?}. Validates range, requires open shift (else 409), inserts a driver_location row.
  • Driver portal UI (driver_viewing.html): shift card with Start/End button, GPS streamer below shows "streamed · N fixes · accuracy 8m" in real time. Wake Lock keeps the screen on; visibilitychange re-acquires it after the tab returns to foreground.
  • Manager live map at /<store>/drivers-live: Leaflet (CDN) + OpenStreetMap free tiles. positions.json JSON feed returns one record per on-shift driver with their latest fix; the page polls it every 5s and updates marker positions in place. Markers fade to grey when stale (>60s after the heartbeat fix below). Stores see only their drivers; corporate/partner see all.
  • Sidebar entry under each store's Ezcater group: "Drivers Live".

Phase B fix #1 — heartbeat (commit 9f1a272)

Sam tested. Marker appeared but didn't move — and aged into the grey "stale" state. Diagnosis: watchPosition only fires when the phone detects movement; if the driver was sitting still, no callback. Fix: a setInterval(15s) that re-POSTs the last known position so the manager always sees a recent timestamp. Stale threshold dropped from 120s to 60s now that 15s heartbeats are reliable.

Phase B fix #2 — switch from watchPosition to polling (commit f361384)

Sam re-tested. Still standing still on the map even when actually moving. Diagnosis (without his answers to my 3 diagnostic questions — he just said "still the same"): iOS Safari aggressively throttles watchPosition and stops firing the callback after the initial fix. The heartbeat from fix #1 was just resending the original coords. Fix: replaced watchPosition entirely with setInterval(getCurrentPosition, 5000). Explicit polling on a fixed cadence is more reliable than the event-driven API on iOS Safari. Trade-off: ~10%/hr battery instead of ~5%/hr. Acceptable for delivery shifts where the phone is plugged into a dash mount anyway. Sam confirmed "Aick it looks good" after this landed.

ezCater revenue parsing + reports integration (commit 1377736)

Sam wanted DoorDash + Uber + ezCater to feed both /reports/sales and the labor cost ratio. Found that DoorDash + Uber were ALREADY in both (Toast aggregates them as channels via the API source + delivery-address detection; labor_report.net_sales already sums every non-voided check.amount). Only ezCater was the gap, since ezCater orders never touch Toast.

Tried to fetch ezCater order totals from the Partner API — got 403 on catererCart.totals.catererTotalDue for our token. Then discovered the prices were already embedded in our DB on every order item: OrderItem.raw_alias is formatted "Item Name @ $XX.XX" by the existing PDF + API ingest. So no API call needed — just parse the line text and multiply by qty.

  • Migration 10_order_total_amount: adds orders.total_amount (Float, nullable). Companion ALTER backfill in create_app() + a one-shot row backfill that computes total_amount for any rows where it's NULL (capped at 500 per boot so a cold-start doesn't time out).
  • New app/services/ezcater_pricing.py — parse_unit_price(raw_alias) + compute_order_total(items). Falls back to the scraped storefront menu in data/ezcater/menu_prices.json for items missing the $ token.
  • New app/services/ezcater_revenue.py — fetch_ezcater_orders(start, end, location_filter) + total_ezcater_revenue(...). Used by the report layer.
  • persistence_service._persist_order calls compute_order_total immediately after flushing OrderItems so new orders are populated at ingest time.
  • toast_reports.third_party_sales_report appends an ezCater channel to the by_channel dict from our Order DB. SALES_CHANNEL_FILTERS gained an ezcater key so the channel filter / multi-select works.
  • toast_reports.labor_report sums ezcater_sales into net_sales alongside Toast revenue so the labor % reflects ALL revenue.
  • Storefront menu scraped via Claude in Chrome MCP from cenas-fajitas-houston — 56 menu items + categories saved as JSON. Sam confirmed prices don't change.
  • Local backfill: 51 historical ezCater orders, $26K revenue YTD.

ezCater XLSX importer (commits 07b37f8 + 6a7be97)

The 51 orders we already had came from the webhook pipeline. Sam pointed out there were many more historical orders (~233 more) we'd never received via webhook. He sent two Caterer Portal "Order Data" xlsx exports (delivery-performance-report didn't have totals; the report_2026-04-01_2026-05-09 + the wider Mar 1 file did, with 270 + 452 rows respectively).

  • New app/services/ezcater_import.py — parse_export_xlsx(stream) + apply_import(rows). Reads "Order Data" sheet, extracts Order Number + Food Total + Caterer Total Due + Store Number. Skips the trailing summary "Total" row (caught after the first xlsx; commit 6a7be97).
  • New app/web/ezcater_import_routes.py — Partner-only blueprint at /partner/developer/ezcater-import. Multipart upload of the xlsx; for each row matches Order Number → existing row by external_order_id and overwrites total_amount with Food Total. For orders not in our DB (pre-pipeline), creates a stub Order row with delivery_date + origin_store_id + total_amount + status="imported".
  • Result page shows: parsed / updated / created / skipped counts plus sample rows. Idempotent — re-uploading the same file is a no-op (matches by Order Number, overwrites total).
  • Default to Food Total (food subtotal, comparable to Toast check.amount) rather than Caterer Total Due (what Cenas nets after fees). Captures both for future use.
  • Local result after upload: 51 → 284 orders, $145K Food Total across 1+ months.
