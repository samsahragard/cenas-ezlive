Site-wide keypad auth (migration 13)

Sam's spec at session open: "Aick Regarding the entirety of the site. I want to entirely change how people login" — replace the shared-password Tier 1 / Tier 2 gates with a per-person 5-digit numeric passcode + a real role model.

Schema (migration 13_users_keypad_auth)

  • New users table: id, created_at, updated_at, full_name, email (unique), phone (unique), passcode_hash (werkzeug), permission_level, store_scope, first_login_done, active, failed_attempts, lockout_until, last_login_at, last_login_ip, session_version.
  • Roles in descending privilege: partner > corporate > gm > manager > expo > corporate-driver. Ranks live in permissions.LEVELS; level_at_least(level, min) checks privilege.
  • store_scope is a CSV string ('tomball', 'copperfield', 'tomball,copperfield') — single-value rows still parse fine, multi-store assignment was Sam's later ask in the same session.
  • Idempotent ALTER block in app/__init__.py creates the table (via Base.metadata.create_all) and back-fills the session_version column on existing prod DBs without needing the alembic migration to run (Render's preDeployCommand is still unset; same pattern as the drivers + orders backfills).
  • Sam seeded on first prod boot as partner with bootstrap passcode 12345, first_login_done=False — forced to change on first login.

Authentication routes (app/web/keypad_auth.py)

  • GET /keypad-login — renders the keypad. If already signed in, redirects to the user's role landing (not the keypad, not the store picker).
  • POST /keypad-login — JSON {"passcode":"12345","next":"/"}. Scans all active users with check_password_hash; first match wins. On success bumps last_login_at, stamps session.user_session_version with the user's current version, sets auth_ok=True (legacy shim so partner-gated routes work). Sets partner_auth_ok=True ONLY for partner level (corporate explicitly excluded — see "Permission audit" below).
  • GET /change-passcode — forced on first login or admin reset.
  • POST /change-passcode — sets a new passcode, marks first_login_done=True, redirects to the user's role landing. Passcode uniqueness enforced across active users at create/change time (no two users can have the same passcode).
  • GET /keypad-logout — clears user_id, auth_ok, partner_auth_ok from session, redirects to keypad.
  • Both keypad pages send Cache-Control: no-store so the browser doesn't serve stale HTML after a deploy (Sam hit a passcode issue from cached pre-21e7e17 JS).
  • Passcode regex: ^[\d*#@+%\-$]{5}$ — digits + every special key on the pad. Sam's spec was "no special characters" first, then revised after he tried #3676 and was rejected. The full set * # @ + % - $ is now accepted on both login and change.

Permissions decorator (app/web/permissions.py)

  • require_level(min_level) view decorator: 302's to /keypad-login if no g.current_user, returns 403 if the user's level is below the minimum.
  • load_current_user() attaches the User on g.current_user and ALSO compares session.user_session_version with User.session_version — mismatch clears the session immediately. That's how admin reset / deactivate force-logs the user out everywhere instantly (bump version → next request anywhere on any device → kicked).
  • accessible_store_slugs(user) returns the list of store_slug values this user can access. Partner → ['partner'], corporate → ['corporate'], gm/manager/expo with single store → ['dos'] or ['uno'], multi-store → both. corporate-driver → [] (driver portal is outside the store_bp).

Force-logout on reset / deactivate

Sam's mid-session ask: "when a password is reset or a person is deactivated, it should automatically log them out from everywhere." Implemented via User.session_version int (defaults 1). Login stamps it on the session cookie; every request load_current_user re-checks. On mismatch the session is cleared and the user is bounced to /keypad-login on their next page load — works across devices instantly.

  • team_reset bumps version + sets first_login_done=False (forces the user to change the temp passcode on next login).
  • team_toggle (active/inactive) bumps version too — flipping to inactive immediately kicks any live session AND blocks future logins via _find_user_by_passcode filtering on active.is_(True).
  • team_edit bumps version only if the role moved (permission_level or store_scope changed) — name/email edits don't kick the user.

Permission audit — Sam's spec decoded

Sam's 06:14 PM answer (parsed via ck because it was voice-to-text and bled together):

Role Sees Doesn't see
Partner Everything. Operations + Insights + Forecasts + Admin (Team) + Developer (Chat, Ezcater Review, App docs). —
Corporate Everything Partner sees EXCEPT Admin and Developer. /partner/team, /partner/developer/*.
GM All Operations + Insights for their assigned store(s) only. Admin, Developer, other stores' URLs.
Manager Same as GM today. (Sam reserved future divergence.) Same as GM.
Expo Operations only — Orders / Order Processor / Driver Payroll / Drivers Admin / Drivers Live / Schedule / Corporate Order / Vendors. Entire Insights section (Performance / Sales / Labor / Forecast), Admin, Developer, other stores.
Corporate Driver /driver/portal only. All dashboards.

Where each rule is enforced

  • Sidebar visibility: base_dashboard.html wraps the Admin + Developer blocks in {% if store_slug == 'partner' %}, wraps the Insights block in {% if not (g.current_user and g.current_user.permission_level == 'expo') %}.
  • Route gates: store_bp._per_store_gate before_request hook (app/web/store_routes.py) does two things — (1) Expo hitting any Insights URL gets a flat 403; (2) GM/Manager/Expo with single store_scope hitting the wrong /<slug>/ get 302'd to their own first store. Partner/Corporate skip both checks.
  • Team admin: require_level('partner') decorator on every team_* route — corporate gets 403 even though they could try to URL-hop to /partner/team. They also can't reach it via sidebar (partner-only block) or via /partner/ (the partner_gate redirects them to /partner-login since corporate users don't get partner_auth_ok at keypad login).

Regression watch: ck's c2ab56a commit (a "docs: ck session 2026-05-11 handoff log" commit) silently dropped the _per_store_gate hook because his working copy was stale. Restored in ebe1e2a. Added an inline comment so future diffs that delete it should raise an eyebrow.
