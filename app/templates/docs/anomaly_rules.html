What this page is

Phase 0 spec for the anomaly-detection rule library. Sister doc to permission_system: permissions answer "who can see what"; this answers "what is the system trying to surface, to whom, and how loudly?" Each rule below is a small query against operational data that fires either an info / warn / alert signal. Signals feed two surfaces:

  • The Phase 1 morning brief — once-a-day digest to partners and GMs covering everything that warned or alerted in the last 24h, plus the day-ahead lookahead.
  • Anomaly cards on relevant pages — a small colored card that appears inline on the page the rule is contextually relevant to (e.g. the late-delivery card appears on /orders/by-store, the tip-rate card on /reports/server-performance). Card color tracks severity: blue / yellow / red.

Scope of this doc: the rule library + the data model for signal storage. The implementation (the rule engine, the morning brief composition, the card renderer) is Phase 1 work and lives in separate specs once we cut Phase 0. Nothing in app/ changes from this doc alone.

1. Signal data model

Every fired rule writes a row to a new Signal table: 

class Signal(Base):
    __tablename__ = "signals"
    id: int
    rule_name: str              # e.g. "orders.no_driver_30min_before"
    severity: str               # "info" | "warn" | "alert"
    store_id: str | None        # store_1..store_4 or None for cross-store
    subject_id: str | None      # e.g. external_order_id, employee_id, vendor_id
    subject_label: str          # human-readable subject (order #65X-814)
    trigger_at: datetime        # when the rule fired
    payload: dict               # rule-specific context (JSON column)
    action_text: str            # the one-line recommended response (copied from rule def)
    surfaces: list[str]         # which pages should render this card
    audience_roles: list[str]   # which roles see the card / get the brief mention
    acknowledged_by: int | None # user_id who clicked "acknowledge" — clears the card
    acknowledged_at: datetime | None
    resolved_at: datetime | None  # nullable: rules that auto-clear set this

Why a Signal row vs computing on the fly: we want acknowledgments to persist (so a once-cleared card stays gone for the day), the morning brief needs a stable list of what happened, and a partner viewing /partner/anomalies wants history not just current state. Rules with auto-clear logic (e.g. the order got a driver after all) write resolved_at and the card disappears without manager intervention.

Surfaces vocabulary: surfaces is a list of page slugs the card renders on. Conventions: "home", "orders.by_store", "reports.labor", "reports.server_perf", "reports.third_party_sales", "produce.queue", "kds.station", "partner.anomalies" (the dedicated index), "morning_brief" (the daily digest channel). A rule can surface to multiple pages; the renderer de-duplicates based on Signal id.

2. Severity scale

SeverityColorWhat it means Where it surfacesBrief inclusion
infoblue FYI — pattern worth knowing about, no action needed today. Trend signals, soft variances. partner.anomalies index + the rule's contextual page if any. Skipped on home dashboard. Aggregated count only ("12 info signals in the last 24h").
warnyellow Needs attention soon but not blocking. Outside normal but not a fire. partner.anomalies + the rule's contextual page + home dashboard for partner/GM. Itemized list with subject + action text.
alertred Immediate action needed. Time-sensitive (e.g. order in 30min with no driver) or financial-impact (e.g. invoice $400 over quote). Same as warn, plus push notification (when push lands), plus banner on home dashboard for the audience role. Top of brief, called out individually.

Calibration rule: if a rule fires more than 3 times per store per day on average, downgrade its severity by one notch (alert becomes warn, warn becomes info, info becomes silent). Noise destroys attention. The brief footer shows which rules were downgraded so we can tune thresholds.

4. Morning brief composition

Generated daily at 7:30am CT. One brief per audience role (partner, gm-per-store, km-per-store, corporate_chef, etc.). Sent via email + posted to the user's /partner/briefs inbox surface. Composition:

  1. Header. Date, store(s) in scope, sleep-quality cheek (signal count by severity since yesterday's brief).
  2. Alerts (red). Itemized list of alert signals from the last 24h that are still open (not resolved_at or acknowledged_by). One per line: subject + action text + link to the surface page.
  3. Warns (yellow). Same shape as alerts.
  4. Today's lookahead. Predicted alerts based on same-rule history (e.g. "no driver assigned" rule projected to fire between 11am-1pm if pending orders stay unassigned).
  5. Info aggregate. "N info signals in the last 24h" with a link to the full list.
  6. Calibration footer. Rules downgraded for noise yesterday + any rules that fired zero times this week (candidates for retirement or threshold adjustment).

The brief is generated by a Claude call with the Signal table + the rule definitions as context (similar pattern to the ezcater resolver ck shipped). Each role gets only their permission-tag-visible signals (the brief composer enforces the same _user_has check that the decorator uses).

5. Anomaly cards on pages

When a page renders, the layout includes an anomaly-card slot that queries: 

SELECT * FROM signals
WHERE 'page.slug' IN unnest(surfaces)
  AND (acknowledged_by IS NULL AND resolved_at IS NULL)
  AND audience_roles && user.role_set
  AND (store_id IS NULL OR store_id = current_store_id)
ORDER BY severity DESC, trigger_at DESC
LIMIT 5;

Cards render colored by severity (blue/yellow/red), with the action_text prominent and a single “Acknowledge” button. Acknowledging marks the row's acknowledged_by for the user; the card disappears for that role going forward (or until the rule re-fires for a new subject_id). Audit log: every acknowledgment writes to a SignalAck table for after-the-fact reviewability.

6. Threshold tuning

Every rule's threshold above is a starting point, not a contract. Each fires into a Signal row with the raw measured value in payload; after two weeks of operation we tune by:

  • Rules that fired >3× per store per day on average — raise threshold or downgrade severity.
  • Rules that fired zero times in two weeks — consider lowering threshold, or retire (the trigger condition may never actually occur in practice).
  • Rules where the recommended action was rarely taken (the SignalAck table can show ack-without-action by cross-referencing manager_log entries) — rework the action text or downgrade severity.

A simple admin page at /partner/anomalies/rules exposes the rule library with per-rule fire counts and last-7-day acknowledgment rates. Partner can edit thresholds inline (writes to rule_overrides table, takes effect on next rule run; the defaults in code stay; overrides win).

7. Implementation notes (for whoever picks up Phase 1)

  • Rule engine lives in app/services/anomaly_engine.py. Each rule is a function returning a list of SignalDraft objects. The engine runs on the cadence specified per rule (cron-driven or on-write hooks).
  • Cron rules slot into the Render Cron Jobs aick wired in Phase 0 Block 1 (crn-d82e5ivaqgkc7393acsg pattern). One Render cron entry per cadence (every-5-min / every-15-min / hourly / daily / weekly), each calling /cron/anomaly-eval?bucket=<cadence>. The handler reads which rules belong to that bucket and runs them.
  • On-write rules wire into existing event paths: invoice ingest, KDS state update, manager_log insert, Toast webhook receipt. Each writes Signal rows in the same transaction so a partial failure rolls back.
  • Push notifications (alerts only, partner phone) require a separate Phase 1 deliverable — out of scope here. For now alerts surface via the home dashboard banner + the morning brief.
  • De-duplication: when a rule re-evaluates the same subject_id, look for an open Signal with the same (rule_name, subject_id, store_id). If found, update trigger_at and merge payload; do not create a duplicate row.
  • Auto-clear: rules with reversible triggers (no_driver_30min_before, late_delivery, equipment_down, missed_order_window) re-check on every run cycle and write resolved_at when the condition no longer holds. Manager doesn't have to ack a resolved signal — the card disappears on its own.

8. What this spec doesn't decide (kicked to Phase 1+)

  • Push channel. Where alerts land in real time (SMS, WhatsApp via ock's existing connection, Telegram, push to the Capacitor app). Pick during Phase 1 build.
  • Cross-rule correlation. e.g. labor_running_hot + sales_daily_vs_forecast firing together implies different action than each alone. Phase 2 layer that reads the Signal table and proposes consolidated narratives.
  • Predictive signals. Forecasted alerts (today's sales projected to miss forecast by 4pm) are noted in the brief's lookahead section but not implemented in Phase 1; come back to this when we have enough signal history to train a simple model.
  • Self-tuning thresholds. The two-week tuning pass above is human-in-loop. A future enhancement is to learn the threshold band per rule per store from the first month of data.
  • Custom rules per user. No mechanism for a partner to write their own one-off rule yet. Workaround: file an issue in the developer chat, samai or aick adds it to this spec.
