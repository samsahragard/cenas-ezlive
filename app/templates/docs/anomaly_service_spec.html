What this page is

Implementation spec for the anomaly-detection service. Counterpart to samai's anomaly_rules rule library (32 rules across 10 domains). That doc answers "what does the system surface, to whom, how loudly". This doc answers "how do we build it" — the data model SQLAlchemy decl, the rule-engine architecture, cron bucket assignments, dedup + auto-clear logic, LLM routing per rule, the morning brief composer, the anomaly-card render path, and the Telegram alert surface.

Scope: code-ready spec — no work in app/ changes from this doc alone, but every section is a single PR away from shipping. Phase 1 / Block 1 picks this up.

2. Data model

Three tables. signals holds the events, signal_ack is the acknowledgment audit log, rule_overrides stores per-rule threshold edits a partner has made via /partner/anomalies/rules.

3. Rule registry + engine

3.1 Registry pattern

Every rule is a Python function decorated with @anomaly_rule(...). The decorator registers the rule into a module-level dict keyed by rule_name: 

# app/services/anomaly_engine.py

from dataclasses import dataclass
from typing import Callable

@dataclass
class SignalDraft:
    """What a rule emits. Engine handles dedup + write."""
    rule_name: str
    severity: str            # "info" | "warn" | "alert"
    store_id: str | None
    subject_id: str | None
    subject_label: str
    payload: dict
    action_text: str
    surfaces: list[str]
    audience_roles: list[str]


@dataclass
class RuleSpec:
    name: str
    bucket: str              # "every_5m" | "every_15m" | "hourly" | "daily" | "weekly" | "on_write"
    severity_default: str
    surfaces: list[str]
    audience_roles: list[str]
    action_text: str
    fn: Callable             # (db: Session) -> list[SignalDraft]
    llm_route: str           # "sql" | "haiku" | "opus"


REGISTRY: dict[str, RuleSpec] = {}


def anomaly_rule(*, name, bucket, severity, surfaces, audience_roles,
                 action_text, llm_route="sql"):
    def deco(fn):
        REGISTRY[name] = RuleSpec(name=name, bucket=bucket,
                                  severity_default=severity,
                                  surfaces=surfaces,
                                  audience_roles=audience_roles,
                                  action_text=action_text,
                                  fn=fn, llm_route=llm_route)
        return fn
    return deco

3.2 Engine loop

def run_bucket(bucket: str) -> dict:
    """Run every rule belonging to a bucket. Returns summary stats."""
    db = SessionLocal()
    fired = saved = updated = resolved = 0
    try:
        for spec in REGISTRY.values():
            if spec.bucket != bucket:
                continue
            try:
                drafts = spec.fn(db)
                seen_keys = set()
                for d in drafts:
                    fired += 1
                    key = (d.rule_name, d.subject_id, d.store_id)
                    seen_keys.add(key)
                    if _upsert_signal(db, d):
                        saved += 1
                    else:
                        updated += 1
                # Auto-clear: any open Signal for this rule whose subject
                # didn't appear in this run gets resolved_at stamped.
                resolved += _auto_clear(db, spec.name, seen_keys)
            except Exception:
                logger.exception("anomaly rule %s crashed (non-fatal)", spec.name)
        db.commit()
    finally:
        db.close()
    return {"bucket": bucket, "fired": fired, "saved": saved,
            "updated": updated, "resolved": resolved}

5. Dedup + auto-clear

5.1 Upsert algorithm

def _upsert_signal(db, d: SignalDraft) -> bool:
    """Returns True if a new row was inserted; False if an open one was updated."""
    existing = (db.query(Signal)
                .filter(Signal.rule_name == d.rule_name)
                .filter(Signal.subject_id == d.subject_id)
                .filter(Signal.store_id == d.store_id)
                .filter(Signal.resolved_at.is_(None))
                .filter(Signal.acknowledged_at.is_(None))
                .first())
    if existing:
        existing.trigger_at = datetime.utcnow()
        existing.payload = {**(existing.payload or {}), **d.payload}
        existing.severity = d.severity   # severity can change between fires
        return False
    db.add(Signal(rule_name=d.rule_name, severity=d.severity,
                  store_id=d.store_id, subject_id=d.subject_id,
                  subject_label=d.subject_label,
                  trigger_at=datetime.utcnow(),
                  payload=d.payload,
                  action_text=d.action_text,
                  surfaces=d.surfaces,
                  audience_roles=d.audience_roles))
    return True


def _auto_clear(db, rule_name: str, seen_keys: set) -> int:
    """Stamp resolved_at on any open Signal whose subject didn't appear
    in this run — only applies to rules with reversible triggers."""
    if rule_name not in REVERSIBLE_RULES:
        return 0
    open_rows = (db.query(Signal)
                 .filter(Signal.rule_name == rule_name)
                 .filter(Signal.resolved_at.is_(None))
                 .filter(Signal.acknowledged_at.is_(None))
                 .all())
    cleared = 0
    for r in open_rows:
        if (r.rule_name, r.subject_id, r.store_id) not in seen_keys:
            r.resolved_at = datetime.utcnow()
            cleared += 1
    return cleared


REVERSIBLE_RULES = {
    "orders.no_driver_30min_before",
    "orders.late_delivery",
    "kds.station_red",
    "kds.expedite_late",
    "equipment.down",
    "attendance.no_clock_in",
}

5.2 Calibration / noise cap (from anomaly_rules §2)

Implemented as a nightly job that walks the last 7 days of Signals, computes fire_count_per_store_per_day per rule, and writes severity_override entries to rule_overrides for rules that exceed the >3/store/day threshold (downgrade by one). The engine's run_bucket consults rule_overrides first, falling back to spec.severity_default.

6. LLM routing per rule

Most rules are pure SQL — pattern detection over the existing tables. Some need LLM help. The llm_route field on RuleSpec decides:

RouteModelUse caseLatency budget
sql(no LLM) Threshold checks, time-windows, count comparisons. ~28 of 32 rules. < 1s per rule
haikuclaude-haiku-4-5-20251001 Narrative summarization on a single Signal payload (per-row enrichment of subject_label, free-text customer-review classification, ambiguous shrinkage cause). ~4 rules. < 3s per rule fire
opusclaude-opus-4-1-20250805 Daily morning brief composition — reads ALL last-24h Signals + the rule library + the day-ahead calendar, writes the narrative brief per audience role. One call per role per day. < 30s per role

Cost guardrails: rule engine logs a per-rule llm_call_count + llm_token_count. If a haiku rule exceeds HAIKU_DAILY_BUDGET tokens (default 1M), subsequent fires fall back to pure SQL emission (no enrichment) and log a warning.

9. Telegram alert path (severity=alert)

Inside the engine, after a new alert-severity Signal is inserted (not on updates of an open one — that would re-fire on every poll). Reuse the existing _tg_send helper from ezcater_webhook.py: 

def _alert_tg_send(s: Signal) -> None:
    from app.web.ezcater_webhook import _tg_send  # existing helper
    text = (
        f"🚨 {s.severity.upper()} · {s.rule_name}\n"
        f"{s.subject_label}\n"
        f"{s.action_text}\n"
        f"Acknowledge: https://app.cenaskitchen.com/partner/anomalies"
    )
    _tg_send(text)

De-duplicated by _upsert_signal: only fires on NEW rows, not updates to existing open ones, so the same alert doesn't ping every 5 min. Channel binding: same SAM_TG_CHAT_ID for now; Phase 2 introduces per-role chat ids.

11. Failure handling

  • Single rule crashes — caught inside the engine loop, logged, never propagates. The bucket run continues with the next rule. Sentry captures the traceback.
  • DB write fails — the entire bucket rolls back (one tx per bucket run). The next run picks up where we left off; no partial signals shipped.
  • LLM call fails — rule falls back to its pure-SQL emission (no enrichment). The Signal still lands.
  • Cron job fails to fire — Render notifies via the cron-failure email; Sentry doesn't see it (the cron service is separate from the web service). Worth wiring Render → Sentry via a heartbeat ping per bucket later.
