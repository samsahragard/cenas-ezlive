Architecture overview

Single Flask app, SQLite DB on a persistent disk. No microservices, no message bus. Everything that needs to be persistent goes through SQLAlchemy ORM. Background work (the produce IMAP poller) runs as a daemon thread inside the gunicorn process, guarded by a file lock so multiple workers don't double-poll.

The store-prefix layer

The store_routes.py blueprint is the navigation skeleton for the four scopes (DOS / UNO / Corporate / Partner). It uses Flask's url_value_preprocessor to extract the slug off the URL and put it on g.current_store, then has thin wrapper routes that delegate to the top-level handlers (e.g., /dos/reports/labor/boh sets g.location_override = 'tomball' and g.labor_filter = 'boh', then calls the existing /reports/labor handler).

Why this design: the legacy top-level URLs (/reports/labor?location=tomball) still work for backwards compatibility and bookmark stability, while the new /<store>/... URLs are the canonical thing the sidebar generates. Reports honor g.location_override first, fall back to the ?location= query param.

Database

SQLite on a Render persistent disk. Path is /var/data/cenas_kitchen.db via the DATABASE_URL env var. Tables:

TablePurpose
ordersCatering orders (ezCater + manual PDF upload)
order_itemsPer-line items in each catering order
driversDriver directory (one row per driver per location)
driver_logsPer-delivery driver performance entries
produce_price_snapshotOne row per (vendor × item × snapshot date), populated by the IMAP poller
developer_chatPersistent chat for Sam + AI agents

Schema-evolution story: alembic migrations live in migrations/versions/. Render's preDeployCommand should run alembic upgrade head but that field is currently unset on the service. As a backstop, create_app() calls Base.metadata.create_all() on startup which is idempotent and creates any new tables (but doesn't apply ALTERs to existing tables).
