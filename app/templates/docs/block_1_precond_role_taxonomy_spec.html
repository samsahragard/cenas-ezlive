1. Scope — why this precondition exists

The Phase 2 task-assignment hierarchy (1A's can_assign_to) and the counseling-cascade (2C's can_view_counseling) both reference hourly roles — cook, server, busser, host, bartender — that do not exist in the current taxonomy. ROLE_PERMISSIONS has ten canonical roles; only expo and driver sit at the hourly tier today. Sam's call (01:05): Path A — round out the taxonomy as one small precondition commit so 1A starts with the full taxonomy in place (no xfail'd can_assign_to branches).

This precondition ships, as one commit:

  • app/services/role_hierarchy.py — the new shared module: ROLE_TIER, role_tier(), role_domain(), for all roles including the five new hourly ones
  • Minimal ROLE_PERMISSIONS entries for cook / server / busser / host / bartender in app/services/permissions.py
  • Tests: the tier + domain mapping for every role, legacy-alias resolution, the unknown-role safe default; the permission matrix from 2255c1d auto-extends to the five new roles

Sequencing note — this changes the 1A spec. The 1A spec §4 said "1A creates app/services/role_hierarchy.py." That responsibility moves here — this precondition creates the module complete; 1A imports and consumes it. The 1A spec §4 will be amended (against git show HEAD, per rule 6) to reflect that once this precondition is samai-PASSed.

2. app/services/role_hierarchy.py — the module

A new, self-contained, pure module (no DB, no Flask). It holds the org-tier / domain concern, kept deliberately separate from permissions.py's tag-grant concern — they answer different questions and evolve independently. Three things:

2.2 role_domain — kitchen / foh / both

_ROLE_DOMAIN: dict[str, str] = {

The kitchen/foh split is the cross-cut can_assign_to needs below the GM tier (a KM assigns to kitchen staff; an FOH manager assigns to FOH staff). The hourly assignments align with the existing role_classifier.py BOH/FOH rule (Sam's 2026-05-09 rule: cook = BOH/kitchen; bussers, expo, servers, hosts = FOH) — so this module's role_domain and role_classifier.classify_role agree on the same staff, just keyed differently (app role vs Toast job title). partner / corporate / gm = both (they cross domains).

2.3 The two functions

def role_tier(role: str | None) -> int:
    """Tier for a User.permission_level value. Resolves _LEGACY_ALIASES
    first ('manager' -> gm -> 3, 'corporate-driver' -> driver -> 1).
    Unknown / None -> 0 (no authority) — never raises."""

def role_domain(role: str | None) -> str:
    """'kitchen' | 'foh' | 'both' for a role. Resolves legacy aliases.
    Unknown / None -> 'foh' — the safe default (matches role_classifier's
    'everything else is front' rule; an unknown role can never be
    mis-assigned kitchen-only work it shouldn't see)."""
  • Legacy aliases: both functions resolve through permissions._LEGACY_ALIASES ("manager"→gm, "corporate-driver"→driver) before lookup — import that dict from permissions.py, do not re-declare it (single source of truth).
  • Unknown role: role_tier → 0 (an unknown role has no authority — it can never out-rank anyone, so can_assign_to and can_view_counseling both fail safe-closed for it). role_domain → "foh" (the role_classifier-consistent safe default). Neither function raises — a stray permission_level value degrades, never 500s.

3. Minimal ROLE_PERMISSIONS entries for the five hourly roles

Add five entries to ROLE_PERMISSIONS in app/services/permissions.py. The point of these entries is taxonomy completeness — "everyone in the system is in the role taxonomy" (Sam's words) — so _user_has resolves these roles to a real set instead of None, and the permission matrix test covers them. The tag sets are deliberately minimal — the baseline every existing staff role already holds: 

# Phase 2 / Block 1 precondition (samai spec, Sam §5.3 Path A 2026-05-14).
# Minimal baseline — the personal-AI + transcripts set that expo + driver
# already share. Operational tag refinement per hourly role (does a cook
# need kds.view_kitchen_display, does a server need orders.view, etc.) is
# a deliberate follow-up once these roles have actual app surfaces — see
# the precondition spec §6.
_HOURLY_BASELINE = {
    "ai.ask_claude_personal", "ai.view_transcripts",
    "transcripts.search", "transcripts.read",
}
"cook":      set(_HOURLY_BASELINE),
"server":    set(_HOURLY_BASELINE),
"busser":    set(_HOURLY_BASELINE),
"host":      set(_HOURLY_BASELINE),
"bartender": set(_HOURLY_BASELINE),

Colocated comment makes the "minimal-by-design, refinement-is-a-follow-up" intent explicit so a future reader doesn't mistake the thin set for an oversight. cook is the one arguable case (kitchen staff plausibly want kds.view_kitchen_display like expo has) — flagged in §6 Q1, but the precondition ships the uniform baseline and the refinement is separate.
