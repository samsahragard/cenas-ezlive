Block 1 Precondition — ScheduledEvent Model

Phase 2 / Block 1 / precondition commit (gates 1C's Caterings + Events adapters). samai spec, 2026-05-14.
Implementation: aick. Review: samai checklist PASS. Sam decision: 1C §13 Path A (2026-05-14 01:05).
Source: Block 1C spec §13 Q1 + Sam's 01:05 decision.

1. Scope — why this precondition exists

The directive names ScheduledEvent as a ribbon source — the Caterings (ribbon category 2) and Events (category 3) sections pull from it — but no Block 1 or Block 2 sub-block defines the model, and aick confirmed it does not exist anywhere in the codebase. Sam's call (01:05): Path A — ship the model now, even minimal, so Block 1 does not launch with 2 of 7 ribbon categories permanently empty (which would train managers to ignore those sections before they ever have content).

This precondition ships, as one commit:

  • ScheduledEvent model in app/models.py
  • One alembic migration
  • Tests: model round-trip, defaults, index declaration

This precondition does NOT ship:

  • The admin form for Sam / Anna / Brittany to add events — that is Block 2 (Sam: "through a small admin form in Phase 2 Block 2").
  • The 1C ScheduledEvent ribbon adapter — that is 1C. This precondition just makes the model exist so 1C's adapter has something to read.
  • ezCater-webhook auto-population of catering events — Phase 2.5 or 3.
  • No created_by_user_id — that is the Block 2 admin-form's concern; it is not a "minimum field needed for ribbon rendering" (Sam's framing), so it stays out of the precondition.

Empty-but-populatable from day one: the table ships empty, the migration creates it, and Block 2's admin form fills it. 1C's adapter renders whatever is in it (nothing, at first — and that is fine, an empty Caterings section with a real model behind it is a launch-ready category, not a hollow one).

2. The ScheduledEvent model

class ScheduledEvent(Base):
    __tablename__ = "scheduled_events"

    id               Integer, primary key
    store            String(20),  NOT NULL          # tomball | copperfield | both
    category         String(20),  NOT NULL          # catering | event
    title            String(200), NOT NULL
    scheduled_at     DateTime,    NOT NULL, indexed  # when the event starts
    scheduled_end_at DateTime,    nullable           # when it ends; NULL = point-in-time
    status           String(20),  NOT NULL, default="scheduled"
        # scheduled | confirmed | completed | cancelled
    notes            Text,        nullable
    created_at       DateTime,    NOT NULL, default=utcnow
    updated_at       DateTime,    NOT NULL, default=utcnow, onupdate=utcnow

    __table_args__ = (
        Index("ix_scheduled_events_ribbon", "store", "status", "scheduled_at"),
    )

The eight fields Sam named (id, store, category, title, scheduled_at, scheduled_end_at, status, notes) plus the standard created_at / updated_at pair every model in the codebase carries — those two are implicit in "minimum fields," not an expansion of scope.

  • store / category / status are String not native ENUM — same call as Task / SalesInsight / BriefFeedback (SQLite-compat + simpler migrations); values validated application-side when the Block 2 admin form writes them, and documented in module-level constants (_VALID_EVENT_STORES, _VALID_EVENT_CATEGORIES, _VALID_EVENT_STATUSES) so the admin form + 1C adapter + tests share one source.
  • store carries the same three values as every other Phase 2 store-scoped surface — tomball | copperfield | both — so 1C's adapter can match an event against the viewer's store scope the same way it filters Tasks and SalesInsights. (Sam's 01:05 field list said "store"; spec keeps that column name and allows both for region-wide events — see §6 Q1.)
  • category = catering | event — the two values 1C's adapter maps to ribbon slugs (catering → caterings, event → events). Minimal per Sam; event subtypes (spirit_night / fundraiser / large_party) are a possible later refinement — §6 Q2.
  • status defaults to "scheduled". 1C's adapter will surface scheduled + confirmed events and skip completed / cancelled — but that filtering is 1C's concern; the precondition just defines the column + the default.
  • scheduled_end_at nullable — a catering delivery is effectively point-in-time (NULL end); a spirit night spans hours (real end). 1C uses it for the ribbon's relevance window and 1E's auto-expiry can lean on it later if events get the same auto-clear treatment as SalesInsights (not this commit's concern).
  • ix_scheduled_events_ribbon (store, status, scheduled_at) — the 1C ribbon query is "upcoming scheduled/confirmed events for this viewer's store," which hits exactly these three columns. Lay the index now so 1C doesn't add it as a separate migration.
  • No before_delete listener, no append-only constraint — ScheduledEvent rows are mutable operational records (an event gets confirmed, rescheduled, cancelled). It is not an audit log.

3. The migration

One alembic migration (versions/18_scheduled_event.py, revises whatever the latest is at implementation time — likely 17_ribbon_category_preference), upgrade() creates the table + the composite index, downgrade() drops them. Documentation-form per the codebase convention (alembic isn't wired on the live service; Base.metadata.create_all() picks up the new table at boot) — same note the recent migrations carry.

4. Test requirements

Test file: tests/test_scheduled_event_model.py. This is a model-only commit, so coverage is small:

  • Round-trip: a ScheduledEvent with all eight named fields persists and reads back; a point-in-time event (scheduled_end_at=None) round-trips fine.
  • Default: a row created without an explicit status gets "scheduled"; created_at / updated_at default-fire.
  • Index declared: ix_scheduled_events_ribbon is present in __table_args__ (the same cheap regression guard the other model tests use).
  • Valid-value constants: the three module-level constant sets exist and contain the documented values — so the Block 2 admin form + 1C adapter have a shared source to validate against.
  • Full suite no regression.

5. samai review checklist

  1. Scope: only the ScheduledEvent model + migration + test file. No admin form, no 1C adapter, no webhook, no created_by_user_id.
  2. Model matches §2: the eight named fields + standard created_at/updated_at, the types, status default, the composite index.
  3. store / category / status are String with the valid-value module constants declared.
  4. No before_delete listener — it is a mutable operational record, not an audit log.
  5. Migration present + reversible.
  6. Tests per §4 green; full suite no regression.
  7. Deploy-verify (rule 3): app boots, the table is created; verified against prod (rule 2).
  8. After PASS: 1C spec §13 Q1 is resolved — note it (the ScheduledEvent adapter is unblocked, 1C ships all 7 ribbon categories non-hollow). Amend 1C spec §13 against git show HEAD per rule 6 if a doc change is warranted.

6. Open questions for Sam

  1. Q1 — store values. Sam's 01:05 field list said "store." Spec keeps that column name and allows tomball | copperfield | both (consistent with every other Phase 2 store-scoped surface, and both covers a region-wide event like an Astros game affecting both stores). Confirm both is wanted, or say if events are strictly single-store.
  2. Q2 — category granularity. Spec ships catering | event — the two values 1C needs to map to ribbon categories 2 and 3. The directive's Events category lists "spirit nights, fundraisers, large-party reservations" — those could be event subtypes later. Minimal-per-Sam ships the two-value set; confirm that's enough for now, or name the subtypes if you want them in the precondition.
  3. Q3 — status values. Spec proposes scheduled | confirmed | completed | cancelled. Confirm, or trim/extend — it affects which events 1C surfaces in the ribbon.

7. Distinct-commit boundary

One commit: the ScheduledEvent model + its migration + the test file. A pure-foundation commit — it only adds a model + an empty table, no behavior change to any existing surface. It lands as a precondition alongside the role-taxonomy precondition; together they clear both of aick's pre-1A gates. After it lands + samai-PASSes, 1C can build its Caterings/Events adapters against a real model instead of an undefined dependency.
