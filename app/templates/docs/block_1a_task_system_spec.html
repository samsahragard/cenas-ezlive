Block 1A — Task Model + Task Assignment System

Phase 2 / Block 1 / sub-block 1A. samai spec, 2026-05-13.
Implementation: aick. Review: ck (with samai) + samai checklist PASS per commit.
Source: Phase 2 Directive §BLOCK 1 / 1A (see Phase 2 Directive).

1. Scope — what 1A is, and what it is NOT

1A is the data foundation of the Block 1 ribbon system: the Task model, its append-only audit log, the ribbon-dismissal table, the hierarchy-aware assignment helper, and the two routes that create and reassign tasks. Nothing renders a ribbon in 1A — 1A makes tasks exist, be owned, be reassigned, and be audited.

1A ships:

  • Task, TaskAuditLog, RibbonItemDismissal models + one alembic migration
  • TaskAuditLog append-only before_delete listener
  • A shared app/services/role_hierarchy.py module (see §4 — 1A is first to need it; 1E and 2C consume it next)
  • can_assign_to(actor, target) assignment helper in that module
  • POST /partner/tasks/create and POST /partner/tasks/<id>/reassign routes with route-layer enforcement + audit emission
  • Tests: model behavior, the can_assign_to parameterized matrix, route enforcement, audit emission

1A does NOT ship (distinct-risk-surface boundary — each is its own sub-block / commit):

  • The ribbon Jinja partial or any rendering — that is 1B
  • The ribbon_items_for content router — that is 1C
  • The X / Check controls and the POST /partner/ribbon/dismiss + /check routes — that is 1D. 1A creates the RibbonItemDismissal table; 1D writes to it.
  • The escalation cron that sets escalated_to_user_id / escalated_at — that is 1E. 1A defines those columns; 1E populates them.
  • Any task-completion path — completed_at / completed_by_user_id are defined in 1A's schema but written by 1D's Check handler.

The audit-log action enum is fully defined in 1A (§7) so the table is stable, but 1A's own routes only ever emit created and reassigned. The other enum values are emitted by the sub-blocks that own those write paths.

2. Schema

2.1 Task

class Task(Base):
    __tablename__ = "tasks"

    id                  Integer, primary key
    title               String(200),  NOT NULL          # short: "SPECS liquor order"
    description         Text,          nullable          # longer detail
    owner_user_id       Integer, FK(users.id, ondelete="RESTRICT"), NOT NULL, indexed
    assigned_by_user_id Integer, FK(users.id, ondelete="RESTRICT"), NOT NULL
    store_scope         String(20),    NOT NULL          # tomball | copperfield | both | none
    category            String(20),    NOT NULL          # vendor | catering | event | employee | maintenance | sales | general
    deadline_at         DateTime,      NOT NULL, indexed
    completed_at        DateTime,      nullable, indexed
    completed_by_user_id Integer, FK(users.id, ondelete="SET NULL"), nullable
    escalated_to_user_id Integer, FK(users.id, ondelete="SET NULL"), nullable, indexed
    escalated_at        DateTime,      nullable
    created_at          DateTime,      NOT NULL, default=utcnow
    updated_at          DateTime,      NOT NULL, default=utcnow, onupdate=utcnow

    __table_args__ = (
        Index("ix_tasks_owner_open", "owner_user_id", "completed_at"),
        Index("ix_tasks_escalation_scan", "completed_at", "escalated_at", "deadline_at"),
    )

Column rationale:

  • store_scope / category are String(20) not native ENUMs — same call ck made for submitted_via in BriefFeedback C1: SQLite-compat + simpler migrations. Values validated application-side (§6). Document the allowed values in a module-level constant (_VALID_STORE_SCOPES, _VALID_CATEGORIES) and reference it in both the route validator and the tests.
  • ix_tasks_owner_open (owner_user_id, completed_at) — the hot ribbon query is "my open tasks" (owner_user_id = me AND completed_at IS NULL). Composite covers it.
  • ix_tasks_escalation_scan (completed_at, escalated_at, deadline_at) — 1E's cron scans WHERE deadline_at < now AND completed_at IS NULL AND escalated_at IS NULL every 5 minutes. 1A lays the index now so 1E doesn't have to add it later as a separate migration.
  • escalated_to_user_id indexed — the ribbon also queries "tasks escalated to me."

FK ondelete:

  • owner_user_id, assigned_by_user_id → RESTRICT. These are NOT NULL. The codebase follows archive-not-delete (users get deactivated, never hard-deleted), so RESTRICT should never actually fire — it is a documented safety net. If a delete path is ever written, it must reassign the task first.
  • completed_by_user_id, escalated_to_user_id → SET NULL. Nullable, historical; safe to null on the (non-expected) user delete.

No hard-delete path for Task in v1. Tasks are completed (completed_at set), not deleted. There is no DELETE route and no before_delete listener on Task itself — but see §3: TaskAuditLog's FK + listener make a Task with audit history effectively undeletable, which is the intended property.

2.2 TaskAuditLog

class TaskAuditLog(Base):
    __tablename__ = "task_audit_log"

    id            Integer, primary key
    task_id       Integer, FK(tasks.id, ondelete="RESTRICT"), NOT NULL, indexed
    actor_user_id Integer, FK(users.id, ondelete="RESTRICT"), NOT NULL
    action        String(20), NOT NULL    # created | assigned | completed | dismissed | reassigned | escalated
    details       JSON,       nullable
    created_at    DateTime,   NOT NULL, default=utcnow, indexed

# Append-only: before_delete listener raises RuntimeError (see §3).
  • task_id ondelete=RESTRICT (not CASCADE): a Task with audit rows cannot be DB-deleted. Combined with archive-not-delete this means Tasks are immutable-once-audited — the right property for an operational audit trail. RESTRICT also avoids the ORM-event-vs-DB-cascade ambiguity (DB-level CASCADE bypasses the SQLAlchemy before_delete event).
  • actor_user_id ondelete=RESTRICT — same archive-not-delete reasoning; the actor on an audit row must remain resolvable.
  • details JSON shape is per-action — see §7.
  • created_at indexed — audit surfaces and the future agent-audit page query by time.

2.3 RibbonItemDismissal

class RibbonItemDismissal(Base):
    __tablename__ = "ribbon_item_dismissals"

    id           Integer, primary key
    user_id      Integer, FK(users.id, ondelete="CASCADE"), NOT NULL
    item_type    String(20), NOT NULL    # task | signal | sales_insight
    item_id      Integer,    NOT NULL    # polymorphic ref — NOT a DB FK (see below)
    dismiss_day  String(10), NOT NULL    # "YYYY-MM-DD" — daily reset boundary
    dismissed_at DateTime,   NOT NULL, default=utcnow

    __table_args__ = (
        UniqueConstraint("user_id", "item_type", "item_id", "dismiss_day",
                         name="uq_ribbon_dismissal_per_day"),
        Index("ix_ribbon_dismissal_lookup", "user_id", "dismiss_day"),
    )
  • item_id is a polymorphic reference — it points at tasks.id, signals.id, or sales_insights.id depending on item_type. It is deliberately not a DB-level FK (a single column can't FK three tables). Referential integrity is application-enforced in 1D's dismiss handler (validate the item exists + the user may dismiss it before INSERT).
  • user_id ondelete=CASCADE — dismissals are ephemeral per-user UI state, meaningless once the user is gone. CASCADE is correct here (unlike Task ownership).
  • Daily-reset decision (the directive flagged this as "samai's call"): use dismiss_day (a "YYYY-MM-DD" string), not a session_id. Rationale: (1) "session" is ambiguous in a web app — cookie session? browser tab? until logout? — whereas a day boundary is operationally meaningful: a manager X's a ribbon item, it's gone for today, it's back tomorrow morning so they re-triage it, matching the daily operational rhythm. (2) No session-tracking infrastructure needed — dismiss_day = date.today().isoformat() at write time. (3) The UniqueConstraint(user_id, item_type, item_id, dismiss_day) makes 1D's dismiss endpoint naturally idempotent (dismissing twice in one day is a harmless no-op). (4) Permanent dismissal of intrinsically time-bound items (sales insights) is handled by 1E's auto-expiry cron, not by this table — so this table only ever needs the "not now, today" semantics.
  • 1A ships the model + migration only. The POST /partner/ribbon/dismiss and /check write routes are 1D.

3. Append-only enforcement on TaskAuditLog

Mirror the established pattern (BriefFeedback C1 / LegalAccessLog): an ORM before_delete event listener on TaskAuditLog that raises RuntimeError with an "append-only" message. 

@_sa_event.listens_for(TaskAuditLog, "before_delete")
def _no_delete_task_audit(mapper, connection, target):
    raise RuntimeError(
        "TaskAuditLog is append-only — task history cannot be deleted. "
        "Corrections are made by appending a new audit row, not by "
        "removing one."
    )

UPDATE is not blocked — but 1A's routes never UPDATE an audit row (they only INSERT), and no other sub-block should either. A review item (§10) checks that no write path mutates an existing TaskAuditLog row.

4. The role-hierarchy foundation — a shared module

Architectural call worth surfacing before implementation: three Phase 2 sub-blocks need the same role-tier / hierarchy logic — 1A's can_assign_to, 1E's "find the owner's immediate manager" escalation step, and 2C's can_view_counseling cascade. The directive references "the existing role hierarchy" for 1E, but app/services/permissions.py only has ROLE_PERMISSIONS (tag grants) — there is no role-tier / hierarchy structure today.

1A is the first sub-block to need it, so 1A creates it: app/services/role_hierarchy.py, consumed by 1E and 2C next. It holds:

  • ROLE_TIER: dict[str, int] — every canonical role mapped to an integer tier (higher = more authority). The tiers, from the directive's §2C hierarchy line: 
    partner            = 5
corporate          = 4   # corporate, corporate_chef
gm                 = 3   # gm, prep_manager        (see open question Q3)
manager-tier       = 2   # km, assistant_km, foh_manager
hourly-tier        = 1   # expo, driver, + (cook/server/busser/host/bartender — see Q1)
  • role_tier(role) -> int — resolves a User.permission_level (including the _LEGACY_ALIASES from permissions.py — "manager"→gm, "corporate-driver"→driver) to its tier. Unknown roles resolve to tier 0 (no authority) rather than raising.
  • role_domain(role) -> str — "kitchen" | "foh" | "both". km/assistant_km/corporate_chef/prep_manager = kitchen; foh_manager = foh; gm/corporate/partner = both. Needed because the assignment rules cross-cut tier with a kitchen/FOH domain (a KM can't assign to a server; an FOH manager can't assign to a cook).
  • immediate_manager(user, db) -> User | None — the lowest-tier user strictly above user's tier in user's store scope. 1E needs this; 1A defines it here so 1E imports rather than re-deriving.

Keeping this in its own module (not buried in permissions.py) keeps the tag-grant concern and the org-hierarchy concern separate — they answer different questions and will evolve independently.

5. can_assign_to — the assignment helper

5.1 Signature

The directive sketched can_assign_to(actor_role, target_role, store_scope). Spec proposes instead: 

def can_assign_to(actor: User, target: User) -> bool:
    """True if `actor` may assign/reassign a task to `target`."""

Reasons: the helper needs actor.store_scope and target.store_scope (the rules say "in their store" / "in their scope"), plus both roles — so passing full User objects is cleaner than four loose primitives, and it matches the existing _user_has(user, ...) convention in permissions.py. The task's own store_scope is validated separately in the route (§6) — it is a property of the task, not of the assignment relationship.

5.2 The assignment rules (from the directive, §1A)

ActorMay assign toStore constraint
partneranyonenone
corporategm and every tier belownone (corporate is store-unscoped)
gmmanager-tier (km / assistant_km / foh_manager) + hourly-tier (expo, driver, …)target must be in the GM's store
km / assistant_kmhourly-tier kitchen staff onlytarget must be in the KM's scope
foh_managerhourly-tier FOH staff + expotarget must be in the manager's store
any rolethemselves (actor.id == target.id)always allowed

Derived rules the helper must encode:

  • Self-assignment short-circuits to True first — every role, including hourly tier, may self-assign.
  • An actor may never assign to a peer at the same tier (gm→gm = False, corporate→corporate = False) — only strictly downward, except self.
  • Tier alone is not sufficient below GM: a KM (kitchen) and an FOH manager (FOH) are the same tier but assign to different domains. The helper checks role_tier(actor) > role_tier(target) and, when the actor is manager-tier, role_domain compatibility.
  • Store constraint: when the actor is store-scoped (gm and below), target.store_scope must intersect actor.store_scope. Partner / corporate / corporate_chef / prep_manager are store-unscoped and skip the store check (corporate_chef + prep_manager pending Q3).

5.3 BLOCKING GAP — escalate to @sam before the km / manager-tier rules can be implemented

The directive's rules ("KM can assign to kitchen staff", "FOH manager can assign to hourly staff") reference cook, server, busser, host, bartender — none of which are roles in the system today. Only expo and driver exist at the hourly tier.

Three ways forward — recommend (A):

  1. (A, recommended) Extend the role taxonomy first, as a Block 1 precondition commit. Add the hourly roles (cook, server, busser, host, bartender) to ROLE_PERMISSIONS with appropriately minimal tag sets, and to ROLE_TIER / role_domain. This is needed anyway for Block 2 (the counseling cascade in 2C and attendance in 2B both reference the same hourly roles), so doing it once up front unblocks the most sub-blocks. samai will spec the taxonomy extension as a tiny standalone commit if Sam +1's this path.
  2. (B) can_assign_to operates over a separate, broader role-tier table decoupled from ROLE_PERMISSIONS — but then two role lists drift apart, which is exactly the spec-drift the permission matrix tests were built to catch. Not recommended.
  3. (C) Ship 1A's can_assign_to with only the partner / corporate / gm / self rules (all of which reference roles that do exist) and defer the km / assistant_km / foh_manager rules to a 1A-follow-up commit once the taxonomy lands. Viable as a stopgap if Sam wants 1A moving immediately, but it leaves the helper visibly incomplete.

Until Sam picks a path: aick can implement everything else in 1A (all three tables, the migration, the append-only listener, the role-hierarchy module skeleton, the partner/corporate/gm/self branches of can_assign_to, both routes, and the tests for all of the above). 6. Routes

1A ships exactly two write routes. Both follow samai methodology rule 1 (audience-eligibility-before-mutation): authorization is checked before any row is written.

6.1 POST /partner/tasks/create

  1. Resolve owner_user_id from the form. If absent, default to current_user.id (self-assignment).
  2. Validate store_scope ∈ _VALID_STORE_SCOPES, category ∈ _VALID_CATEGORIES, title non-empty + ≤200 chars, deadline_at parses to a future-or-present datetime. Any failure → 400.
  3. Load the owner User. If not found / inactive → 400.
  4. Enforce can_assign_to(current_user, owner). False → 403 (do not write the Task row).
  5. INSERT the Task with assigned_by_user_id = current_user.id.
  6. INSERT a TaskAuditLog row, action="created" (see §7 for the details shape).
  7. Return the created task (redirect or JSON per the surface 1B/1C will consume — coordinate with aick; JSON is fine for v1).

6.2 POST /partner/tasks/<id>/reassign

  1. Load the Task. Not found → 404.
  2. Resolve the proposed new_owner_user_id from the form; load that User. Not found / inactive → 400.
  3. Enforce can_assign_to(current_user, new_owner). False → 403 (do not mutate the Task).
  4. Capture from_owner_user_id (the current value), then UPDATE owner_user_id = new_owner.id, assigned_by_user_id = current_user.id, updated_at = utcnow.
  5. INSERT a TaskAuditLog row, action="reassigned".

Permission-tag gating: 1A introduces no new requires_permission tag. Task create/own/reassign is available to every authenticated user — the directive's "anyone can assign to themselves" means hourly-tier users (a driver self-assigning a task) must reach these routes. can_assign_to is the authorization gate for the assignment relationship; the route still runs inside the normal authenticated-session guard. If Sam wants a coarse tasks.use tag on top, that's a one-line follow-up — flagged as Q4, not assumed.

Route file: new blueprint app/web/tasks.py (tasks_bp), registered in app/__init__.py. Keeps the 1A surface a clean, self-contained review target — same shape as the briefs blueprint in calibration C2.

7. TaskAuditLog action emission map

actionEmitted bydetails JSON shape
created1A — create route{owner_user_id, store_scope, category, deadline_at, title}
reassigned1A — reassign route{from_owner_user_id, to_owner_user_id, reassigned_by_user_id}

escalated1E — escalation cron{escalated_to_user_id, tier, deadline_at}
assignedreserved — Block 3B delegation-approval execution path{owner_user_id, via_delegation_request_id}

Note on dismissed: the audit-log dismissed action means a task was cancelled / killed — a permanent state change to the Task. It is not the same as the ephemeral ribbon X (which writes a RibbonItemDismissal row and resets daily). Two different concepts that happen to share the word. 1A defines the enum value; the task-cancellation route that emits it is 1D's (or a later sub-block's) — flagged as Q5 since the directive doesn't fully specify a cancel path.

8. Test requirements

1A's test file: tests/test_task_system.py. Required coverage:

  • Model: Task / TaskAuditLog / RibbonItemDismissal round-trip; TaskAuditLog before_delete raises RuntimeError matching /append-only/; FK ondelete declarations assert correctly (RESTRICT on owner/assigned_by/task_id/actor; SET NULL on completed_by/escalated_to; CASCADE on dismissal.user_id); the uq_ribbon_dismissal_per_day unique constraint rejects a duplicate.
  • can_assign_to matrix: a parameterized test over (actor_role × target_role × same-store / cross-store), asserting the gate decision against the §5.2 table — the same shape as the permission matrix in test_permission_matrix.py (2255c1d). Use a _test_id()-style self-describing id (actor=gm__target=expo__store=same__expected=allow) so a failure names the exact combo. Include the explicit self-assignment case for every role. If Sam picks path (C) in §5.3, the km/foh_manager rows are marked xfail with a reason until the taxonomy lands.
  • Routes: create with an allowed assignment → 200 + Task row + created audit row; create with a disallowed assignment → 403 + no Task row written (audience-eligibility-before-mutation); reassign allowed → 200 + owner changed + reassigned audit row with correct from/to; reassign disallowed → 403 + Task unchanged; invalid store_scope/category/deadline_at → 400.
  • Audit emission: assert the details JSON shape per §7 for the two actions 1A emits.

  1. Scope: only the 1A surfaces touched — no ribbon partial, no cron, no dismiss/check/complete route.
  2. Three tables match §2 column-for-column: types, nullability, indexes, FK ondelete.
  3. TaskAuditLog before_delete listener present and raising; no write path UPDATEs an existing audit row.
  4. RibbonItemDismissal uses dismiss_day (not session_id) + the unique constraint + the lookup index.
  5. app/services/role_hierarchy.py exists with ROLE_TIER, role_tier, role_domain, immediate_manager; legacy aliases resolved; unknown role → tier 0, not a raise.

  7. Both routes enforce can_assign_to before any write (rule 1); 403 leaves no row written.
  8. Route validation: store_scope / category / title / deadline_at validated, 400 on bad input.
  9. Audit rows emitted with the §7 details shape for created and reassigned.
  10. tasks_bp registered in app/__init__.py; migration present and reversible.
  11. Tests per §8 all green; full suite no regression.
  12. Deploy-verify (rule 3) after aick's push: app boots, a representative authenticated route still 200s.
