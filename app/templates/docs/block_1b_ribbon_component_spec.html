1. Scope — what 1B is, and what it is NOT

1B is the rendering shell of the universal ribbon: the Jinja partial, its placement in base.html, the seven-category fixed layout, the per-user per-category collapse preference, and the X / Check control markup. 1B renders whatever the content router hands it — it does not decide which items appear.

1B ships:

  • app/templates/partials/_ribbon.html — the partial
  • The base.html include (after the topbar, before page content)
  • RibbonCategoryPreference model + one alembic migration
  • POST /partner/ribbon/collapse/<category> — the collapse-toggle endpoint
  • The X / Check control markup on each item (data-attributes + button structure) — inert in 1B
  • Tests: partial renders against a stubbed router, collapse toggle round-trips, base.html include is defensive

1B does NOT ship (distinct-risk-surface boundary):

  • ribbon_items_for() — the content router is 1C (aick). 1B builds against a stub that returns [].
  • RibbonItem the class + render_for(user) — 1C defines and implements them. 1B depends only on the render contract in §2.
  • The X / Check behavior — the ribbon.js fetch handler and the POST /partner/ribbon/dismiss + /check endpoints are 1D (ck's next sub-block). 1B renders the buttons; 1D makes them live.
  • The escalation cron, sales-insight synthesis, any data production — those are 1E / 1F.

Parallelization (ck's 23:52 question): 1B is safe to build in parallel with aick's 1A. 1B renders RibbonItem render-contract objects (§2), not Task rows directly — so a Task schema change during 1A review does not ripple into the 1B partial. The only hard pre-req for 1B is this spec + the render contract being agreed. Build the partial against a stub ribbon_items_for that returns []; wire the real call when 1C lands.

2. The RibbonItem render contract — the shared interface

This is the contract between 1B (renderer) and 1C (producer). Both sub-blocks build against it; 1C implements the class, 1B's partial only ever touches these attributes / methods. Any change to this contract is a coordinated change across 1B + 1C and gets re-specced. 

# Produced by 1C's ribbon_items_for(); consumed by 1B's _ribbon.html.
class RibbonItem:
    category:    str    # one of RIBBON_CATEGORIES (§3) — which section it renders in
    severity:    str    # "info" | "warn" | "alert" — drives styling + (in 1C) sort
    item_type:   str    # "task" | "signal" | "sales_insight" — drives X/Check targets
    item_id:     int    # the underlying row id in the table named by item_type
    deadline_at: datetime | None   # nullable; used by 1C for sort, shown by 1B if present
    can_dismiss: bool   # whether the X control renders for this item
    can_check:   bool   # whether the Check control renders for this item

    def render_for(self, user) -> dict:
        """Role-aware framing (1C's cross-cutting rule). Returns:
           { "text": str,            # the primary line
             "sub_text": str | None, # optional secondary line (owner/status framing)
             "styling_class": str }  # CSS class hook, e.g. "ribbon-item--alert"
        """

1B's partial: for each item, call item.render_for(current_user), render text + optional sub_text with styling_class, and emit the X / Check markup keyed by item_type + item_id (§6), gated by can_dismiss / can_check. 1B does not sort — 1C returns the list already sorted (severity DESC, deadline ASC per the directive). 1B renders in received order.

3. The seven categories — fixed vertical order

Module-level constant (define in app/services/ribbon.py so 1B + 1C share it; 1C owns that module, but the constant is needed by 1B's partial first — coordinate, or 1B defines it and 1C imports):

RIBBON_CATEGORIES = [
    ("todo",        "To-do"),
    ("caterings",   "Caterings"),
    ("events",      "Events"),
    ("employee",    "Employee"),
    ("vendors",     "Vendors"),
    ("maintenance", "Maintenance"),
    ("sales",       "Sales"),
]
  • Fixed order, always all seven. The partial renders all seven category headers every time, even when a category is empty — predictable structure + stable collapse toggles. An empty category renders a muted empty-state line ("Nothing here right now"), not a hidden section.
  • Note the slug vs. Task.category mismatch: ribbon slugs are mostly plural (caterings, events, vendors) while Task.category is singular (catering, event, vendor) and includes general. The mapping from a Task/Signal/Insight to its ribbon category is 1C's content-router concern — 1B only needs the seven ribbon slugs above. general-category tasks surface under todo, never their own section (1C handles that).
  • Each category section: a header (label + item count badge + collapse chevron) and a body (the items, or the empty-state).

4. The partial — app/templates/partials/_ribbon.html

  • Iterates RIBBON_CATEGORIES in order. For each: render the header, then if not collapsed-for-this-user render the body.
  • Calls the router once: {{ ribbon_items_for(page_slug, current_user, store_scope) }} — registered as a Jinja global by 1C. During the stub period, 1B registers a stub global that returns [] so the partial renders an all-empty ribbon without erroring. 1C replaces the stub with the real implementation.
  • Groups the returned flat list by item.category into the seven buckets. (Or: call the router per-category with the category= kwarg — 1C's signature supports both. 1B's call: one call, group client-side in the template — fewer round-trips. Confirm with 1C which it prefers; spec default = one call + group.)
  • Per-category collapse state read from RibbonCategoryPreference (§5). Default (no row) = expanded.
  • Defensive: the entire partial body is wrapped so that any failure inside it (router raises, item missing an attribute) renders an empty ribbon and logs at WARN — it must never 500 a page. The ribbon is on every authenticated page; a ribbon bug cannot take down the app.

4.1 base.html include

  • Include _ribbon.html in base.html immediately after the topbar partial and before the {% block content %} wrapper.
  • The include is itself guarded: {% if current_user %} (authenticated pages only — the ribbon never renders on the login / passcode pages).
  • This is a shared-surface touch — every authenticated page now renders the ribbon. The 1B review (samai rule 3, deploy-verify) explicitly checks a representative sample of existing pages (dashboard, an orders page, a developer-docs page) still 200 after the include lands.

5. RibbonCategoryPreference — per-user collapse state

class RibbonCategoryPreference(Base):
    __tablename__ = "ribbon_category_preferences"

    id           Integer, primary key
    user_id      Integer, FK(users.id, ondelete="CASCADE"), NOT NULL
    category     String(20), NOT NULL    # one of the seven RIBBON_CATEGORIES slugs
    is_collapsed Boolean,    NOT NULL, default=False
    updated_at   DateTime,   NOT NULL, default=utcnow, onupdate=utcnow

    __table_args__ = (
        UniqueConstraint("user_id", "category", name="uq_ribbon_pref_user_category"),
    )
  • Absence of a row = expanded (the default state). A row with is_collapsed=True = collapsed. This means a fresh user has zero preference rows and sees everything expanded — no backfill needed.
  • user_id ondelete=CASCADE — pure per-user UI state, meaningless once the user is gone (same reasoning as RibbonItemDismissal in 1A).
  • UniqueConstraint(user_id, category) — one preference row per user per category; the toggle endpoint upserts against it.
  • category is validated against the seven RIBBON_CATEGORIES slugs in the toggle endpoint — reject anything else with 400.

6. The collapse-toggle endpoint + the X/Check markup contract

6.1 POST /partner/ribbon/collapse/<category>

  1. Validate <category> ∈ the seven slugs → 400 otherwise.
  2. Upsert RibbonCategoryPreference for (current_user.id, category): if no row, INSERT with is_collapsed=True; if a row exists, flip is_collapsed.
  3. Return the new state (JSON {category, is_collapsed}) so the JS can update the chevron without a reload.
  4. Authenticated-session guard only — no new permission tag (any user may collapse their own ribbon). See §8.

6.2 X / Check markup contract — what 1B renders, what 1D wires

1B renders the controls; 1D (ck's next sub-block) ships ribbon.js + the dismiss / check endpoints that make them live. For the wiring to "just work" when 1D lands, 1B must emit exactly this markup contract per item: 

<div class="ribbon-item {{ payload.styling_class }}"
     data-item-type="{{ item.item_type }}"
     data-item-id="{{ item.item_id }}">
  <span class="ribbon-item__text">{{ payload.text }}</span>
  {% if payload.sub_text %}<span class="ribbon-item__sub">{{ payload.sub_text }}</span>{% endif %}
  {% if item.can_dismiss %}
    <button class="ribbon-item__x" data-action="dismiss" aria-label="Dismiss">×</button>
  {% endif %}
  {% if item.can_check %}
    <button class="ribbon-item__check" data-action="check" aria-label="Mark complete">✓</button>
  {% endif %}
</div>
  • 1D's ribbon.js delegates clicks on [data-action], reads data-item-type + data-item-id off the closest .ribbon-item, and POSTs to /partner/ribbon/dismiss/<item_type>/<item_id> or /check/<item_type>/<item_id>.
  • In 1B the buttons are inert — no JS yet. They render, they're styled, they do nothing on click until 1D ships. That's the intended shell state.
  • 1B owns the CSS for .ribbon-item, .ribbon-item__x, .ribbon-item__check, the category headers, collapse chevrons, and the severity styling classes (ribbon-item--info / --warn / --alert). Put it in the existing stylesheet or a dedicated ribbon.css — ck's call, note it in the commit.

7. Mobile / graceful degradation

  • The ribbon is on every authenticated page, so it must degrade on narrow viewports. Spec minimum: on mobile widths the ribbon is collapsed by default (all seven categories collapsed) with the count badges still visible, so it's a compact triage summary that expands on tap rather than a wall of items pushing page content down.
  • Exact mobile treatment (drawer? accordion? sticky?) is ck's implementation call within the partial — the spec requires only that it not dominate the viewport and not error. Flag the chosen treatment in the commit message.

8. No new permission tag — rationale

The directive: the ribbon is "role-aware but not role-restricted — all management tier sees the same operational signals." The ribbon renders for every authenticated user; what it contains is filtered by 1C's router (role + store scope). So 1B introduces no requires_permission tag — the partial renders inside the normal authenticated-session guard, and the collapse endpoint likewise. The role-filtering of content is 1C's concern, not 1B's.

9. Test requirements

1B's test file: tests/test_ribbon_component.py. Required coverage:

  • Partial renders against a stub: with ribbon_items_for stubbed to return [], the partial renders all seven category headers + seven empty-states, no error.
  • Partial renders items: with the stub returning a few fake render-contract objects across categories, each lands in the correct category section, X/Check markup present iff can_dismiss/can_check, styling_class applied.
  • Collapse preference: no row → category renders expanded; is_collapsed=True row → body not rendered, chevron in collapsed state. Toggle endpoint: first POST creates a row is_collapsed=True; second POST flips to False; invalid category → 400; unauthenticated → redirect to login.
  • base.html include is defensive: a router stub that raises still renders the page (empty ribbon + WARN log), does not 500. This is the most important 1B test — it proves a ribbon bug can't take down every page.
  • Deploy-verify surface (samai rule 3): after ck's push, a representative sample of existing authenticated pages (dashboard, an orders page, a developer-docs page) still return 200 with the ribbon include live.

10. samai review checklist (run per commit)

  1. Scope: only the 1B surfaces — no router logic, no X/Check endpoints, no ribbon.js, no data production.
  2. _ribbon.html renders all seven categories in fixed order, always (empty categories show the muted empty-state, not hidden).
  3. The partial consumes only the §2 render contract — no direct Task / Signal / SalesInsight attribute access.
  4. Stub ribbon_items_for registered as a Jinja global returning []; partial renders clean against it.
  5. RibbonCategoryPreference matches §5: types, CASCADE on user_id, unique constraint; absence-of-row = expanded.
  6. Collapse endpoint upserts correctly, validates category, returns JSON, no new permission tag, authenticated-guard only.
  7. X/Check markup matches the §6.2 contract exactly (data-attributes, button classes, can_dismiss/can_check gating) so 1D's ribbon.js wires with zero markup changes.
  8. base.html include placed after topbar, before content, guarded by {% if current_user %}.
  9. Defensive rendering verified: a raising router does not 500 the page (the §9 test).
  10. Mobile: ribbon collapsed-by-default on narrow viewports, count badges visible, doesn't dominate the viewport.
  11. Migration present + reversible; tests per §9 green; full suite no regression.
  12. Deploy-verify (rule 3): representative existing pages still 200 with the include live.

11. Open questions for Sam / coordination notes

  1. Q1 (coordination, not blocking): the RIBBON_CATEGORIES constant + the stub ribbon_items_for Jinja global — these straddle 1B and 1C. Spec default: 1B defines both (the constant in app/services/ribbon.py, the stub global in app init); 1C replaces the stub with the real router and imports the constant. ck + aick confirm this split when 1C starts so neither double-defines.
  2. Q2: one router call + group-in-template vs. seven per-category calls. Spec default = one call + group (fewer round-trips). 1C's signature supports both; confirm with aick.
  3. Q3: page_slug + store_scope — the partial passes these to the router. page_slug needs to be available in every template's context. Is there an existing context processor that exposes the current page slug + the user's store scope, or does 1B need to add one? If 1B adds a context processor, that's a small additional 1B surface — flag it in the commit. (samai's read: check for an existing one first; the developer-docs pages already track an active slug, there may be a pattern to reuse.)
  4. Q4: CSS location — dedicated ribbon.css vs. appending to the existing stylesheet. ck's call; spec just requires the severity + collapse + item styling exist and are scoped so they don't bleed into other components.

12. Distinct-commit boundary recap

1B is one commit: the partial + base.html include + RibbonCategoryPreference model + migration + collapse endpoint + X/Check markup (inert) + CSS + tests. It does not ship the content router (1C), the X/Check behavior or ribbon.js or the dismiss/check endpoints (1D), or any data production (1E/1F). 1B is fully implementable now, in parallel with aick's 1A — its only dependency is this spec and the §2 render contract being agreed.
