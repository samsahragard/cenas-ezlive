Block 1C — Ribbon Content Router

Phase 2 / Block 1 / sub-block 1C. samai spec, 2026-05-14.
Implementation: aick. Review: samai checklist PASS per commit.
Source: Phase 2 Directive §BLOCK 1 / 1C (see Phase 2 Directive). Consumes the render contract from Block 1B spec §2.

1. Scope — what 1C is, and what it is NOT

1C is the content brain of the ribbon: the module that decides which items appear on whose ribbon on which page, and how each is framed. 1B renders; 1C decides.

1C ships:

  • app/services/ribbon.py — the module (1B may have created a stub file + the RIBBON_CATEGORIES constant; 1C fills it in — confirm the split with ck per 1B spec Q1)
  • The RibbonItem class, implementing the 1B §2 render contract + the relation field (§3)
  • ribbon_items_for(page_slug, user, store_scope, category=None) -> list[RibbonItem] — the router (§4)
  • RibbonItem.render_for(user) — the cross-cutting framing rule (§5)
  • Source adapters: Task, Signal, SalesInsight, ScheduledEvent → RibbonItem (§6)
  • Replacing 1B's stub Jinja global with the real ribbon_items_for (§9)
  • Tests: the framing matrix, the filtering rules, the sort, the dismissal exclusion

1C does NOT ship: the partial or any rendering (1B); the X/Check endpoints or ribbon.js (1D); the escalation cron (1E); SalesInsight production (1F — 1C reads the table, 1F writes it); the pay-masking helper (1H — 1C calls it).

Dependency: 1C has a HARD dependency on 1A — it reads Task and RibbonItemDismissal. 1C can be specced and its Signal branch partially built now (the anomaly engine's Signal model already exists), but the Task branch + the dismissal-exclusion join cannot be finished until 1A lands and is samai-PASSed. Sequencing: 1C starts in earnest once 1A is PASSed.

2. The module — app/services/ribbon.py

One module. Holds RIBBON_CATEGORIES (shared with 1B — see coordination note §10), the RibbonItem class, the ribbon_items_for router, the per-source adapter functions, and the sort + filter helpers. No route handlers live here — 1C is pure service-layer logic, called by 1B's partial via the Jinja global.

3. The RibbonItem class

Implements the 1B §2 render contract exactly, plus one field 1C needs internally for framing:

                           #   (private — render_for reads it; the partial never does)

    def render_for(self, user) -> dict:
        # returns { "text", "sub_text", "styling_class" } per §5

One source row can yield more than one RibbonItem. Andres's "SPECS liquor order" task, when Andres is on the Vendors page, produces two items: one in the todo category (relation="owner_todo") and one in the vendors category (relation="owner_category") — same underlying Task, two ribbon rows, two framings. This is intentional and is the mechanism behind the cross-cutting rule. item_type + item_id are identical on both; 1D's X/Check acts on the underlying row regardless of which rendered copy was clicked.

item_type note: the 1B render contract listed task | signal | sales_insight. 1C adds scheduled_event as a fourth — it must be in the contract. This is a coordinated contract change with 1B (§10); flag it so ck's 1B markup + 1D's X/Check handler both account for the fourth type.

4. ribbon_items_for — the router

def ribbon_items_for(page_slug, user, store_scope, category=None) -> list[RibbonItem]:
    """All ribbon items relevant to (page_slug, user, store_scope).
       category=None → all seven categories; else → just that one.
       Returns the list ALREADY SORTED (severity DESC, deadline ASC)."""

Algorithm, in order:

  1. Gather candidate source rows from the four sources, each filtered by the role + store-scope rules in §7. Tasks owned-by / escalated-to user; open Signals in scope; live SalesInsights in scope; upcoming ScheduledEvents in scope.
  2. Adapt each source row into one or two RibbonItems with the right category + relation (§6).
  3. Apply the page-relevance filter (§7.3) — on the main dashboard, drop everything except alert-severity items and the user's todo; on a domain page, keep that domain's full detail.
  4. Exclude dismissed items — drop any item whose (user.id, item_type, item_id, today) has a RibbonItemDismissal row (§7.4).
  5. Filter to category if the arg is set.
  6. Sort (§8) and return.

5. The cross-cutting framing rule — render_for(user)

Sam's "SPECS by Andres" example, generalized. The same item is framed by the viewer's relation to it. relation is set by the router when it adapts the source row; render_for switches on it:

relationWhen the router sets itFramingExample text
owner_todo viewer owns the item, rendering in the todo category action-required — imperative, what to do + when "Place SPECS order by 5pm"
owner_category viewer owns the item, rendering in its domain category (e.g. vendors) informational — you already see it actionably in To-do, so here it's just context "Owner: you, due 5pm"
observer viewer does not own it, but it's in scope on a domain page status — who owns it + what's pending "SPECS order to be placed by Andres at 5pm"
escalated_to the item escalated to the viewer (Task.escalated_to_user_id == user.id) escalation — flags it's late + landed on them + who the original owner is "ESCALATED: SPECS order, was due 5pm, owner Andres"

The "manager's to-do shows nothing" rule: a manager does not see a subordinate's task in their own todo category — only in the domain category as an observer, unless it escalated to them (then it's escalated_to and does appear in their todo). The router enforces this by only ever assigning relation="owner_todo" when item._source.owner_user_id == user.id, and relation="escalated_to" when item._source.escalated_to_user_id == user.id.

render_for returns { text, sub_text, styling_class }. styling_class is severity-driven (ribbon-item--info / --warn / --alert) with an optional relation modifier (e.g. ribbon-item--escalated). Signals and SalesInsights have simpler framing (they have no "owner" — they're always observer-style status text); the four-way table above is primarily for Tasks. render_for branches on item_type first, then relation.

6. Source adapters

Source→ which ribbon categoryrelation logiccan_dismiss / can_check
Task todo if owned-by / escalated-to viewer; and/or its category mapped to a ribbon slug (vendor→vendors, catering→caterings, event→events, employee→employee, maintenance→maintenance, sales→sales; general → todo only) owner_todo / owner_category / observer / escalated_to per §5 dismiss: yes (X = "not now today"). check: yes (Check = complete the task) — but only for the owner or escalated-to viewer; an observer gets can_check=False

Each adapter is its own function (_task_to_items, _signal_to_item, _sales_insight_to_item, _scheduled_event_to_item) so they're independently testable and 1C can ship the Signal adapter before 1A lands.

7. Filtering

7.1 Role + store-scope

  • Store-scoped users (gm and below) see only items whose store scope intersects their store_scope. A Tomball GM never sees Copperfield's vendor tasks or Copperfield-scoped signals.
  • Partner / corporate / corporate_chef / prep_manager are store-unscoped — they see both stores' items. (Same store-scope model as can_assign_to in 1A — reuse role_hierarchy helpers where they fit rather than re-deriving.)
  • Tasks: a user sees their own (owner_user_id == user.id), ones escalated to them, and — on domain pages — in-scope tasks as observer. They never see out-of-scope tasks.

7.2 Pay-data masking — coordination with 1H

Any item in the sales category that surfaces labor-cost data must run through 1H's render_labor_breakdown(store, user) so manager-tier pay is aggregated for non-partner viewers. 1C does not re-implement masking — it calls 1H's helper. This is a hard coordination point: 1C's sales adapter cannot be considered complete until 1H's helper exists. Sequence 1C's sales branch after 1H, or stub the call and wire it when 1H lands.

7.3 page_slug relevance (the directive's "page-relevance principle")

  • Main dashboard (page_slug = the dashboard): return only severity="alert" items + the viewer's todo category. High-level only — no bulk operational detail.
  • Domain page (orders, roster, maintenance, vendors, etc.): return that domain's full detail (all severities) plus the viewer's todo scoped to that domain's category.
  • 1C needs a page_slug → relevant_categories map. Spec proposes 1C owns that map as a module constant; it will need a row per real page slug. The set of page slugs that exist is itself a small open question (§13 Q2) — the map can start with the known pages and default unknown slugs to "main-dashboard behaviour" (alert + todo only), which is the safe default.

7.4 Dismissal exclusion

After adapting, drop any RibbonItem for which a RibbonItemDismissal row exists with user_id == user.id AND item_type == item.item_type AND item_id == item.item_id AND dismiss_day == date.today().isoformat(). One query up front (load today's dismissals for the user into a set), then filter in memory — don't N+1.

8. Sorting

Within the returned list (and within each category when 1B groups): severity DESC then deadline ASC. Severity rank alert=3, warn=2, info=1. deadline_at IS NULL sorts last within its severity band (no deadline = least time-urgent). Spec the sort key as a module helper _ribbon_sort_key(item) so the test asserts it directly.

9. Jinja global registration

1C replaces 1B's stub global with the real ribbon_items_for. Registration happens in create_app() — and per the bee8cdc lesson (the import-shadow bug from Phase 0), do the import at module scope with a clear alias, not inside create_app in a way that rebinds app. Deploy-verify (rule 3) must confirm the app boots and the ribbon renders real content after the swap, because this is the commit where the stub becomes live on every authenticated page.

11. Test requirements

1C's test file: tests/test_ribbon_router.py. Required coverage:

  • Framing matrix: the same Task, viewed by (owner on dashboard / owner on its domain page / observer-manager on the domain page / escalated-to manager) → asserts the four relation values and the four framings from §5. Self-describing test ids (relation=owner_todo etc.).
  • One-source-two-items: owner on the matching domain page gets exactly two items (todo + category) for the one Task, both with the same item_type/item_id.
  • Role + store-scope filter: a Tomball GM does not get Copperfield items; partner gets both stores'.
  • page-relevance: dashboard returns only alerts + todo; a domain page returns full detail for that domain.
  • Dismissal exclusion: an item with a today-dated RibbonItemDismissal row for the viewer is absent; a yesterday-dated one is present again (daily reset).
  • Sort: _ribbon_sort_key orders alert-before-warn-before-info, soonest-deadline-first within band, null deadline last.
  • Per-adapter: each of the four adapters tested in isolation; the Signal adapter test can run before 1A lands.
