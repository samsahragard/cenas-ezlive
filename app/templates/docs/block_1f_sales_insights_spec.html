Block 1F — Sales Insights Synthesis

Phase 2 / Block 1 / sub-block 1F. samai spec, 2026-05-14.
Implementation: aick. Review: samai checklist PASS per commit.
Source: Phase 2 Directive §BLOCK 1 / 1F.

1. Scope — what 1F is, and what it is NOT

1F is the producer of the ribbon's Sales category: a daily 5am-CT cron job that pulls external intelligence (weather, events, school calendars, traffic, outages, local news) and synthesizes it — via a Haiku normalize → Opus synthesize pipeline — into structured SalesInsight rows. The ribbon's 1C router reads those rows; 1F writes them.

1F ships:

  • SalesInsight model + one alembic migration
  • A daily cron endpoint (POST /cron/sales-insights, CRON_TOKEN-protected) + its 5am-CT registration
  • app/services/sales_insights.py — the synthesis pipeline (parallel source adapters → Haiku normalize → Opus synthesize → row insert)
  • The seven source adapters, each behind a clean interface (some wire-complete now, some stubbed pending credentials — §7)
  • Tests: the pipeline orchestration, the synthesis-output → row mapping, the valid_until_at rules, cost logging

1F does NOT ship: the ribbon rendering or the Sales-category routing (1C reads SalesInsight); the auto-expiry scan (1E's every-5m cron expires rows past valid_until_at — 1F just sets the column); the X/Check dismissal endpoints (1D).

Two-part shippability (like 1H): the pipeline architecture + the model + the cron + the Opus synthesis stage + auto-expiry-column logic are all buildable + testable now, with the source adapters mocked in tests. The real adapter wiring for the paid sources waits on Sam providing credentials (§7). Ship the skeleton + synthesis real + tested; wire the paid adapters as credentials land. A 1F that runs with three of seven adapters live (the free ones + Claude-search) still produces useful insight rows on day one.

2. The SalesInsight model

class SalesInsight(Base):
    __tablename__ = "sales_insights"

    id            Integer, primary key
    created_at    DateTime, NOT NULL, default=utcnow, indexed
    valid_until_at DateTime, NOT NULL, indexed   # 1E's auto-expiry scans this
    category      String(24), NOT NULL
        # weather | events | school_calendar | traffic | outage
        #   | yoy_comparison | ai_synthesized
    store_scope   String(20), NOT NULL           # tomball | copperfield | both
    severity      String(10), NOT NULL           # info | warn | alert
    headline      String(200), NOT NULL          # short, ribbon-renderable
    detail        Text, nullable                 # longer, for click-through
    source_url    String(500), nullable          # optional link to the source
    dismissed_by  JSON, nullable                 # list[int] of user_ids — see §2.1

    __table_args__ = (
        Index("ix_sales_insights_live", "valid_until_at", "store_scope"),
    )
  • category / store_scope / severity are String not native ENUM — same call as Task / BriefFeedback; values validated in the synthesis writer against module-level constants.
  • ix_sales_insights_live (valid_until_at, store_scope) — both the 1C ribbon query ("live insights for this store") and 1E's expiry scan ("rows past valid_until_at") hit these two columns. One composite covers both.
  • No FK to User on dismissed_by — it's a JSON list, not a relation (§2.1).

2.1 dismissed_by — and the dual-dismissal reconciliation

dismissed_by is a JSON list of user_ids. It is the permanent dismissal channel. There are two ways a user removes a sales insight from their ribbon, and they are intentionally different (per the directive's 1D):

  • X (dismiss) → a RibbonItemDismissal row (from 1A), item_type="sales_insight", daily reset — "not now, today." The insight comes back tomorrow if still valid.
  • Check (mark done) → appends the user's id to SalesInsight.dismissed_by, permanent — the directive: "for sales insights, [Check] dismisses permanently (insights are intrinsically time-bound)." That user never sees this specific insight again.

1F ships the dismissed_by column only. 1D wires both write paths (the X handler → RibbonItemDismissal, the Check handler → append to dismissed_by). 1C's router, when it reads SalesInsight, filters out rows where the viewer's id is in dismissed_by and rows with a same-day RibbonItemDismissal. Flag this dual-channel to 1C + 1D so all three sub-blocks agree on it — it is the one genuinely subtle part of the Sales category's dismissal model.

3. The cron job

  • POST /cron/sales-insights — CRON_TOKEN Authorization header required, same pattern as /cron/anomaly-brief. Not user-facing.
  • Registered to fire daily at 5am CT (the directive's time). Use whatever cron registration the codebase already uses for the morning-brief 7:30-CT job — reuse, don't reinvent. aick: confirm the existing daily-cron registration mechanism and mount 1F's job the same way.
  • The handler calls the synthesis pipeline (§4), inserts the resulting SalesInsight rows, and returns a JSON summary (rows produced per category, per store, total token cost) so a dry-run or a manual trigger is inspectable — same shape as the brief cron's dispatch_results.
  • Idempotency: if the cron is triggered twice in one day (catch-up after a missed 5am, or a manual re-run), it should not double-insert. Spec proposes: the synthesis writer first expires/supersedes any still-live rows from the same created_at date before inserting the new batch — so a re-run replaces rather than duplicates. Confirm this is the wanted behavior (§12 Q1).

4. The synthesis pipeline

Two stages, in app/services/sales_insights.py:

Stage 1 — parallel source fetch + normalize

  • Seven source adapters (§5), each a function with a uniform signature: fetch(store_locations) -> list[RawSignal] where RawSignal is a small common shape {source, store_scope, raw_text, structured, source_url}.
  • Adapters run in parallel (thread pool — they're I/O-bound HTTP calls).
  • Haiku's role: the directive says "Haiku 4.5 pulls raw data from each source." Interpreted: the adapters that fetch unstructured data (the CenterPoint outage-map scrape; the Claude-search local-news/Reddit results) use Haiku 4.5 to normalize their messy output into the compact RawSignal shape. The adapters fetching structured APIs (OpenWeatherMap, Ticketmaster, Google Calendar, Google Maps, NOAA) are plain HTTP + parse — no model needed. Flag this interpretation in §12 Q2; it's the reading that makes "Haiku pulls" operationally sensible.
  • An adapter that fails (API down, no credentials yet) returns [] and logs WARN — one dead source never breaks the run. This is the same defensive posture as 1B's ribbon: degraded, not broken.

Stage 2 — Opus synthesis

  • Opus 4.7 receives the full normalized RawSignal bundle + the two store contexts (locations, the operational note that these are restaurants and the signal is "does this affect today's covers / dinner rush / driver routes").
  • Opus returns structured output — a list of insight objects, each {category, store_scope, severity, headline, detail, source_url} — using the tool-use / structured-output pattern the morning-brief composer already uses (reuse that pattern; cache_control:ephemeral on the static system-prompt portion).
  • The synthesis writer validates each returned object against the model's allowed values, sets valid_until_at per §6, and inserts the rows.
  • Fallback: if the Opus call fails, the pipeline still inserts the structured signals that don't need synthesis (a NOAA severe-weather alert, a confirmed Astros home game) as plain insight rows via a deterministic _fallback_insights path — mirrors the brief composer's _fallback_brief. A failed Opus call degrades to fewer, blunter insights, not zero.

6. valid_until_at rules — for 1E's auto-expiry

The synthesis writer sets valid_until_at per category so 1E's every-5m cron expires each insight at its natural end-of-relevance:

  • weather → end of the forecast day (23:59 CT today).
  • events / school_calendar → end of the event's date (a Friday football game expires Friday 23:59; a week-long ISD spring break expires on its last day).
  • traffic / outage → short horizon — spec proposes end of day, but a multi-day road closure or a multi-day outage should carry its real end if the source gives one. Default end-of-day if unknown.
  • yoy_comparison → end of day (it's a "today vs last-year-today" insight).
  • ai_synthesized → Opus sets it from the insight's own content (a "this weekend" insight lives until Sunday night); default end-of-day if Opus doesn't specify.

valid_until_at is NOT NULL — every insight must have an expiry, so nothing lingers in the ribbon forever. A synthesis output object with no derivable expiry gets end-of-day as the floor.

8. Cost controls

  • Every synthesis run logs its token cost (Haiku normalize calls + the Opus synthesis call + the Claude-search call) at INFO, and the cron's JSON summary includes total_cost_usd.
  • A soft daily ceiling — spec proposes a SALES_INSIGHTS_COST_CEILING_USD env var (default ~$5, well above the $1–3 projection). If a run's cost exceeds it, the run completes but logs a WARN and the summary flags it — so a runaway cost is visible the next morning, not discovered in a monthly bill. Confirm the ceiling value with Sam (§12 Q3).
  • The directive's projection: ~$1–3/day, ~$50–100/month. The cost log is how we verify the projection holds against reality.

9. Coordination points

  • 1C (ribbon Sales category): 1C's router reads SalesInsight rows — live ones (valid_until_at > now), in the viewer's store scope, not in dismissed_by for that viewer, not same-day RibbonItemDismissal'd. 1F just produces rows; 1C does the read-filtering. The §2.1 dual-dismissal model is the coordination subtlety.
  • 1E (auto-expiry): 1E's every-5m cron scans SalesInsight WHERE valid_until_at < now and clears them (mirrors the anomaly engine's _auto_clear). 1F sets valid_until_at correctly (§6); 1E does the scan. No code dependency, just the shared column.
  • 1D (X/Check): 1D's Check handler appends to dismissed_by; 1D's X handler writes RibbonItemDismissal. 1F ships the column; 1D wires the writes.
  • Morning brief composer: the brief composer's Sales section and 1F's sales insights overlap conceptually. Out of 1F's scope to reconcile, but worth a note: if the calibration round shows the brief's sales section is thin, 1F's SalesInsight rows are a candidate feed for it. Flag for the Round 1 aggregation doc, not for 1F.

10. Test requirements

1F's test file: tests/test_sales_insights.py.

  • Pipeline orchestration: with all seven adapters mocked, the pipeline fetches in parallel, hands the bundle to a mocked Opus, and inserts the returned rows. One dead adapter (raises / returns []) does not break the run.
  • Synthesis output → row mapping: a fixture Opus output (list of insight objects) maps to SalesInsight rows with validated category/store_scope/severity and correctly-set valid_until_at per §6.
  • Fallback path: a failing Opus call still inserts the deterministic structured signals (NOAA alert, confirmed event) via _fallback_insights — degraded, not zero.
  • valid_until_at rules: table-driven over the categories in §6 — weather → end-of-today, event → end-of-event-date, no-expiry-derivable → end-of-day floor, never NULL.
  • Idempotency: running the cron twice in one day replaces rather than duplicates (per §3 / §12 Q1).
  • Cost logging: the run logs total_cost_usd; exceeding the ceiling logs WARN + flags the summary.
  • The three free adapters get real (network-mocked) tests; the four paid adapters get stub-returns-[] tests until wired.
  • Cron auth: POST /cron/sales-insights without the CRON_TOKEN header → 401/403; with it → runs.
