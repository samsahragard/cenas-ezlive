1. Scope — what 1G is, and what it is NOT

1G is the task-based team-reports tab at /partner/team-reports/: per-employee task miss rate, per-manager escalation response time, most-missed task types, and a partner/corporate-only per-store comparison. It is read-only — no mutations — so the entire risk surface is who can see what.

1G ships:

  • app/web/team_reports.py — blueprint team_reports_bp, mounted at /partner/team-reports/
  • Two new permission tags — team_reports.view and team_reports.view_all_stores — added to ROLE_PERMISSIONS (§8)
  • The four v1 reports (§5), computed from Task + TaskAuditLog
  • The sidebar link, gated (§7)
  • One or two templates under app/templates/team_reports/
  • Tests: the role-and-scope gating matrix (§9 — the single most important test in this sub-block)

1G does NOT ship: the manager-tools reports (attendance miss rates, counseling frequency, training pipeline, maintenance spend) — those are 2H, a separate sub-block that adds to this tab later. 1G is the task-based reports only. 1G also does not ship any data production — it reads what 1A's tables already hold.

Dependency: HARD dependency on 1A. Every report reads Task + TaskAuditLog. 1G cannot start until 1A is landed + samai-PASSed. (Per the directive's dependency graph, that is the expected sequence — 1G is not on the parallelizable path the way 1B is.)

2. The permission model — the core of this sub-block

Gating happens at four layers. All four must hold; each is a distinct review item.

LayerRuleMechanism
1. Route access Only partner / corporate / gm reach /partner/team-reports/ at all. Everyone else is redirected to /access-denied (a 302, the requires_permission mechanism — not a literal 403). @requires_permission("team_reports.view") on every route in the blueprint.
2. Sidebar link The nav link renders only for holders of team_reports.view — partner / corporate / gm. The twelve canonical non-holder roles see no link. (See §9 for the exact denied set + the manager-alias note.) {% if has_permission('team_reports.view') %} wrapping the sidebar entry.
3. Store scope (data-level) A GM sees only their own store's team. partner + corporate see all stores. A GM cannot widen scope by any request parameter — see §4. Every report query filters by a server-derived store scope (§4). Defense-in-depth: the decorator grants the tab, the query filter enforces the scope — the same layered shape Sam praised in the 251621f unassign-courier review.
4. Report-level Report #4 (per-store comparison) is partner / corporate only. A GM holding team_reports.view still does not see report #4. Report #4's section + its underlying query are gated by the second tag team_reports.view_all_stores (partner + corporate only).

Why two tags, not one + a role check: team_reports.view = the tab + reports 1–3 (store-scoped for GM). team_reports.view_all_stores = report #4 + the unfiltered cross-store view. This mirrors the existing labor.view_store_summary vs labor.view_all_stores split already in ROLE_PERMISSIONS — established precedent, and it makes the distinction matrix-testable (the test_permission_matrix.py from 2255c1d auto-extends to both new tags). Avoid an inline if user.role in (...) — that is exactly the kind of un-matrix-tested gate the permission spec was built to eliminate.

3. The route surface

Blueprint app/web/team_reports.py, mounted at /partner/team-reports/, registered in app/__init__.py.

  • GET /partner/team-reports/ — the tab. Renders reports 1–3 always (store-scoped per §4); renders report #4 only if the viewer holds team_reports.view_all_stores.
  • Every route in the blueprint carries @requires_permission("team_reports.view") — belt-and-suspenders alongside the blueprint-level guard. No route in this blueprint is reachable without the tag.
  • v1 is a single page. If ck splits reports into sub-routes for performance, every sub-route carries the same decorator + the §4 scope derivation — no sub-route may be a gap.

4. Store-scoping — the defense-in-depth rule

This is the rule most likely to be mis-implemented, so it is stated explicitly:

  • The store scope for every query is derived server-side from current_user, never read from a request parameter:
    • gm → scope = current_user.store_scope (their own store, always — no override path).
    • partner / corporate → scope = all stores (unfiltered), or an optional store filter they choose from a UI control; that control is only rendered for holders of team_reports.view_all_stores.
  • A GM cannot pass ?store=copperfield (or any equivalent) to see another store. If a store parameter is present in the request and the viewer is a GM, it is ignored — the scope is still their own store. The route does not 400 on it, it just doesn't honor it (a 400 would leak that the parameter is meaningful).
  • This is the same shape as ck's 251621f cross-check: the decorator gates the tag, the server-derived scope gates the data — a tampered request cannot get past the second layer.

5. The four v1 reports

All four read from Task + TaskAuditLog only. Each is computed over a window (default 30 days, with a 90-day toggle for reports 1 + 3).

5.1 Report 1 — Per-employee task miss rate

For each employee who owns tasks in scope + window: miss_rate = missed_tasks / total_tasks_owned. "Missed" is defined precisely in §6 — get that definition right, it drives personnel decisions. Show: employee, total tasks, missed count, miss rate %, for both the 30-day and 90-day windows.

5.2 Report 2 — Per-manager escalation response time

For each manager who has had tasks escalated to them (Task.escalated_to_user_id == manager) in scope + window: the latency from escalated_at to the task's completed_at. Report median + mean latency, and count of still-open escalations (escalated, completed_at IS NULL). Median is the headline number — mean is skewed by a single stale task.

5.3 Report 3 — Most-frequently-missed task types

Group missed tasks (per §6) by Task.category, count, in scope + window. Which categories get dropped most. 30/90-day toggle.

5.4 Report 4 — Per-store comparison (partner / corporate only)

Reports 1–3's headline numbers, computed per store, shown side by side (Tomball vs Copperfield). Gated by team_reports.view_all_stores — never rendered, and its queries never run, for a GM.

6. The "miss" definition — get this exactly right

Because miss rate "drives personnel decisions," the definition must be precise and defensible — an employee should never be characterized as having missed a task they actually completed on time.

A task owned by an employee, with deadline_at inside the window, counts as missed if either:

  • it was escalated — escalated_at IS NOT NULL (it passed its deadline without completion and 1E's cron escalated it), OR
  • it was completed late — completed_at IS NOT NULL AND completed_at > deadline_at.

It counts as not missed (a clean task) if:

  • completed_at IS NOT NULL AND completed_at <= deadline_at — completed on time.

Edge cases the spec rules on explicitly:

  • A task still open with deadline_at in the future → not counted at all (neither missed nor clean — it's not yet resolvable). Only tasks whose deadline falls in the window and have passed their deadline OR been completed are in the denominator.
  • A task still open, deadline passed, not yet escalated (inside the <5min escalation-cron lag) → counts as missed (deadline passed, incomplete — the cron lag shouldn't hide a miss).
  • A reassigned task → the miss is attributed to the owner at the time the deadline passed, not whoever owns it now. TaskAuditLog's reassigned rows carry the ownership history; the report must reconstruct point-in-time ownership from them rather than trusting the current owner_user_id. This is the subtlest part of the report — flag it in the commit + the review.

7. Sidebar gating

The nav link to /partner/team-reports/ is wrapped in {% if has_permission('team_reports.view') %} — so only holders of team_reports.view (partner / corporate / gm) see it; the twelve canonical non-holder roles never do. Per samai rule 4 (doc-registration-checks-both-route-and-sidebar) the review verifies both the route decorator and the sidebar gate are present — a route gated but a link still showing, or a link hidden but the route ungated, are both failures.

8. Permission tag registration

Add to app/services/permissions.py ROLE_PERMISSIONS:

  • team_reports.view → corporate set + gm set (partner gets it via the "*" wildcard). Colocated comment explaining the partner/corporate/GM audience + that GM access is store-scoped at the query layer.
  • team_reports.view_all_stores → corporate set only (partner via wildcard). Not in the gm set — this is the line that keeps report #4 + the cross-store view partner/corporate-only.

Both tags are operational reporting tags, not §2.2 global-config tags — GM legitimately holds team_reports.view because their access is store-scoped (gm operating within their store scope, exactly per §2.2). The test_permission_matrix.py suite from 2255c1d auto-extends to both new tags — a row per role confirms the gate; the spec-drift informational test will list them if they end up partner-only-via-wildcard, which they will not be (corporate holds them explicitly).

9. Test requirements

1G's test file: tests/test_team_reports.py. The gating matrix is THE test of this sub-block.

  • Role-and-scope gating matrix: parameterized over every role × the team-tab route. Assert: partner / corporate / gm reach the view; the twelve canonical non-holder roles — corporate_chef, prep_manager, km, assistant_km, foh_manager, expo, driver, cook, server, busser, host, bartender — are redirected to /access-denied (a 302, the requires_permission mechanism since the 2026-05-13 enforce flip — not a literal 403). Legacy aliases: manager resolves to gm via _LEGACY_ALIASES and therefore passes — it inherits gm's full set including team_reports.view; corporate-driver resolves to driver and is denied. The matrix must cover every canonical role and both aliases. Self-describing ids.
  • GM store-scoping: a Tomball GM's report data contains only Tomball employees; a Copperfield GM's only Copperfield. A Tomball GM passing ?store=copperfield still gets Tomball data — the param is ignored, not honored, no 400.
  • Report #4 gating: partner/corporate see the per-store comparison section; a GM does not — and the GM's request never runs the cross-store query (assert via query observation or the section's absence, not just CSS hiding).
  • Sidebar: the link renders for partner / corporate / gm, is absent for the twelve canonical non-holder roles.
  • The "miss" definition (§6): a table-driven test over every §6 case — on-time completion = clean; late completion = missed; escalated = missed; open-future-deadline = not counted; open-deadline-passed-not-yet-escalated = missed.
  • Point-in-time ownership: a task created for A, reassigned to B before its deadline, then missed → the miss is attributed to B (owner when the deadline passed), not A; and vice-versa if reassigned after. This is the §6 subtle case — it gets its own explicit test.

10. samai review checklist (run per commit — extra-careful, highest-stakes permission surface in Phase 2)

  1. Scope: only the 1G surfaces — no 2H manager-tools reports, no data production, no mutations anywhere.
  2. Layer 1: every route in team_reports_bp carries @requires_permission("team_reports.view") — no route is a gap.
  3. Layer 2: the sidebar link is wrapped in has_permission('team_reports.view'); absent for the twelve canonical non-holder roles (rule 4 — both route and sidebar).
  4. Layer 3: store scope is server-derived from current_user, never from a request parameter; a GM cannot widen scope by any param; a stray store= param from a GM is ignored, not honored, not 400'd.
  5. Layer 4: report #4 + the unfiltered cross-store view are gated by team_reports.view_all_stores; a GM's request never even runs the cross-store query.
  6. Both tags added to ROLE_PERMISSIONS correctly: team_reports.view → corporate + gm; team_reports.view_all_stores → corporate only; both reach partner via wildcard; colocated comment present.
  7. The "miss" definition matches §6 exactly, including all four edge cases.
  8. Point-in-time ownership reconstructed from TaskAuditLog reassigned rows — a reassigned task's miss is attributed to the owner-at-deadline, not the current owner.
  9. Median (not just mean) reported for response time.
  10. The gating matrix test + the miss-definition test + the point-in-time-ownership test are all present and green.
  11. Deploy-verify (rule 3): the tab 200s for partner against prod, redirects a non-holder role to /access-denied; verified against prod, not local (rule 2).
  12. Full suite no regression; the permission matrix picks up both new tags green.

11. Open questions for Sam

  1. Q1: the window toggle — spec assumes 30-day default with a 90-day toggle on reports 1 + 3 (the directive says "last 30 / 90 days" for report 1). Report 2 (response time) and report 4 (comparison) — same 30/90 toggle, or fixed 30? Spec assumes the toggle applies tab-wide for consistency. Confirm.
  2. Q2: "employee" in report 1 — is it every Task.owner regardless of role (including managers who own tasks), or only hourly-tier staff? Spec assumes every task owner appears (a manager who misses their own tasks is also signal). Confirm — this interacts with whether managers see their own miss rate.
  3. Q3: does a GM see their own miss rate / response time in the reports, or are they excluded from the lists they can view? Spec assumes a GM sees everyone in their store including themselves (transparency), but flag in case Sam wants managers excluded from the reports they can read.
  4. Q4: response-time report — only count escalations that were eventually completed, or also surface still-open escalations as their own "stale" line? Spec assumes both: median/mean over completed ones + a separate still-open count. Confirm.

1G is one commit: team_reports_bp + the two permission tags + the four reports + the sidebar gate + templates + tests. It does not ship the 2H manager-tools reports (separate sub-block), any mutation, or any data production. Hard dependency on 1A landed + PASSed.
