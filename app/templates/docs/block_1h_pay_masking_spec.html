Block 1H — Pay Masking Layer

Phase 2 / Block 1 / sub-block 1H. samai spec, 2026-05-14.
Implementation: aick. Consumed in UI by ck. Review: samai checklist PASS — extra-careful, this is a pay-data-leakage surface.
Source: Phase 2 Directive §BLOCK 1 / 1H.

1. Scope — what 1H is, and what it is NOT

1H is a single helper — render_labor_breakdown(store, user) — that takes a store's labor breakdown and returns it masked to the requesting user's role. It is the one place pay-aggregation policy lives, so every surface that shows labor data (the ribbon's sales category in 1C, the team tab in 1G, the Block 3 Q&A agent) calls this helper instead of rendering labor data directly.

1H ships:

  • app/services/pay_masking.py — the module
  • render_labor_breakdown(store, user) -> LaborBreakdown — the public helper
  • The masking transform (_apply_masking) — the security-critical part
  • The role → labor-tier mapping (reuses role_hierarchy from 1A)
  • A defense-in-depth store-scope guard (§6)
  • Tests: the masking matrix (every role × every line kind), the store-scope guard

1H does NOT ship: any UI (ck consumes the helper in 1G's team tab + elsewhere); the ribbon (1C calls the helper); the labor data source itself — see §7, this is a flagged gap.

1H's value is the masking transform. That part is fully specifiable and fully testable against a documented raw-input shape, independent of where the raw labor numbers come from. Spec + build + test the transform now; wire the data source per §7.

2. The masking rule

Two things combine: who is asking, and which tier the labor line belongs to.

2.1 The viewer split is binary

  • partner → sees everything: every line individual, every dollar figure. No masking.
  • everyone else (corporate included) → manager-tier pay is aggregated; see §2.2. The directive is explicit: "Partner sees all individual values. Everyone else sees aggregated manager-tier." Corporate does not get individual manager comp — comp detail is partner-only.

2.2 The three labor tiers

TierRoles in itHow it renders for "everyone else"
Kitchen-manager km, assistant_km (directive's examples: Janet, Sebastian, Yasmine) one rolled-up line "Kitchen manager labor", percentage of total labor only — no dollar figure, no individual breakdown
FOH-manager + GM gm, foh_manager (directive's examples: Anna, Brittany = GM; Adriana, Andres = FOH manager) one rolled-up line "FOH manager labor", percentage of total labor only
Hourly cook, busser, server, host, bartender, expo, driver shown normally — grouped by role, with dollars + headcount + percentage. Not aggregated away.

For partner, all three tiers render as full detail — the two manager rollup lines instead break out into individual manager labor lines with dollars.

Taxonomy dependency: the Hourly tier references cook / busser / server / host / bartender — the same roles 1A §5.3 flagged as missing from ROLE_PERMISSIONS. 1H is a second surface that needs them. This reinforces 1A §5.3 path A (extend the role taxonomy as a Block 1 precondition) — it unblocks 1A's can_assign_to, 2B/2C, and 1H's tier mapping in one commit. Until the taxonomy lands, 1H's _labor_tier(role) maps the roles that do exist and treats unknown roles as Hourly (the safe default — unknown role never lands in a manager rollup, so it can't accidentally leak by being mis-tiered upward; worst case it shows as its own hourly line).

3. render_labor_breakdown — the public helper

def render_labor_breakdown(store: str, user: User) -> LaborBreakdown:
    """Return `store`'s labor breakdown, masked to `user`'s role.

    partner            → full detail, all lines individual, all dollars.
    everyone else      → kitchen-manager + FOH-manager tiers each collapse
                         to a single percentage-of-total line; hourly tier
                         shown normally.

    Defense-in-depth: if `user` is store-scoped and `store` is not in
    their scope, returns LaborBreakdown.refused() — see §6.
    """

4. The return structure

@dataclass
class LaborLine:
    label:        str            # "Kitchen manager labor" | "Cooks" | ...
    kind:         str            # "rollup_pct_only" | "detail"
    pct_of_total: float          # always present — the % of total store labor
    dollars:      Decimal | None # present only when kind == "detail"
    headcount:    int | None     # present only when kind == "detail"

@dataclass
class LaborBreakdown:
    store:        str
    viewer_role:  str
    lines:        list[LaborLine]
    total_labor:  Decimal | None # present only for partner; None for everyone else
    refused:      bool = False   # True when the store-scope guard (§6) refused

    @classmethod
    def refused(cls, store, user): ...   # lines=[], total_labor=None, refused=True
  • kind="rollup_pct_only" lines carry only pct_of_total — dollars and headcount are None. This is the security boundary: a rollup line can never carry a dollar figure. A review item (§10) checks this invariant holds in code, not just by convention.
  • kind="detail" lines carry dollars + headcount + pct_of_total.
  • total_labor is None for non-partner viewers — without it, the rollup percentages can't be back-solved into dollar figures. (If a non-partner had both the % and the total, they could compute the masked dollars. Withholding the total closes that inference channel.) Partner gets total_labor.
  • For non-partner viewers the lines list is: exactly one "Kitchen manager labor" rollup line, exactly one "FOH manager labor" rollup line, then N hourly detail lines. For partner: individual manager detail lines + hourly detail lines, no rollups.

5. Role → labor-tier mapping

_labor_tier(role) -> "kitchen_manager" | "foh_manager_gm" | "hourly". Built on role_hierarchy from 1A — reuse role_tier / role_domain rather than re-deriving. Resolve _LEGACY_ALIASES first ("manager" → gm → foh_manager_gm tier). Unknown role → "hourly" (the safe default per §2.2). corporate_chef and prep_manager placement is an open question — §11 Q2.

6. Defense-in-depth store-scope guard

1H's primary job is pay-tier masking, not store-scope authorization — but because this is a leakage surface, 1H adds a second layer rather than trusting every caller to gate correctly:

  • If user is store-scoped (gm and below) and store is not in user.store_scope → return LaborBreakdown.refused(store, user) (empty lines, refused=True). The caller renders a "not in your scope" state, not a data view.
  • partner / corporate / corporate_chef / prep_manager are store-unscoped — the guard is a no-op for them.
  • This does not replace the caller's own gate. 1G's team tab must still gate store access before calling 1H — the §10 checklist verifies callers gate too. 1H's guard is the inner belt of a belt-and-suspenders pair, the same layered-auth shape Sam praised in the 251621f unassign-courier review.

9. Test requirements

1H's test file: tests/test_pay_masking.py. This is a leakage surface — the masking matrix is the highest-value test in the sub-block.

  • The masking matrix: parameterized over viewer role × labor tier. For each cell, assert: partner sees the manager tiers as detail with dollars; every non-partner role sees them as rollup_pct_only with dollars is None. Self-describing test ids (viewer=gm__tier=foh_manager_gm__expected=rollup_pct_only).
  • The dollar-leak invariant: assert directly that no rollup_pct_only line ever has a non-None dollars or headcount, for any non-partner viewer, across the whole fixture. This is the one-line regression net against the worst-case bug.
  • total_labor withholding: non-partner viewers get total_labor is None; partner gets a value. (Closes the back-solve inference channel.)
  • Store-scope guard: a Tomball GM asking for Copperfield → refused=True, empty lines; partner asking for any store → never refused.
  • Unknown role → hourly: a role not in the tier map renders as an hourly detail line, never folded into a manager rollup.
  • Legacy alias: a "manager"-role user is tiered as foh_manager_gm (via the gm alias), masked accordingly.
