Block 1H — Pay Masking Layer

Phase 2 / Block 1 / sub-block 1H. samai spec, 2026-05-14. Revised 2026-05-14 after aick confirmed the labor data source already exists — see §2.
Implementation: aick. Consumed in UI by ck. Review: samai checklist PASS — pay-data-leakage surface, but most of the masking already ships in the codebase (§2).
Source: Phase 2 Directive §BLOCK 1 / 1H.

1. Scope — what 1H is, and what it is NOT

Revised understanding. The first draft of this spec described a from-scratch masking engine. That was wrong: the masking already exists. app/services/toast_reports.py's labor_report(...) already redacts management pay, using role_classifier.is_management_position() — it implements Sam's 2026-05-09 redaction rule. 1H does not reinvent that.

1H is a thin canonical adapter: render_labor_breakdown(store, user) — the one place a User is mapped to the redact_management flag, so every Phase 2 surface that shows labor data (the ribbon's sales category in 1C, the team tab in 1G, the Block 3 Q&A agent) masks consistently instead of each surface re-deriving "should this viewer see management pay."

1H ships:

  • app/services/pay_masking.py — the module
  • render_labor_breakdown(store, user) -> dict — the canonical adapter (§3)
  • The user → redact_management derivation — the one piece of genuinely new logic (§4)
  • A defense-in-depth store-scope guard (§5)
  • Optionally — pending Sam's call in §6 Q1 — a collapse of labor_report's per-management-title redacted rows into the directive's two named rollup lines
  • Tests: the user → redact_management matrix, the store-scope guard, the no-dollars-leak invariant carried through from labor_report

1H does NOT ship: the labor data source (labor_report already exists); the redaction logic itself (already exists); any UI (ck consumes it); the ribbon (1C calls it).

2. What already exists — read this before implementing

app/services/toast_reports.py → labor_report():

labor_report(start, end, location_filter=None, role_filter=None,
             redact_management=False, refresh=False) -> dict
  • Aggregates Toast time entries by job title into per-title rows: {title, role, people_count, hours, labor_cost, pct_net_sales, pct_of_labor, shifts, people[]}, plus report-level total_cost, overall_pct, etc.
  • redact_management=True: for any row where is_management_position(title) is true, people_count / hours / labor_cost / pct_of_labor / shifts are set to None and people[] is emptied — only pct_net_sales survives. The row still contributes to the report-level totals (so KPI strips stay correct) but carries no per-row detail. The docstring is explicit: "Partner view (owners only) sets this False."

app/services/role_classifier.py:

  • is_management_position(title) — true for kitchen manager / asst kitchen manager / floor manager / general manager / gm / owner / corporate / corporate admin (+ substring "manager"/"owner"/"corporate"). This is the redaction classifier.
  • classify_role(title) — 'boh' | 'foh' from the job-title string (Sam's 2026-05-09 rule). A display axis, not a redaction axis.
  • Both classify from Toast job-title strings, not from User.permission_level app roles.

Correction to the first draft (and to my chat post — walking it back): 1H is NOT blocked by, and does not need, the §5.3 role-taxonomy extension. I previously told Sam 1H was a "third surface" needing cook/server/busser/host/bartender added to ROLE_PERMISSIONS. That was wrong — labor_report + role_classifier work off Toast job-title strings and never touch ROLE_PERMISSIONS. §5.3 path A still unblocks 1A's can_assign_to (and should be re-verified for 2B/2C), but 1H is not on that list. The methodology miss: I asserted a dependency without verifying against the codebase first — exactly the verify-against-prod-not- -assumptions rule, applied to code structure. aick's "the source exists" confirmation is what surfaced it.

3. render_labor_breakdown — the canonical adapter

def render_labor_breakdown(store: str, user: User,
                           start: datetime | None = None,
                           end: datetime | None = None) -> dict:
    """Canonical entry point for labor data anywhere in Phase 2.

    Derives redact_management from `user` (§4), resolves `store` to a
    location_filter, applies the default date window if start/end omitted,
    runs the defense-in-depth store-scope guard (§5), then returns
    labor_report(...)'s dict — optionally with the two-line management
    collapse from §6 Q1 if Sam wants it.
    """

It is a thin wrapper. The body is essentially: guard → derive flag → call labor_report → (optional collapse) → return. The value is that it is the one place the user → redact_management decision lives.

4. The user → redact_management derivation — the one new piece of logic

  • partner → redact_management = False — sees everything, every management row in full detail. (Matches labor_report's docstring: "Partner view (owners only) sets this False.")
  • every other role → redact_management = True — corporate included. Management rows show only pct_net_sales.

Implement as _redact_management_for(user) -> bool: return not _is_partner(user), with _is_partner resolving _LEGACY_ALIASES first. One function, one test matrix (§7). This is the whole "new logic" of 1H — everything else is plumbing over the existing labor_report.

5. Defense-in-depth store-scope guard

Unchanged from the first draft — still valuable, still cheap:

  • If user is store-scoped (gm and below) and store is not in user.store_scope → return a refused result (empty rows, a refused: True flag) rather than calling labor_report at all.
  • partner / corporate / corporate_chef / prep_manager are store-unscoped — guard is a no-op.
  • This does not replace the caller's own gate (1G's team tab still gates store access before calling 1H). It is the inner belt of a belt-and-suspenders pair — the same layered shape as ck's 251621f cross-check.

6. Open questions for Sam

  1. Q1 — per-title-redacted vs. the directive's two rollup lines. labor_report redacts per management job title — a "Kitchen Manager" row, a "Floor Manager" row, an "Assistant Kitchen Manager" row, each showing only pct_net_sales. The Phase 2 directive's 1H text wants two collapsed lines: "Kitchen manager labor" (one) and "FOH manager labor" (one). Two paths: (a) 1H collapses the redacted management rows into those two named lines using classify_role(title) (boh → kitchen-manager line, foh → FOH-manager line) — extra presentation logic in 1H; or (b) keep labor_report's existing per-title-redacted output as-is — arguably better (more granular, still fully redacted) and zero new code. samai's lean: (b) — the existing per-title-redacted output already satisfies the privacy intent ("no dollars, only %"), and it's strictly more informative than two collapsed lines. But the directive said "one line," so this is Sam's call.
  2. Q2 — which percentage survives redaction. labor_report's redaction preserves pct_net_sales (% of net sales). The directive's 1H text says "percentage of total labor only" — i.e. pct_of_labor. These are different numbers. The existing code keeps % net sales and nulls % of labor. samai's read: % net sales is the more operationally meaningful number (it's the one the labor reports headline), and changing it would mean editing labor_report itself — a wider blast radius than 1H. Recommend keeping the existing behavior (pct_net_sales survives) unless Sam specifically wants % of labor. Confirm.
  3. Q3 — default date window. When a caller (the ribbon) doesn't pass start/end, what window does render_labor_breakdown default to? Spec proposes the current week-to-date. Confirm, or name a different default.

7. Test requirements

1H's test file: tests/test_pay_masking.py.

  • The user → redact_management matrix: parameterized over every role. partner → False; every other role (corporate, gm, km, assistant_km, corporate_chef, prep_manager, foh_manager, expo, driver, + legacy aliases) → True. Self-describing ids. This is the core test — it's the only genuinely new logic.
  • Store-scope guard: a Tomball GM asking for Copperfield → refused; partner asking for any store → never refused; store-unscoped roles → guard is a no-op.
  • No-dollars-leak invariant (carried through): with a fixture labor_report result, assert that after render_labor_breakdown runs for a non-partner viewer, no management-title row carries a non-None labor_cost / hours / people_count, and people[] is empty. This re-asserts labor_report's contract through 1H — proving 1H's wrapper doesn't accidentally un-redact.
  • Partner sees detail: for a partner viewer, management rows carry full detail (the inverse of the leak test).
  • If Sam picks §6 Q1 path (a): a test that the collapse produces exactly two management lines with the right classify_role split.
  • labor_report itself is mocked/fixtured in these tests — 1H's tests verify 1H's wrapper logic, not Toast aggregation (that has its own coverage).

8. Integration points

  • 1C (ribbon sales category): 1C §7.2 — any sales-category ribbon item carrying labor data calls render_labor_breakdown. Now that 1H is a thin adapter over an existing function, 1C can call it as soon as 1H lands; little sequencing risk.
  • 1G (team tab): the team tab's labor views render through this helper; 1G also does its own store-scope gate.
  • Block 3 (Q&A agent): the "no-pay-data-leakage" system principle points at this helper — any agent action surfacing labor data routes through it.
  • Existing labor views: the current Tomball / Copperfield / Corporate / Partner labor views already set redact_management somewhere. aick — worth checking whether those existing call sites should be refactored to go through render_labor_breakdown too, so there is genuinely one derivation. If that refactor is clean, fold it into 1H; if it widens the blast radius, leave it as a flagged follow-up. Either way 1H is the canonical path going forward.

  1. Scope: only app/services/pay_masking.py + tests (+ optionally the existing-call-site refactor per §8, if clean).
  2. 1H does not reimplement redaction — it calls labor_report(redact_management=...); is_management_position stays the single classifier.
  3. _redact_management_for(user): partner → False; every other role incl. corporate → True; legacy aliases resolved.

  6. The user → redact_management matrix test + the leak invariant test are present and green.
  7. §6 Q1/Q2/Q3 resolved per Sam's answers (or spec defaults applied + noted if Sam deferred).
  8. Full suite no regression.

10. Distinct-commit boundary recap

1H is one commit: app/services/pay_masking.py (the render_labor_breakdown adapter + the user → redact_management derivation + the store-scope guard + optional collapse) + tests. It does not touch labor_report or role_classifier (the redaction already lives there). It has no dependency on the §5.3 role-taxonomy extension — that was a first-draft error, corrected in §2. It is small because the hard part — the redaction policy — already shipped in the codebase as Sam's 2026-05-09 rule.
