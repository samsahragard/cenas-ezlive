1. Scope — what 1J is, and what it is NOT

1J is the in-app data-plane / control-plane separation. Today the "control plane" (the 1C ribbon, the 1F /cron/sales-insights pipeline) pulls external intelligence directly — every consumer re-fetches weather / events / outages itself. 1J inserts a data plane between the sources and the consumers: a single AmbientSignal table that six independent per-source crons write, and that the ribbon + the sales-insights pipeline read. One producer, many consumers.

1J ships:

  • AmbientSignal model + AmbientSignalRun audit model + one migration
  • ambient_signal_upsert() — the shared id-stable upsert helper (§3)
  • Six per-source cron endpoints (/cron/refresh-*) + their Render cron resources (§4)
  • The /cron/sales-insights refactor — read AmbientSignal instead of pulling sources directly (§5)
  • The 1C ribbon-router AmbientSignal adapter — Caterings / Events / Maintenance now also read ambient signals (§7)
  • Tests: the three cross-system test contracts (§10)

1J does NOT ship — explicitly, per Sam #1386:

  • The full two-service extraction (Cenas Kitchen app = consumer; a separate data-platform service = provider, Toast ingestion moved there). That is DEFERRED to Phase 2.5 — 2–3 weeks of properly-paced work after Block 1 closes and the team rests. 1J is the in-app refactor only — same process, same DB, new internal seam.
  • Any new ribbon rendering shell (1B), any new dismiss/check write path beyond wiring ambient_signal into the existing 1D handlers (§7.3), any change to the 1F Haiku→Opus synthesis logic itself (§5 moves only the source-fetch stage).

Dependency status: 1J depends on 1A (Task / RibbonItemDismissal — landed), 1C (the ribbon router — landed), and 1F (SalesInsight + sales_insights.py — landed, 021720e). All three are in-tree and samai-PASSed. 1J is a refactor on top of a complete Block 1; it touches 1C + 1F surfaces but adds no dependency on 1I or anything unbuilt.

2. The AmbientSignal model — the data-plane row

One AmbientSignal row = one logical piece of external intelligence — "Tomball weather today", "Astros home game 5/16", "outage near Copperfield" — that has a stable identity across refreshes. The per-source cron re-pulls every cadence; the row's id does NOT change when the same logical signal is re-seen with a fresh payload (§2.2 — this is the property the whole sub-block hinges on). 

class AmbientSignal(Base):
    __tablename__ = "ambient_signals"

    id            Integer, primary key
    source        String(32),  NOT NULL   # weather | events | outages
                                          #   | catering_pipeline
                                          #   | vendor_status | traffic
    signal_key    String(200), NOT NULL   # source-scoped LOGICAL identity
                                          #   (see §2.1) — NOT the payload
    payload       JSON,        NOT NULL   # the structured signal content
    payload_hash  String(64),  NOT NULL   # sha256 of the canonical payload
                                          #   (§2.1) — the change detector
    store_scope   String(20),  NOT NULL   # tomball | copperfield | both
    category      String(24),  NOT NULL   # caterings | events | maintenance
                                          #   (the ribbon category — §7)
    severity      String(10),  NOT NULL   # info | warn | alert
    valid_until_at DateTime,   NOT NULL, indexed   # when it ages out (§2.3)
    created_at    DateTime,    NOT NULL, default=utcnow
    updated_at    DateTime,    NOT NULL, default=utcnow, onupdate=utcnow
    last_seen_at  DateTime,    NOT NULL, default=utcnow   # last cron that
                                          #   re-confirmed this signal (§2.2)

    __table_args__ = (
        # The id-stable upsert lookup AND the logical-identity guarantee:
        # one row per (source, signal_key). This unique constraint is what
        # makes "re-pull the same logical signal" deterministically find
        # the existing row instead of inserting a duplicate.
        UniqueConstraint("source", "signal_key",
                         name="uq_ambient_signal_identity"),
        # The hot read paths: the 1C ribbon live-query and each cron's
        # own expiry sweep both hit (valid_until_at, store_scope) /
        # (source, valid_until_at). Mirrors SalesInsight's
        # ix_sales_insights_live.
        Index("ix_ambient_signals_live", "valid_until_at", "store_scope"),
        Index("ix_ambient_signals_source_expiry", "source", "valid_until_at"),
    )
  • source / category / store_scope / severity are String not native ENUM — same call as Task / SalesInsight; values validated application-side against module-level _VALID_AMBIENT_* constants (and severity reuses the existing info|warn|alert set the ribbon already sorts on).
  • NOT an audit log — like SalesInsight, an AmbientSignal is ephemeral operational intelligence, not history. There is deliberately no before_delete listener: expired rows are DELETEd (§2.3).
  • No dismissed_by column (contrast SalesInsight §2.1). 1J v1 gives ambient signals X-dismiss only (daily, via RibbonItemDismissal — §7.3); they have no permanent "Check" dismiss because they age out on their own via valid_until_at. can_check=False for ambient signals (§7.2). If Sam later wants permanent-dismiss it is an additive follow-up (a dismissed_by column + a 1D branch) — out of 1J scope.

2.1 signal_key + payload_hash — the change-detection contract

These two columns are the heart of 1J. Get them exactly right.

  • signal_key — the logical identity. A source-scoped string that answers "is this the same logical signal I saw last cycle?" It is derived from the signal's identity, never its content. Examples:
    • weather → "tomball:forecast:2026-05-14" (store + kind + the day it forecasts)
    • events → "ticketmaster:evt:vG1zABC123" (the source's own stable event id)
    • outages → "centerpoint:area:77375" (the affected area, not the customer count)
    Two pulls of the same logical signal MUST produce the same signal_key even when the payload changed (the forecast updated, the outage grew). Each per-source cron owns its signal_key construction and documents it in a module constant/comment.
  • payload_hash — the change detector. sha256 of a canonical serialization of payload — json.dumps(payload, sort_keys=True, separators=(",",":")), encoded utf-8. Canonical so that semantically-identical payloads hash identically (dict ordering can never cause a spurious "changed"). The hash covers the content fields the consumers care about (headline, detail, severity, valid_until_at, the structured body) — it does NOT cover last_seen_at / updated_at / id (those are row metadata, not signal content). Provide one shared _ambient_payload_hash(payload: dict) -> str so all six crons hash identically.

2.2 id-stable update semantics — the three cases

When a per-source cron re-pulls a signal, ambient_signal_upsert() (§3) does a lookup by (source, signal_key) and branches:

CaseConditionActionid
unchanged row found, incoming payload_hash == stored hash NO-OP on content. Bump last_seen_at only (so an unchanged-but-still-live signal isn't mistaken for stale). Return "unchanged". same
updated row found, incoming payload_hash != stored hash IN-PLACE UPDATE of the existing row — overwrite payload, payload_hash, store_scope, category, severity, valid_until_at; bump updated_at + last_seen_at. Return "updated". SAME — never changes
created no row for (source, signal_key) INSERT a new row. Return "created". new

The "updated" case keeping the SAME id is the single most important property in 1J. It is what makes the dismissal-survival invariant (§6) hold: a RibbonItemDismissal references (item_type, item_id), so a payload refresh that preserves id means the dismissal still matches the (updated) row. A new id on every refresh would orphan every dismissal — and for a 15-minute outage cron, that means a dismissed outage signal reappears on the user's ribbon four times an hour. Changed payload ⇒ in-place update, never delete-and-reinsert.

NOTE — a directive-internal contradiction, resolved here (see §12 Q4): Sam #1387 point 1 says "changed hash in-place update of existing row"; Sam #1387 point 7 names a test "hash-change-creates-new-signal" and ck #1388 reads it as "hash B → new row written." Those conflict. This spec resolves toward point 1 — changed hash = in-place UPDATE, same id — because the §6 dismissal-survival invariant (Sam's own "the critical invariant") forces it: a new row on a hash change breaks dismissal survival outright. Only a genuinely new signal_key produces a new row. Flagged for Sam in §12 Q4; ck's Day-4 test name + framing must follow this resolution.

2.3 valid_until_at + expiry — how a signal ages out

  • valid_until_at is NOT NULL — every ambient signal must expire so nothing lingers on the ribbon forever (same rule as SalesInsight §6). Each per-source cron sets it from the signal's natural relevance window (a weather forecast → end of that day; an event → end of the event date; an outage → its estimated restoration time, or a short default). An end-of-day floor when nothing else is derivable.
  • Expiry = DELETE, per source, self-cleaning. At the end of each per-source cron's run it sweeps its OWN source's expired rows: DELETE FROM ambient_signals WHERE source = '<this source>' AND valid_until_at < now. Per-source (not a shared expiry cron) so the crons stay independent (§8) — no cron depends on another, and a dead cron's stale rows just age past valid_until_at and get swept on its next successful run (or, if it stays dead, the ribbon's own valid_until_at >= now read-filter already hides them). DELETE not flag-flip — same reasoning as the 1F §9 SalesInsight call: an expired ambient signal has zero post-expiry value, no consumer of expired rows.
  • Expiry vs. refresh — the dismissal boundary (§6): a dismissal SURVIVES a refresh (id-stable update). A dismissal does NOT survive expiry — the row is DELETEd, the RibbonItemDismissal for it orphans, and that orphan is harmless inert cruft (the signal is genuinely gone; it's day-scoped and never matches anything again). That asymmetry is correct and intended.

2.4 AmbientSignalRun — the per-run audit table

One row per cron run — the operational record of "did each refresh cron run, and what did it do."

class AmbientSignalRun(Base):
    __tablename__ = "ambient_signal_runs"

    id                 Integer, primary key
    source             String(32),  NOT NULL, indexed   # which /cron/refresh-*
    started_at         DateTime,    NOT NULL, default=utcnow
    finished_at        DateTime,    nullable
    status             String(16),  NOT NULL   # success | error | partial
    signals_created    Integer,     NOT NULL, default=0
    signals_updated    Integer,     NOT NULL, default=0
    signals_unchanged  Integer,     NOT NULL, default=0
    signals_expired    Integer,     NOT NULL, default=0   # swept this run
    error_text         Text,        nullable   # set when status != success
    created_at         DateTime,    NOT NULL, default=utcnow, indexed
  • status: success (ran clean), error (the run failed before writing — e.g. the source API was down and the adapter raised past its own guard), partial (the run completed but something inside degraded — e.g. a sub-fetch failed but others wrote). The cron endpoint returns this run summary as its JSON response so a manual trigger is inspectable — same shape as 1E/1F's cron summaries.
  • §12 Q3 — "skipped_unchanged" status: Sam asked whether the enum needs a fourth state for an all-no-op cycle. Spec's answer: no. signals_unchanged already carries that detail per-run, and a cycle where every signal was unchanged is still a successful run (it ran, it checked, nothing changed) — not a distinct status. A separate status would force every consumer of the audit table to treat "success" and "skipped_unchanged" as equivalent anyway. The count is enough. Flagged for Sam to confirm.

3. ambient_signal_upsert() — the shared helper contract

The single source of truth for the §2.2 id-stable semantics. All six per-source crons call it; none of them re-implement the lookup/branch. Lives in app/services/ambient_signals.py (the new data-plane service module — the per-source adapters live here too, §4). 

def ambient_signal_upsert(
    db, *, source: str, signal_key: str, payload: dict,
    store_scope: str, category: str, severity: str,
    valid_until_at: datetime,
) -> str:
    """Upsert one logical ambient signal. Returns "created" | "updated"
    | "unchanged" (§2.2). Does NOT commit — the calling cron owns the
    transaction (same split as run_escalation_scan / detect_no_shows).

    - computes payload_hash via _ambient_payload_hash(payload)
    - SELECT ... WHERE source = :source AND signal_key = :signal_key
    - no row              -> INSERT, return "created"
    - hash unchanged      -> bump last_seen_at only, return "unchanged"
    - hash changed        -> IN-PLACE UPDATE of the SAME row (id never
                             changes), return "updated"

    Validates source/category/store_scope/severity against the
    _VALID_AMBIENT_* constants; a bad value raises ValueError (the
    caller's per-signal try/except records it and moves on — one bad
    signal never aborts the run).
    """
  • Returns a string verdict so the caller accumulates the signals_created / updated / unchanged counts for its AmbientSignalRun row.
  • Caller-owned transaction: the helper mutates the session, the cron endpoint commits once at the end (and rolls back on a hard failure). One run = one transaction, so a crash mid-run leaves the table exactly as it was — the next cadence retries cleanly (the upsert is idempotent by construction).
  • It is import-safe to call from any cron; it imports only app.models + stdlib. No Flask.

6. The dismissal-survival invariant — the critical property

Sam #1387 point 6 calls this "the critical invariant." Stated precisely:

A user's RibbonItemDismissal of an ambient-signal ribbon item MUST continue to hide that item across payload refreshes of the same logical signal.

Why it holds — the mechanism, end to end:

  • 1C's _exclude_dismissed() (already in ribbon.py) matches dismissals on the tuple (item_type, item_id).
  • An ambient-signal ribbon item carries item_type="ambient_signal" (§7.1 / §12 Q1) and item_id = AmbientSignal.id.
  • A user dismisses it → 1D writes RibbonItemDismissal(item_type="ambient_signal", item_id=<that id>, dismiss_day=today).
  • 15 minutes later the outage cron re-pulls. The payload changed (customer count grew) → ambient_signal_upsert() does an in-place UPDATE (§2.2) → the row's id is unchanged.
  • Next ribbon render: _exclude_dismissed() looks up ("ambient_signal", <that same id>) → still in the user's dismissal set → the refreshed signal stays hidden. Invariant holds.

The boundary — what the invariant does NOT cover (and correctly so):

  • A genuinely new logical signal (new signal_key → new id): a new dismissal is required, which is correct — it is a different thing the user has not triaged.
  • Expiry (§2.3): the signal ages out and is DELETEd; the RibbonItemDismissal orphans harmlessly (day-scoped, never matches again).
  • Daily reset: RibbonItemDismissal is keyed on dismiss_day — the X-dismiss is "not now, today," and the signal re-surfaces tomorrow if still valid. That is the existing, intended 1A/1D semantics; 1J does not change it.

The anti-pattern this invariant rules out: a refresh implemented as delete-and-reinsert (new id every cadence). For a 15-minute outage cron that would resurface a dismissed signal four times an hour. The §2.2 in-place update is the entire defense — §10's dismissal-survives-refresh test is the proof.

7. 1C ribbon-router changes — the AmbientSignal adapter

1J adds AmbientSignal as a fifth gather source to ribbon_items_for() in app/services/ribbon.py, alongside the existing Task / Signal / ScheduledEvent / SalesInsight sources. Per Sam #1387 point 5 the new reads feed the Caterings, Events, and Maintenance ribbon categories — alongside the existing logic (ScheduledEvent already feeds caterings/events; Task + Signal already feed maintenance). AmbientSignal is purely additive — no existing adapter is removed or changed.

7.1 The adapter

  • New _ambient_signal_to_item(signal) -> RibbonItem — same shape as the existing _sales_insight_to_item / _scheduled_event_to_item adapters: pure, reads .id / .category / .severity / .payload, returns a RibbonItem with relation="observer" (ambient signals have no owner).
  • New gather block in ribbon_items_for(): query live AmbientSignal rows (valid_until_at >= now), store-scope-filter them via the existing _in_scope() helper, run each through the adapter. Same store-scoping rule as every other source.
  • category on the RibbonItem comes straight from AmbientSignal.category (which is already constrained to caterings | events | maintenance — §2). No new _*_TO_RIBBON map needed.

7.2 can_dismiss / can_check

  • can_dismiss=True — the X control renders; X writes a daily RibbonItemDismissal (§6).
  • can_check=False — ambient signals are not "completable." They are observed external state; they age out via valid_until_at. Mirrors scheduled_event (also can_check=False), not sales_insight (whose Check = permanent dismiss, which needs a dismissed_by column AmbientSignal deliberately does not have — §2). This is a spec decision, not a Sam question; Sam can redirect in review.

7.3 item_type — the render-contract change (Sam #1387 Q1)

Sam's open question #1: does ambient_signal become a fifth item_type in the ribbon render contract, or reuse an existing one? Spec's analysis points hard at: it must be a fifth item_type. Reasoning:

  • An AmbientSignal is a distinct underlying table. 1D's _load_item_row() resolves a ribbon item's (item_type, item_id) back to its source row — for dismiss/check eligibility — by branching on item_type. There is no way for item_id to point at the ambient_signals table without an "ambient_signal" branch in _load_item_row(). Reusing (say) scheduled_event would make _load_item_row() query the wrong table and 404.
  • 1B's _ribbon.html partial is item_type-agnostic (it passes the value straight through — confirmed in ribbon.py's contract docstring). So the cost of a fifth item_type is contained on the router/handler side, not the template side.
  • This is exactly the "coordinated 1B+1C contract change" the ribbon.py docstring already describes — 1C itself added scheduled_event as the fourth the same way.

If Q1 resolves to "fifth item_type" (the spec's strong lean), the coordinated changes are — all on Day 4 (ck):

  • ribbon_routes.py: add "ambient_signal" to _VALID_RIBBON_ITEM_TYPES.
  • ribbon_routes.py _load_item_row(): add an item_type == "ambient_signal" branch → db.get(AmbientSignal, item_id).
  • ribbon_routes.py _can_check_item(): add an ambient_signal branch returning (False, "ambient signal") — not checkable (§7.2), but X-dismissable.
  • ribbon.py RibbonItem.render_for(): add an ambient_signal branch (observer-style text from the payload — like the sales_insight branch).

None of these touch a 1J data-plane surface — they are all 1C/1D ribbon surfaces, which is why they are ck's Day-4 lane. The spec flags Q1 for Sam, but absent a redirect, implement the fifth-item_type path.

8. Cron-independence

The six per-source crons are fully independent — this is a named test contract (§10). Concretely:

  • Six separate endpoints, six separate Render cron resources, six separate transactions. No cron imports or calls another.
  • One cron failing produces an AmbientSignalRun(status="error") and nothing else — the other five run on their own schedules, the ribbon keeps reading whatever live signals exist, /cron/sales-insights keeps synthesizing from whatever is there.
  • A persistently-dead cron degrades gracefully: its signals pass valid_until_at and the ribbon's valid_until_at >= now read-filter stops surfacing them — degraded, never broken. Same defensive posture as 1B's ribbon and 1F's "one dead adapter."
  • Each cron handler wraps its source fetch + per-signal upsert loop so one bad signal (a ValueError from ambient_signal_upsert's validation) is recorded and skipped, never aborts the run.

10. Test contracts (Sam #1387 point 7)

1J's test files: tests/test_ambient_signals.py (the model + upsert + per-source crons, aick Days 1–2) and integration coverage in ck's Day-4 lane. The three named contracts:

  • dismissal-survives-refresh — write an AmbientSignal; dismiss it via the 1D dismiss path (RibbonItemDismissal, item_type="ambient_signal"); re-run the source cron / call ambient_signal_upsert() with (a) an unchanged payload and (b) a changed payload; after each, render the ribbon and confirm the signal is still hidden. The changed-payload case is the real proof — it asserts the id did not change (§2.2) so the dismissal still matches.
  • hash-change → in-place update — write an AmbientSignal (hash A); call ambient_signal_upsert() again with a changed payload (hash B); assert the same row was updated in place — same id, new payload / payload_hash / updated_at — and NO second row was created. Separately: a new signal_key DOES create a new row. (NB — this corrects the framing in Sam #1387 point 7's test name and ck #1388: per §2.2 + §12 Q4, a hash change is an in-place update, not a new row. ck's Day-4 test must follow the §2.2 resolution.)
  • cron-independence — mock one per-source cron's fetch to raise; run it + the others; confirm the failing one writes an AmbientSignalRun(status="error") and the others still write their signals + their own runs normally. No cross-contamination.

Deploy-verify — two distinct verifications, two different days (ck #1418 clarification). The original draft listed a single "six-cron deploy-verify" without drawing the line; these are separate things owned by separate days:

  • Day 2 (aick) — the per-cron route + resource verify. Build-specific (rule 3, per rule 2), done per-commit as each cron is built: (1) the endpoint — POST /cron/refresh-<source> goes 404 (pre-deploy) → 403 (post-deploy, CRON_TOKEN gate) — the build-specific route-existence proof; AND (2) the Render cron resource exists and is scheduled at the §4 cadence.
