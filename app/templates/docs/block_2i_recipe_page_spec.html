Block 2I — Recipe Page + Import

Phase 2 / Block 2 / sub-block 2I. samai spec, 2026-05-14.
Implementation: ck (models + browse UI) + aick (import script). Review: samai checklist PASS per commit.
Source: Phase 2 Directive §BLOCK 2 / 2I.

1. Scope — what 2I is, and the aick / ck split

2I is a read-only browse surface for the 24 prep recipes Sam previously uploaded — /manager/recipes: a searchable, category-filtered list and a per-recipe detail page showing category code + name, prep time, yield, shelf life, preparation steps, and an ingredients table with SINGLE / DOUBLE quantity columns. It is a clean standalone: zero dependency on 1A or any Block 1 surface — pure model + route + template + a one-time import script.

2I is two commits, per the directive's aick/ck split:

  • 2I-a (ck) — models + browse UI. The three models (Recipe, RecipeStep, RecipeIngredient) + migration + the /manager/recipes list and detail routes + templates + the recipes.view permission tag + tests. ck owns the UI surface, so ck defines the models (both the UI and the import need them; the UI is the primary surface). This is the outage-window build-and-hold lane — 1A-independent, render-contract-independent, nothing about it can ripple into Block 1.
  • 2I-b (aick) — the import script. A one-time, idempotent script that parses the 24 uploaded recipes' extracted text into Recipe + RecipeStep + RecipeIngredient rows. Depends on 2I-a's models existing. Not on 1A's critical path — aick runs it when free.

2I does NOT ship: any edit UI (the directive: "Edit access: partner only in v1" — and v1 has no edit route at all, it is read-only browse; a Recipe Editor for the corporate_chef role is explicitly Future); the Portions page (2J — separate sub-block, depends on 2I's Recipe model); any KDS integration.

2. The three models

2.1 Recipe

class Recipe(Base):
    __tablename__ = "recipes"

    id                 Integer, primary key
    category_code      String(20),  NOT NULL          # "HOT" | "SAUCE" | ...
    recipe_number      Integer,     NOT NULL          # the "HOT 1" / "SAUCE 7" id
    name               String(200), NOT NULL
    prep_time_minutes  Integer,     nullable          # "LISTO EN"
    yield_amount       String(60),  nullable          # "4½ GALL", "38 LB" — free text, units vary
    shelf_life_days    Integer,     nullable          # "DURACION"
    preparation_steps  Text,        nullable          # multi-paragraph; used when is_multi_step=False
    is_multi_step      Boolean,     NOT NULL, default=False
    notes              Text,        nullable
    source_doc         String(200), nullable          # filename of the original Word doc
    created_at         DateTime,    NOT NULL, default=utcnow
    updated_at         DateTime,    NOT NULL, default=utcnow, onupdate=utcnow

    __table_args__ = (
        UniqueConstraint("category_code", "recipe_number",
                         name="uq_recipe_category_number"),
        Index("ix_recipes_category", "category_code"),
    )
  • UniqueConstraint(category_code, recipe_number) — "HOT 1" is one recipe; the import script's idempotency (§4) leans on this to upsert rather than duplicate on re-run.
  • ix_recipes_category — the list view's category filter.
  • is_multi_step drives the detail-page render: False → render preparation_steps as the body + the ingredients table flat; True → render the RecipeStep rows, each with its own ingredient sub-table.
  • yield_amount is free-text String (not Decimal + unit) — the directive is explicit ("4½ GALL" or "38 LB", units vary). Don't try to parse it.

2.2 RecipeStep

class RecipeStep(Base):
    __tablename__ = "recipe_steps"

    id           Integer, primary key
    recipe_id    Integer, FK(recipes.id, ondelete="CASCADE"), NOT NULL, indexed
    step_number  Integer, NOT NULL
    title        String(200), nullable
    instructions Text, nullable

    __table_args__ = (
        UniqueConstraint("recipe_id", "step_number",
                         name="uq_recipe_step_number"),
    )
  • Only populated for is_multi_step=True recipes. A single-step recipe has zero RecipeStep rows — its body lives in Recipe.preparation_steps.
  • recipe_id ondelete=CASCADE — steps are wholly owned by their recipe; if a recipe is removed (re-import / correction), its steps go with it. Recipes are reference data, not an audit trail — CASCADE is correct here (unlike Block 1A's audit-log RESTRICT).

2.3 RecipeIngredient

class RecipeIngredient(Base):
    __tablename__ = "recipe_ingredients"

    id              Integer, primary key
    recipe_id       Integer, FK(recipes.id, ondelete="CASCADE"), NOT NULL, indexed
    step_id         Integer, FK(recipe_steps.id, ondelete="CASCADE"), nullable
        # NULL for single-step recipes; set for multi-step (ties the ingredient to its step)
    ingredient_name String(200), NOT NULL
    quantity        Numeric(10, 3), nullable          # SINGLE-recipe quantity
    quantity_double Numeric(10, 3), nullable          # DOUBLE-recipe column, where the source has one
    unit            String(40),  nullable             # POUNDS / GALLONS / CUPS / OZ / TBSP / BAG / CAN / ...
    notes           String(200), nullable
    sort_order      Integer, NOT NULL, default=0      # preserve the source ordering within a recipe/step
  • quantity / quantity_double are Numeric (not free-text) — the directive says Decimal. Both nullable: some ingredients in the source are "to taste" / "as needed" with no number, and quantity_double is only present where the source recipe has a DOUBLE column.
  • step_id: NULL for single-step recipes (the ingredient hangs directly off the recipe); set for multi-step (the ingredient belongs to a specific step). Both FKs CASCADE — ingredients are wholly owned.
  • sort_order — the source recipes list ingredients in a meaningful order; preserve it. The import script sets it; the detail page renders by it. This is a samai addition beyond the directive's field list — without it, ingredient order is undefined and the rendered table would scramble vs. the original Word doc. Flagged in §8 Q3 in case Sam wants it dropped, but recommend keeping.
  • unit is free-text String — the directive's example list (POUNDS / GALLONS / CUPS / OZ / TBSP / BAG / CAN) is open-ended; don't enum it.

3. The browse UI (2I-a, ck)

Blueprint app/web/recipes.py (recipes_bp), mounted at /manager/recipes, registered in app/__init__.py. Two routes, both GET, both read-only:

3.1 GET /manager/recipes — the list

  • Searchable (by name + category_code — a simple ILIKE substring match is fine for 24 rows; no full-text index needed at this scale).
  • Category filter — a control listing the distinct category_code values; filtering narrows the list. Server-derived filter (a query param is fine here — it is a read-only filter on non-sensitive data, no scope/permission implication, unlike 1G's store-scope param).
  • Each row links to the detail page. Show category code + recipe number + name + yield at the list level.
  • Mobile-friendly — the directive flags line cooks may reference on tablets. The list + detail must render usably on a tablet/phone width. Exact treatment is ck's call within the templates; the requirement is "usable on a tablet," same bar as 1B's mobile ribbon.

3.2 GET /manager/recipes/<id> — the detail

  • Header: category code + recipe number + name; the metadata strip — prep time ("LISTO EN"), yield ("CANTIDAD" / yield_amount), shelf life ("DURACION").
  • Body — branches on is_multi_step:
    • False: render preparation_steps (multi-paragraph) + one flat ingredients table (the RecipeIngredient rows with step_id IS NULL, ordered by sort_order).
    • True: render each RecipeStep in step_number order — its title + instructions + its own ingredient sub-table (the RecipeIngredient rows whose step_id matches).
  • Ingredients table: ingredient_name | quantity + unit | (quantity_double + unit — this column renders only if any ingredient in the recipe has a non-null quantity_double; a recipe with no DOUBLE data shows a single quantity column, not an empty one) | notes.
  • notes (recipe-level) rendered if present. 404 on an unknown id.

4. The import script (2I-b, aick)

  • A one-time script (scripts/import_recipes.py or a CLI command) that parses the 24 uploaded recipes' extracted text into Recipe + RecipeStep + RecipeIngredient rows.
  • Idempotent: re-running it must not duplicate. Upsert keyed on (category_code, recipe_number) — if the recipe exists, replace it (delete its RecipeStep + RecipeIngredient rows via CASCADE, re-insert); if not, insert. A re-run after a parser fix should converge, not pile up.
  • Source: the directive points at extracted text at /home/claude/recipes/*.txt from a prior session, "or Sam re-uploads." aick — confirm the source files are present + readable before building the parser; if they are not on this AiCk machine, flag it to Sam for re-upload. The parser's exact logic depends on the actual text format, which this spec cannot pin down without seeing the files — so §4 specs the contract (what rows the import must produce, idempotency, the upsert key), and the parsing detail is aick's to implement against the real source format.
  • Recon output: the script prints a summary — recipes imported, steps, ingredients, and any recipe it could not fully parse (a malformed source file should be surfaced, not silently skipped — same shape as the Toast pull's recon JSON). 24 recipes in, 24 recipes (or a named list of what failed) out.
  • The import is data-population, not a deploy surface — it runs against the DB directly. It does not need deploy-verify in the route sense; its verification is the recon output + a spot-check that a sampled recipe's rows match its source doc.

5. Permissions

  • New tag recipes.view — added to ROLE_PERMISSIONS, granted to the manager tier and above: km, assistant_km, foh_manager, gm, corporate, corporate_chef, prep_manager (partner via wildcard). Both routes carry @requires_permission("recipes.view"); the sidebar link is wrapped in has_permission('recipes.view') (samai rule 4 — both surfaces).
  • Recipes are operational reference, not sensitive — the directive is explicit: "No counseling-style cascade." So recipes.view is a flat tag, no store-scope, no tier cascade. It is not a §2.2 global-config tag either — it gates reference content, not a config surface — so granting it to GM + the manager tier is correct and does not violate the §2.2 partner+corporate rule.
  • No edit tag in v1 — there is no edit route, so there is nothing to gate. When the Future corporate_chef Recipe Editor lands, that sub-block adds recipes.edit.
  • The matrix test (2255c1d) auto-extends to recipes.view — one row per role confirms the gate.

6. Test requirements

2I-a (tests/test_recipes.py):

  • Models: Recipe / RecipeStep / RecipeIngredient round-trip; the two unique constraints reject duplicates; FK CASCADE declared on recipe_steps.recipe_id, recipe_ingredients.recipe_id, recipe_ingredients.step_id; a single-step recipe (no steps, ingredient step_id NULL) and a multi-step recipe (steps + ingredient step_id set) both round-trip.
  • Routes: list 200 + renders seeded recipes; category filter narrows correctly; search matches name + category substring; detail 200 for a known id, 404 for unknown; the DOUBLE column renders iff some ingredient has quantity_double; a multi-step recipe renders steps in step_number order with per-step ingredient tables.
  • Permission gating: the role matrix — manager-tier+ → 200, hourly tier (expo/driver/cook/server/busser/host/bartender) → 403; sidebar link present for the granted roles, absent for the rest.

2I-b (tests/test_recipe_import.py):

  • Parse fixture → rows: a small fixture recipe text (one single-step, one multi-step) parses into the expected Recipe + RecipeStep + RecipeIngredient rows with correct sort_order and SINGLE/DOUBLE quantities.
  • Idempotency: running the import twice produces the same row count — the second run upserts, does not duplicate (the (category_code, recipe_number) key + CASCADE-delete-then-reinsert).
  • Malformed source: a deliberately-broken fixture is surfaced in the recon output, not silently dropped.

7. samai review checklist (run per commit)

  1. Scope: 2I-a touches only the recipe models + migration + recipes_bp + templates + the recipes.view tag + tests — no edit route, no 2J Portions surface, no KDS. 2I-b touches only the import script + its tests.
  2. Three models match §2 — types, the two unique constraints, FK CASCADE, the indexes; sort_order present (§8 Q3).
  3. List route: searchable (name + category), category filter, mobile-usable; detail route: metadata strip, is_multi_step branch correct, DOUBLE column conditional on actual data, 404 on unknown id.
  4. recipes.view added to ROLE_PERMISSIONS for manager-tier+; both routes decorated; sidebar gated (rule 4 — both surfaces); matrix test auto-extends green.
  5. Import script idempotent (upsert on category_code+recipe_number); recon output surfaces parse failures; aick confirmed the source files exist or flagged Sam for re-upload.
  6. Migration present + reversible; tests per §6 green; full suite no regression.
  7. Deploy-verify: BLOCKED-PENDING-RENDER until the build outage clears (pipeline_minutes_exhausted). When builds resume, deploy-verify is build-specific (rule 2 clause): hit /manager/recipes on prod under an authed manager session and confirm it renders the imported recipes — a route that only exists on this commit.
