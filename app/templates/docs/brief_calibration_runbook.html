Morning Brief Calibration Runbook

Phase 1 / Block 6 calibration — C4 of 4.
How to flip the panel-test on, run the 5-morning Round 1, and roll back if needed.

1. What this is

The morning-brief composer has shipped end-to-end (compose → email dispatch → in-app surface) but the LLM-composed brief has never been read by a real human on production signal data. The calibration panel is the validation step: 4 readers (Sam + 3 corporate) get a tagged version of the brief for 5 consecutive mornings and feed back what's missing, what's noise, and what would make it worth reading every morning. The Round 1 → Round 2 aggregation doc is the artifact that gates broad dispatch.

3. Pre-flight — before flipping any env var

3.1 Resolve the panel User IDs

Sam needs the integer User IDs for the 4 Round 1 panel members. Run this against the prod DB (Render shell or psql against DATABASE_URL): 

SELECT id, full_name, email, permission_level
FROM users
WHERE full_name IN ('Sam Sahragard', 'Masood', 'Angelica', 'Javier')
  AND active = true;

Verify all 4 rows return with valid emails. If any name doesn't match exactly, adjust the WHERE clause to whatever the full_name column actually holds (the users table is the source of truth — don't trust the names in this doc verbatim).

Capture the 4 IDs as a comma-separated string for the env var. Example: 1,7,12,19.

3.2 Confirm panel members in chat

Post in /partner/developer/chat before flipping the flags so the panel members know to expect the first [Calibration] email at the next 7:30 CT cron. Sam's earlier chat with samai already covered scope, but a one-line "panel is live tomorrow morning, please reply with answers to the 4 questions inline" message reduces surprise.

3.3 Verify BRIEF_EMAIL_DISPATCH state

The two flags are independent on purpose (Sam 2026-05-13 20:13 — "panel-before-broad-flip"):

  • Calibration only — BRIEF_EMAIL_DISPATCH=1 + BRIEF_CALIBRATION_MODE=1. Only panel members get mail. This is Round 1.
  • Calibration dry-run — BRIEF_EMAIL_DISPATCH unset + BRIEF_CALIBRATION_MODE=1. Compose runs, dispatch logs but no SMTP send. Useful for a pre-flip rehearsal day.
  • Broad live — BRIEF_EMAIL_DISPATCH=1 + BRIEF_CALIBRATION_MODE unset/0. Everyone in _ENROLLED_ROLES gets mail. This is the post-Round-1 destination state — only after the aggregation doc passes.
  • Off — both unset. Compose runs, all dispatches dry-run.

4. The flip — exact order

Order matters. BRIEF_CALIBRATION_MODE=1 + empty BRIEF_CALIBRATION_PANEL = everyone gets skipped_not_panel (safe-fail, no broad sends) — but it also means the panel gets no mail until the panel env is populated. Set the panel first, then enable the mode.

  1. Set BRIEF_CALIBRATION_PANEL in Render env to the comma-separated IDs from §3.1 (e.g. 1,7,12,19). Save.
  2. Set BRIEF_EMAIL_DISPATCH=1 in Render env if it isn't already on. Save.
  3. Set BRIEF_CALIBRATION_MODE=1 in Render env LAST. Save.
  4. Trigger a redeploy in the Render UI. Env-var changes don't auto-deploy on this service — the next request needs the new env loaded. Wait for the deploy to go live (~5 min).
  5. Manual first run (optional) — if you don't want to wait until 7:30 CT, POST to /cron/anomaly-brief with the CRON_TOKEN Authorization header. The cron will compose + dispatch one brief per enrolled audience; in calibration mode only the panel members get mail. If you wait, the 7:30 CT cron fires automatically.

5. Verifying the first calibration send

  1. Panel inboxes — Sam (sam@cenaskitchen.com), Masood, Angelica, Javier (corporate addresses from §3.1) should each see one email with subject [Calibration] Cenas brief — {today}. Body opens with "This is a calibration test…" and closes with the 4 feedback questions + a link to /partner/briefs/<brief_id>/feedback.
  2. MorningBrief rows — one row per enrolled user per day. Verify against the prod DB: 
    SELECT id, brief_id, audience_role, audience_user_id, composer_model, fallback_used
FROM morning_briefs
WHERE brief_date = '{today}'
ORDER BY id;
    Composer cost: all enrolled users get composed (calibration only gates dispatch). If composer_model shows the LLM model (e.g. claude-opus-4-5) rather than deterministic, the LLM path fired. fallback_used=true means the LLM call failed and the deterministic template was used.
  3. Dispatch results — the cron summary returned by /cron/anomaly-brief includes a dispatch_results list (one entry per user with status: sent / dry_run / skipped / error). Confirm 4 entries with status="sent" and the rest with status="skipped" + reason="skipped_not_panel".
  4. Render logs — log lines should read: brief dispatch sent user_id=N to=<email> subject=... for each panel send, and brief dispatch skipped user_id=M role=<role> reason=skipped_not_panel for each non-panel skip.

6. Monitoring during Round 1 (5 mornings)

6.1 Reply parsing — the active Round 1 channel

Per Sam 20:13 + samai 20:23 design, Round 1 uses reply-to-email as the active channel. Panel members reply to the [Calibration] email with their answers inline. samai/aick reads the replies and creates BriefFeedback rows manually with submitted_via='email_reply' and submitted_at set to the reply's receive time.

The form endpoint at /partner/briefs/<brief_id>/feedback exists and works (Round 2 channel + always-available alternative for Round 1), but Sam's directive is that Round 1 doesn't depend on the form — the email-reply path is the canonical route.

6.2 Manual BriefFeedback INSERT shape

When parsing an email reply, INSERT a row roughly like: 

INSERT INTO brief_feedback
  (morning_brief_id, user_id, useful_score, missed_something,
   was_noise, single_change, submitted_via, submitted_at, created_at)
VALUES
  ( (SELECT id FROM morning_briefs WHERE brief_id = '<uuid>'),
    <panel_user_id>,
    <1-5 or NULL>,
    '<Q2 free text>',
    '<Q3 free text>',
    '<Q4 free text>',
    'email_reply',
    '<reply timestamp ISO>',
    NOW());

The brief_id is the UUID from the email subject/body (look for the calibration email's body footer link). The user_id matches the panel member who replied. Skip any field the recipient didn't answer (NULL is fine — the model allows NULL on all answer fields).

6.3 Latency join (for the aggregation doc)

Response latency is a real signal per Sam 20:47 — a 30-min reply from Angelica vs a 36-hour reply means different things about utility. The join: 

SELECT
  mb.brief_date,
  mb.audience_role,
  u.full_name AS panel_member,
  bf.useful_score,
  EXTRACT(EPOCH FROM (bf.submitted_at - mb.composed_at))/60 AS latency_min,
  bf.missed_something,
  bf.was_noise,
  bf.single_change
FROM brief_feedback bf
JOIN morning_briefs mb ON mb.id = bf.morning_brief_id
LEFT JOIN users u ON u.id = bf.user_id
WHERE bf.submitted_via = 'email_reply'
  AND mb.brief_date BETWEEN '<round_1_start>' AND '<round_1_end>'
ORDER BY mb.brief_date, latency_min;

This is the data shape the Round 1 aggregation doc (samai's lane) is derived from.

7. Rollback

Two paths depending on what went wrong:

7.1 Stop the calibration only (keep broad dispatch off)

  1. Set BRIEF_CALIBRATION_MODE=0 in Render env (or unset entirely).
  2. Leave BRIEF_CALIBRATION_PANEL set — it's inert when mode is off.
  3. Leave BRIEF_EMAIL_DISPATCH in whatever state. If it's 1, next 7:30 CT cron fires broad dispatch — make sure that's intended. If you don't want broad sends either, unset BRIEF_EMAIL_DISPATCH too.
  4. Trigger a redeploy.

7.2 Stop everything (no morning emails of any kind)

  1. Unset BRIEF_EMAIL_DISPATCH + BRIEF_CALIBRATION_MODE.
  2. Trigger a redeploy.
  3. Compose still runs and persists MorningBrief rows; dispatch is dry-run-only.

7.3 Code-level rollback (if a brief reads catastrophically badly)

The 4 calibration commits (13a2e44, bc6ce15, d4fded8, this doc) can be reverted individually. C1's model + migration shouldn't be reverted in isolation (the table is fine being empty if calibration goes back to off). C2's routes are inert without panel data. C3 is the only commit that materially changes dispatch behavior — reverting C3 returns dispatch to pre-calibration shape; C1+C2 stay landed.

8. Optional: staging dry-run

If you want to rehearse the flip without sending to real recipients, set BRIEF_FEEDBACK_BASE_URL to a staging URL (default in prod is https://app.cenaskitchen.com). The feedback link in the email body points to the override URL. Combined with BRIEF_EMAIL_DISPATCH unset, you get a full composition + dry-run cycle with no actual sends — useful for eyeballing the email body shape on a Render preview / staging env.

9. Round 1 → Round 2 transition

samai's lane. After Round 1's 5 mornings close, samai writes the aggregation doc at /partner/developer/app/brief-calibration-round-1 with sections (a) per-question summary, (b) per-recipient summary, (c) identified patterns across the 5 mornings, (d) proposed tuning changes with explicit before/after, and (e) LLM-vs-signal-vs-presentation layer attribution (samai 20:48 refinement). That doc gates the broad flip — when Sam +1's the doc, that's the signal to set BRIEF_CALIBRATION_MODE=0 and let BRIEF_EMAIL_DISPATCH=1 carry the brief to the full enrolled audience.

Anna + Brittany (GMs) join the panel in Round 2 but only after:
