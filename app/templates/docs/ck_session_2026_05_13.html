PERMISSION_ENFORCE flip (commit 3373a28)

Once Sam confirmed at 18:39 that the team was populated, I shipped the enforce flip as its own commit (distinct risk surface = gate-misfire, not data-entry correctness — samai re-runs her checklist specifically). One file, +21/-7.

  • app/services/permissions.py:_enforcing() default flipped. Was return os.getenv("PERMISSION_ENFORCE") == "1"; now return os.getenv("PERMISSION_ENFORCE", "1") != "0". Unset or any non-"0" value enforces; only the literal "0" disables (escape hatch for a future role-schema migration where transient mismatch might deny legitimate users).
  • Module-level docstring "Dark-launch" section rewritten as "Enforcement mode" explaining the new default + the escape hatch + the flip date and reason. Function docstring updated to match. The log_denial label still toggles ENFORCING / DARK-LAUNCH based on live state so debugging stays clean.
  • Net effect on prod: every @requires_permission-decorated route that previously logged-and-passed now redirects denials to /access-denied?need=<tag>&next=<path>. Partner wildcard {"*"} short-circuits everything for Sam so his own flows are unaffected. Routes NOT yet migrated to the new decorator (orders.assign_driver, legal_routes, anomaly_routes, the Admin/Developer/Anomalies sidebar wrappers) still use the legacy _enforce_partner path and gate correctly — those are in the deferred-followup list, not blocked by this flip.
