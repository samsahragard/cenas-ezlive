Toast POS

Auth flow: machine-client OAuth. POST clientId+clientSecret+userAccessType=TOAST_MACHINE_CLIENT to the auth endpoint. Response includes a JWT access token valid ~24h. We cache the token in-process via ToastClient.shared() and refresh on exp - 5min or on 401.

Endpoints used:

  • /labor/v1/employees — full org directory per restaurant
  • /labor/v1/jobs — job titles + GUIDs per restaurant
  • /labor/v1/timeEntries?startDate=&endDate= — clock-in/out per (employee, job, shift)
  • /labor/v1/shifts?startDate=&endDate= — scheduled shifts (Tomball only on this account; Sling has both locations)
  • /orders/v2/ordersBulk?businessDate=YYYYMMDD&page=N&pageSize=100 — paginated orders for one business date
  • /restaurants/v1/restaurants/<guid> — basic restaurant info (name, locationName, address)

Required headers (Cloudflare WAF blocks without them):

Authorization: Bearer <token>
Toast-Restaurant-External-ID: <restaurant guid>
Accept: application/json

Cache: Disk JSON files at /var/data/toast/<file>.json. Refresh by deleting the file or passing refresh=True in the service call.

Two restaurants on the account: Tomball (27727 Tomball Pkwy) + Copperfield (15650 FM 529). Each has its own restaurant GUID; the same management group GUID covers both.

Sling (getsling.com)

Why we use Sling at all: Toast's /labor/v1/shifts only returns scheduled shifts for the Tomball restaurant entity (Sling pushes them there but not to Copperfield). To see both locations' schedules cleanly, we have to call Sling directly. Sling treats both Cenas Kitchen locations as a single "Group" so one query returns both.

Auth: Sling doesn't issue developer API keys via UI. The SLING_API_KEY env var is the value of the Authorization header captured from a logged-in browser session. The token lasts ~30 days.

Refresh procedure (when calls start returning 401):

  1. Log into app.getsling.com in a browser
  2. Open DevTools → Network tab
  3. Click any request to api.getsling.com/v1/*
  4. Copy the Authorization header value
  5. Update the SLING_API_KEY env var on Render

Endpoints used:

  • /v1/account/session — current session, returns org.id + user.id
  • /v1/groups — locations + positions + "everyone" buckets (typed via type field)
  • /v1/groups/<id>/users — members of one group (location or position)
  • /v1/users — all users in the org (active + inactive)
  • /v1/calendar/<orgId>/users/<userId>?dates=ISO_START/ISO_END — calendar entries (shifts + availability + leave). The userId in the path is just the requestor — the call returns ORG-WIDE data regardless

Cloudflare WAF in front of api.getsling.com rejects plain urllib requests (1010 "browser signature blocked"). The Sling client sends User-Agent + Origin + Referer headers like a real browser to clear it.

Cache: /var/data/sling/<file>.json.

EzCater Partner API

What it is: ezCater's GraphQL API at api.ezcater.com/graphql — supports webhook subscriptions, order pulls by id, and courier-assignment mutations.

Auth: Bearer token (renewable, ~24h life).

Webhook flow (auto-pipeline):

  1. Customer places an order on ezCater.
  2. ezCater fires Order.submitted webhook to https://app.cenaskitchen.com/ezcater/webhook.
  3. Our handler pulls the full order via Partner API order(id:) query.
  4. Distance check via Google Routes API picks the closer of Tomball / Copperfield.
  5. courierAssign mutation labels the kitchen that will prep (Sam-CK#2 for Tomball, Masood-CK#1 for Copperfield).
  6. POST to /orders/ingest_structured (self-call) writes the order to our DB.
  7. Telegram message to Sam.

Subscription state: 4 caterer UUIDs × 4 events (submitted, cancelled, succeeded, updated) = 16 active subscriptions. Subscriber id cenaskitchen-render.

Produce vendor IMAP polling

Setup: Alvarado and J. Luna each email weekly price sheets to orders@cenaskitchen.com (SiteGround mailbox). Alvarado sends photo attachments; J. Luna sends an xlsx file.

Polling loop (in app/services/produce_ingest.py):

  1. Daemon thread spawned by create_app() when PRODUCE_INGEST_ENABLED=1
  2. fcntl file lock at /var/data/produce/.ingest.lock — only one gunicorn worker actually polls
  3. Every 60s: connect to IMAP, find new mail since last seen UID, filter to approved senders
  4. For Alvarado emails: extract attached image, send to Anthropic vision with a structured prompt → JSON list of items
  5. For J. Luna emails: parse the xlsx with openpyxl
  6. Apply aliases (data/produce/aliases.json) to map vendor item names → canonical names
  7. Write {vendor}.json to /var/data/produce/
  8. Also write one produce_price_snapshot row per item (idempotent on the unique constraint) — feeds the price-history page
