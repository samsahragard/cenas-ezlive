What this page is

Append-only log of permission denials from the @requires_permission decorator. Each row is one denied request — partners can see what's getting blocked and decide if a role's tag set needs adjustment, an audience boundary is wrong, or a route was decorated with the wrong tag.

Populated by app/services/permissions.py:_log_denial on every denial, in addition to the standard logger output. The DB write is best-effort — if it fails the routing path still completes and the log line still goes to Sentry / stdout.

Mode column

  • ENFORCING — request was redirected to /access-denied. This is the default since the 2026-05-13 flip (PERMISSION_ENFORCE defaults to enforce; only the literal "0" turns it off).
  • DARK-LAUNCH — denial logged but request was allowed through. Only possible if PERMISSION_ENFORCE=0 is set in env, which we'd do briefly during a future role-schema migration where transient mismatches might deny legitimate users.
