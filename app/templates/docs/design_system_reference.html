Cenas Kitchen — Design System Reference

The living reference for every visual + structural decision in the Cenas Kitchen app. Authored 2026-05-17 by dck as Track 9 of Sam's operational cutover (per Sam #1958 + Cena #1996). Captures the design system as it stands today — strengths, debt, and recommended fixes.

How to use this doc

  • Read at every session start (per DCK_CHARTER §3a once the auto-load wiring lands; until then, read it manually).
  • Before proposing any new design, check it against the reference. Use the canonical tokens. Don't introduce new patterns when an existing one fits.
  • When you ship a design change, update the relevant section here in the same work cycle. Stale references are worse than no reference.
  • When you notice live-vs-reference drift, flag in dev chat and propose a fix — either update the reference to match new reality, or restore the app to match the reference, depending on which was the deliberate decision.

1 · Design language overview

Cenas Kitchen is a dark-mode, glassmorphism-on-a-tequila-bar-photo operational dashboard. The visual identity is anchored to the logo: saturated red (#C73B36), warm gold (#E4B340 and #FFD970), and a warm cream (#FAF6EC) on dark earth tones. Tabler icons throughout via webfont. The overall vibe: a calm, premium operator's pane rather than a generic SaaS dashboard.

1.1 · The honest state: two coexisting token systems (V1 + V2)

There are two parallel CSS token systems in production today, plus three special-purpose divergences. This is the most important fact about the design system, because surfaces look almost-but-not-quite the same depending on which system they pulled from.

SystemWhere it's definedUsed byStatus

V2 tokens (brand-canonical)

Rule of thumb for new design work: use V2 (--ck-*) tokens. The V1 system is older and the warm-cream brand decision (--ck-text: #FAF6EC) came after V1's --text: #ffffff. New surfaces should converge on V2; older surfaces should migrate as they're touched.

1.2 · Three special-purpose divergences

  • Sam Chat (sam_chat.html) — uses raw rgba() values throughout, ~258 lines of inline CSS. Picks up #FAF6EC brand cream as explicit hex rather than via var(--ck-text). Two near-but-not-equal oranges (rgba(232,122,62,*) vs #E87A3E). Should be migrated to V2 tokens; the divergence has no design intent.
  • Keypad login (keypad_login.html, driver_keypad_login.html) — standalone <!DOCTYPE html> outside base_dashboard. Sam-approved cream + red-speckles 3D-layered header on the bar-photo backdrop. Distinct palette (--blue / --red / --cream). Intentional divergence — this is a brand moment, not debt. Don't touch.
  • Orphaned style.css — 936 lines of an entirely different design (warm gray #484041 nav, red #A44A3F active, rem-based units). Linked in zero live templates. Vestigial — recommend audit-then-delete.

1.4 · Backdrop and atmosphere

The single most distinctive element is the persistent backdrop: brand/bg-tequila.jpg blurred 5px at brightness 0.82 + saturate 1.10, with a warm radial-gradient wash on top (orange at 20% 30%, gold at 80% 20%, plus a 180deg dark linear vignette). Defined in dashboard.css body::before + body::after. Repeated on the standalone keypad_login.html. Don't change this without a real reason — it's a load-bearing piece of the brand feel.

1.5 · Type, motion, and icon vocabulary

  • Font stack: -apple-system, BlinkMacSystemFont, 'Inter', 'Segoe UI', Roboto, sans-serif — system-native first, Inter as cross-platform fallback. Base 14px, line-height 1.5.
  • Type scale (observed in V2 surfaces): 22px primary value · 18px h2 · 15px h3 / card title · 14px body · 13px secondary · 12px metadata · 11px uppercase labels · 10px tiny meta. Letter-spacing 0.12em on uppercase section labels is a recurring V2 signature.
  • Motion: --ck-ease: cubic-bezier(0.22, 0.8, 0.24, 1) on V2 surfaces (200ms to 240ms). V1 surfaces use raw ease at 0.15s. The V2 ease is the brand-canonical motion shape.
  • Glow recipe (V2): two-part text-shadow — a hard carve (0 1px 1px rgba(0,0,0,0.55)) + a warm bloom (0 0 12px rgba(255,240,215,0.18)). Cream glow (--ck-glow-cream) and gold glow (--ck-glow-gold) are pre-tokenized. Caveat from CENA.md: bloom can blur on low-DPI Windows + old Android; drop bloom on those breakpoints if fuzzy.
  • Icons: Tabler webfont via CDN (cdn.jsdelivr.net/npm/@tabler/icons-webfont@2.40.0), classes ti ti-*. Consistent vocabulary across the app — don't introduce other icon sets without a real reason.
  • Breakpoints: three in play today — --ck-bp-mobile: 1024px (sidebar.css drawer), 900px (dashboard.css legacy mobile cascade), 720px (dashboard.css phone tier). See debt item #4.

2 · Layout chrome (universal)

Every authenticated page composes the same outer chrome from base_dashboard.html (app/templates/base_dashboard.html):

  1. Sidebar (aside.sidebar on desktop, drawer below 1024px) — store/logo at top, role-gated section groups, sidebar-footer with avatar.
    • 5 role tiers gate sections: driver (collapsed driver-only menu) · expo (hides Insights) · partner (shows Admin + Developer) · corporate · gm (default).
    • Store-switcher strip lives above the sidebar logo (.ck-store-switcher-strip).
  2. Mobile hamburger — bridges old body.sidebar-open + new .ck-sidebar-host[data-open] mechanisms (transitional design-debt; see debt item #6).
  3. Topbar — page title + optional subtitle (left), topbar_right block (right) typically holding search input + notification bell. Bell shows .dot red indicator when attention is truthy.
  4. Universal ribbon (partials/_ribbon.html) — Phase 2 / Block 1B, ck 2026-05-14. Operational signals on every authenticated page, between topbar and content. Defensive: a ribbon bug can't 500 a page.
  5. Subnav row — persistent category sub-nav above content if subnav_for() returns entries. Tab-strip look: flat tabs in a row, active gets dark recessed bg + orange bottom-border accent.
  6. Anomaly cards (partials/_anomaly_cards.html) — opt-in via child setting anomaly_page_slug = '<slug>'. Renders signals filtered by current store.
  7. Content block — {% block content %}{% endblock %}.
  8. Decorative wordmark bottom-right at z-index: 0 + opacity: 0.48 — floats faintly over empty backdrop.

5 · Recommended improvements (dck design judgment)

Ordered roughly by impact-per-effort. Each item is a concrete fix with a starting point, not a generic complaint. None of these have been implemented yet — they're the audit's bottom line. Cena/Sam route each through the normal flow (dck spec → ck/aick build → samai gate) on approval.

5.1 · Tokenize the chart palette + migrate Sam Chat to V2 (HIGH impact)

Observation: Chart.js color palette is declared as a JS literal twice in home.html (once at line 86, once at line 304) — ['#6dd16d', '#e87a3e', '#4080e8', '#b46ee8', '#f0c060', '#e85d8e', '#5dd4d4', '#999']. Of these, only #e87a3e matches --accent; the rest drift from --ck-status-* + brand tokens. CENA.md flags this exact problem at the design language layer.

Fix: Add --ck-chart-1..--ck-chart-10 to sidebar.css :root using brand-aligned palette (anchor on --ck-gold, --ck-red, --ck-status-open, derived hues). Extract a one-file app/static/js/ck-chart-palette.js exporting CK_CHART_PALETTE that reads from CSS variables via getComputedStyle. Replace both home.html palette literals with the imported constant. Same change in any future chart surface (reports/insights when those wire up).

Companion fix: migrate sam_chat.html's 258 lines of inline CSS to --ck-* tokens. Most rgba values already approximate brand cream/orange — direct token swap. Removes the dual-orange (rgba(232,122,62,*) vs #E87A3E) inconsistency in one pass.

5.2 · Consolidate the four-identical-button-classes problem (HIGH impact)

Observation: dashboard.css defines .btn-portal, .btn-primary, .btn-login, .btn-add-driver — all the same orange primary button with identical hover color (#c66631 — itself a magic non-tokenized value). Naming-by-context, not by shape.

Fix: One canonical .btn-primary class. Add --ck-accent + --ck-accent-hover tokens (currently the V1 --accent works but needs a V2 home). Delete the three context-named clones and update every template usage. Reduces ~40 lines and removes the magic-color duplication.

5.3 · Decommission orphaned style.css (MEDIUM impact, low risk)

Observation: 936 lines of an entirely different design system (warm gray #484041 navbar, red #A44A3F active, rem-based). Linked in zero live templates per grep across app/templates/. Likely vestigial from an earlier prototype layer.

Fix: Confirm-then-delete. Audit any non-template references (raw <link> in served HTML, doc pages, anything outside Jinja). If clean, delete the file. Cuts 936 lines of dead CSS.

5.4 · Align the three responsive breakpoints (MEDIUM impact)

Observation: Three mobile breakpoints currently active — 1024px (sidebar.css drawer threshold), 900px (dashboard.css legacy mobile), 720px (dashboard.css phone tier). Surfaces respond at different points; the chrome and the content drift apart between 1024 and 900.

Fix: Adopt --ck-bp-mobile (1024px) + a --ck-bp-phone (720px) as the two canonical points. Migrate dashboard.css @media (max-width: 900px) cascade either up to 1024 or down to 720 — pick one based on actual content overflow behavior (browser-test required). Single source of truth, then propagate.

5.5 · Donut "soon" anti-pattern (LOW impact)

Observation: dashboard.css .donut::after { content: 'soon' } is hardcoded as a CSS pseudo-element. CENA.md flagged this at the design language layer: "single-segment donuts are visual noise — render as a single stat with the donut as fallback once a second segment exists."

Fix: Behavior change, not CSS. Where the donut would render as a single segment, render the value as a large text figure with a small donut as a fallback decoration. Touches every page that uses .donut-wrap. Lighter-touch alternative: hide the donut + show "data pending" / "single channel" copy when slices.length ≤ 1.

5.6 · Extract a ck-donut.js helper module (MEDIUM impact, refactor only)

Observation: home.html defines makeDonut + renderLegend + setLoading + color-palette + format helpers TWICE — once for sales/labor donuts (lines 28-220) and once for Toast Analytics (lines 222-470+). ~120 lines of duplicated JS within a single template.

Fix: Pure refactor. Extract a app/static/js/ck-donut.js exporting renderDonut({canvas, slices, totalForPct, palette}) + renderLegend({el, slices, totalForPct, palette}). Both home.html JS blocks shrink to ~10 lines of config. Same module reused by future report/insight surfaces.

5.7 · Bridge the V1 / V2 token systems (LARGE effort but pays compound interest)

Observation: V1 (--accent, --text, --glass) and V2 (--ck-*) coexist. Surfaces are split between systems with no clear migration plan. The cream-text divergence (V1 --text: #fff vs V2 --ck-text: #FAF6EC) means surfaces look subtly off-brand depending on which system they inherit.

Fix shape: Treat V2 as canonical. In dashboard.css :root, redefine V1 tokens to point at V2 equivalents (--text: var(--ck-text), --accent: var(--ck-gold), --glass: rgba(40,22,14,0.55), etc.) so unmigrated surfaces inherit the brand values via the old token names. Then opportunistically migrate surfaces to use --ck-* directly as they're touched. Closes the divergence without a Big Bang rewrite.

5.8 · Sidebar mobile-vs-desktop bridge cleanup (LOW impact, transitional debt)

Observation: base_dashboard.html JS bridges body.sidebar-open (V1 dashboard.css mobile) AND .ck-sidebar-host[data-open] (V2 sidebar.css mockup-v3). Inline comment confirms this is transitional ("both sliding mechanisms cooperate").

Fix: Pick V2 (.ck-sidebar-host) as the canonical mechanism, retire the V1 mobile rules from dashboard.css, simplify window.toggleSidebar to a single path. Touches dashboard.css mobile cascade + base_dashboard.html JS + sidebar.js.

5.9 · Inline CSS extraction policy (POLICY rather than fix)

Observation: Inline <style> blocks are everywhere — sam_chat.html 258 lines, base_dashboard.html ~120 lines, ez_manage.html ~60 lines, dozens more. Many of these patterns are reusable but locked in their host template.

Fix shape: Adopt a rule going forward — any inline <style> block over 50 lines extracts to a per-surface module under app/static/css/<surface>.css. Existing offenders migrate as they're touched.

5.10 · The structural / IA observations

Per Sam's amendment (2026-05-17) that design includes structure, IA, and user-friendliness:

  • "Partner is dashboard + extras" is good IA. The home.html partner view is the same template as the store view with {% if g.current_store == 'partner' %} additions. Keeps the operator's mental model consistent. Worth preserving.
  • Sidebar IA "section headers should pay rent" (CENA.md) — verify that no sidebar section contains exactly one item; if any do, fold the item into a parent or section-less area.
  • Doc page proliferation. DOC_PAGES in developer_chat.py is ~45 entries long; the top sub-nav across docs (rendered by _doc_base.html) becomes a wall of pills. Consider grouping by category (Spec / Reference / Handoff / Architecture) with collapsible groups or a left rail at the docs root, similar to the main sidebar.
  • Dev chat cost badge color (samai #2018 flag 2) — the cost badge in developer_chat.html uses teal that doesn't match the gold/orange brand palette. Recommend re-color to --ck-gold-soft bg + --ck-gold-bright text to keep cost displays inside the brand vocabulary.
  • Two tab patterns coexist (.tabs in dashboard.css for Order Processor vs .dash-chart-tab + .ezm-tab for period/section switchers vs .subnav-pill in base for the universal subnav). All three are visually distinct. Pick one shape; if multiple are needed, name them by function (tabs-master / tabs-period / tabs-section) rather than by surface (.dash-chart-tab / .ezm-tab).
