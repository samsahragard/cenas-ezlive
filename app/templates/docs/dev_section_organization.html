What this section is for

This is the engineering reference surface for Cenas Kitchen — specs, architecture, methodology, integration notes, agent handoffs, and operational runbooks. It is not the user-facing app; it lives behind the partner gate. The goal of the organization below is engineering future-proofness: every doc has a clear bucket, status, and lifecycle.

If you're new, start here (top 6)

  1. Phase 2 Directive — the current sprint's master roadmap. What we're building and why.
  2. Methodology Rules — samai's 6 rules + the three-gate deploy-verify rubric. The non-negotiable engineering gates.
  3. Deployment runbook — how the three-gate verification actually runs in practice (the procedure layer; the rules above are the "why").
  4. System Inventory — the operational map: every account, every machine, where every credential lives. No secret values recorded; this is the where, not the what.
  5. Cena Operational Spec — the Sam-only AI surface (tunnel topology, tool surface, audit log, escalate-on-uncertainty).
  6. Architecture — high-level system architecture + service-graph entry point.

Status tags — what each one means

Every doc that opts into the status header partial ({% include 'partials/_doc_status_header.html' %}) shows a small box with one of these states:

  • ACTIVE — doc describes work currently in progress. Subject to change.
  • SHIPPED — doc describes work that has shipped + been samai-reviewed PASS. Amend only via formal spec amendment, citing commit SHA.
  • REFERENCE — long-lived operational doc. Updated as state changes; rarely "done".
  • DRAFT — spec authored, implementation not yet started.
  • SUPERSEDED — replaced by another doc; banner points to the replacement. The slug stays valid forever.
  • ARCHIVED — historical only, no active replacement.

The header also shows last updated date, the block / topic the doc belongs to, and (if SUPERSEDED) the replacement slug. Docs that haven't opted in render normally without the header — backward-compatible.

Update protocol — how to add or modify a doc

  1. Write the doc at app/templates/docs/<slug_snake_case>.html, extending docs/_doc_base.html, with the body wrapped in {% raw %}...{% endraw %} if it contains literal Jinja-looking braces.
  2. Register the route by adding a tuple to DOC_PAGES in app/web/developer_chat.py: ("slug-kebab", "Display Title", "doc_slug_snake_case").
  3. Register the nav by adding the link to the appropriate section in app/templates/partials/sidebar.html.
  4. Optionally include the status header via {% include 'partials/_doc_status_header.html' with context %} and the right block variables.
  5. Three-gate verify per samai rules 2 + 3: local pytest, CI green, build-specific deploy probe (the URL renders 200 with the expected markers).
