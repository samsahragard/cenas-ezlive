What this page is

Operational rules for our ezCater driver workflow — what each pay component is, why tracking matters, when paychecks cut, and the messy bits about how ezCater ships driver names. Codified at app/services/ezcater_payroll.py; rules attributed to Sam (2026-05-10).

Pay structure (per delivery)

ComponentAmountCondition
Base pay$25.00Every delivery, no conditions.
Tracked bonus+$10.00Only when tracking_status == 'Tracked' on the order.
Extra-miles bonus+$1.50 / mile over 20Only when Tracked. Distance is kitchen → 1st drop-off, one-way, from ezCater's reported pickup miles.
Five-star bonus+$5.00Only when Tracked AND the delivery received a 5-star review on ezCater. ezCater's API doesn't expose review data, so this comes from a manual flag in DriverLog.five_star.

An untracked delivery gets only the $25 base — none of the bonuses apply. That's why driver tracking matters operationally; see next section.

Why tracking matters

Three of the four pay components ($10 tracked bonus, the per-mile extra, and the $5 five-star bonus) are gated on tracking_status == 'Tracked'. If a driver opens the delivery and runs it without the ezCater tracking link, the order comes back to us as untracked and the driver gets the base $25 only — they leave $10–$50+ on the table per run depending on distance.

Tracking also gives us the verified pickup-miles number that the per-mile bonus is computed against. Without a tracked run, we have no defensible mile figure and the bonus can't be honored.

Multi-stop routes — the leg-2 gotcha

ezCater resets the mile counter to 0 after the first drop-off on multi-stop routes. So a driver doing kitchen → drop A → drop B will see real miles only on the first leg; the second leg's pickup_miles is usually 0 or near-zero. This means the per-mile bonus rarely fires on legs 2+ of a multi-drop route, even though the driver is in fact driving those miles. Not a bug we can fix on our side — it's a quirk of how ezCater reports the data.

Pay period & check dates

  • Bi-weekly pay period (14 days).
  • Anchor: 2026-04-26 → 2026-05-09 (period starts Sunday).
  • Check date = period_end + 5 days. So the 4/26–5/9 period pays out 2026-05-14.
  • Cancelled orders are excluded from the period sum.

Driver-name matching (the fuzzy part)

ezCater ships the same driver under wildly inconsistent name formats across deliveries:

  • CK #1 - Jose Gonzalez
  • CK#1 ANA ISA Perez
  • Ck #1 Bryan Ruiz
  • CK# 1 BRITNEY MATHIS

The pay calc normalizes both sides before matching: lowercase → drop the CK#1 / CK#2 kitchen prefix → collapse whitespace → drop dashes/underscores. This works for the common variants ezCater throws at us. If a delivery still misses the right driver, the order ships in the candidate list but doesn't match any signed-up driver in our DB — surfaces as an unattributed run in the Driver Payroll report.

Five-star bonus — current state

ezCater's Partner API doesn't expose review data. Until that ships (or we scrape the Caterer Portal for it), the 5-star bonus is gated on a manual flag in the DriverLog table — a manager checks a box on a per-delivery log entry. The flag is keyed on driver_name + pickup_date + order_link (best-effort match against Order.external_order_id).

Reading the Driver Payroll page

Each row in /<store>/driver-tracking is one delivery in the current pay period. The total at the bottom is the gross paycheck. Per-row breakdown shows base / tracked-bonus / extra-miles / 5★ separately so a driver (or manager) can see exactly why a number is what it is. The "notes" column surfaces the tracking_status when it's anything other than Tracked, so it's obvious which deliveries left bonus money behind.
