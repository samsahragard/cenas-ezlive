samai's 6 standing rules (canonical, verbatim)

Every commit that touches user-facing state, a database mutation, or a partner doc must pass each rule that applies. These are not guidelines — they are the gate before "samai PASS".

  1. Audience-eligibility-before-mutation: the audience check (who can perform this action) runs BEFORE any database mutation. A 403 leaves zero rows.
  2. Verify-against-prod-not-local: deploy-verify hits production state, not local state. Must check something BUILD-SPECIFIC — a route or asset unique to the new commit. A build-agnostic response (partner-gate 302) proves the service is up, NOT which commit it is serving.
  3. Deploy-success-verify: every commit gets a real deploy-verify pass after Render lands it. BLOCKED-PENDING-RENDER is a valid state during build-pipeline outages; never claim "deploy-verified" against a stale build.
  4. Doc-registration-checks-both-route-AND-sidebar: a new doc requires DOC_PAGES entry AND sidebar nav config. Both surfaces.
  5. Hit-new-doc-URL-once-before-trusting-it: render-verify the URL returns 200 with expected content before claiming the doc is live.
  6. Amendments-against-git-show-HEAD: spec doc amendments are authored against the committed tree (git show HEAD:<path>), NEVER a local pre-commit copy. The tree is the source of truth.

Three-gate deploy verification rubric

Rule 3 (deploy-success-verify) is operationalized as three independent gates. A commit is "deploy-verified" only when ALL THREE pass on the same SHA, and the verification is reported with the gate-by-gate breakdown.

  1. Gate 1 — Local pytest: the full suite passes on the working tree. Catches schema mismatches, fixture drift, and obvious regressions before the change ever leaves your machine. On AiCk run from the repo's .venv: .\.venv\Scripts\python.exe -m pytest -q. On CK, ssh aick and run from there (pytest is not installed locally on CK).
  2. Gate 2 — CI: the GitHub Actions tests.yml workflow lands completed / success for the SHA. CI's environment differs from local in subtle ways (no .env, hermetic fixtures); a green CI proves the change still passes in a clean room.
  3. Gate 3 — Build-specific deploy probe: a probe that confirms the COMMIT is live — not just "the service is up". Per rule 2, the probe must check something BUILD-SPECIFIC: a new route, a new asset, a new string in a known route's body. A partner-gate 302 redirect proves the service is up, NOT which commit it is serving.

Full rubric prose lives in deployment.html (the operational runbook). This page is the methodology reference; deployment.html is the day-to-day procedure.

Workflow norms (uncodified but observed)

Patterns that have emerged across the team. Not rules with the same weight as samai's 6, but the way work actually flows when it goes well.

  • Push-on-behalf for ck and samai: only AiCk has wincredman git push credentials. ck / samai SSH to aick, commit on AiCk's working tree, then ask aick in dev chat: aick — push <SHA>. aick verifies diff + pushes.
  • Pull-rebase before commit; stage by name: git add file1 file2 with explicit filenames, never -A — the working tree on AiCk often has scratch files from prior debugging, and -A sweeps them in. Pull-rebase before staging so your commit lands on top of the latest origin/main, not on a stale local copy.
  • Announce shared-file edits in dev chat first: for high-traffic files (base_dashboard.html, store_routes.py, sidebar.html, etc.), drop a one-line heads-up in chat before you start editing so other agents stay clear.
  • Self-echo handling: the Monitor chat tail replays your own posts back. Recognize and don't re-respond — the agent layer should detect own-author and skip the wake-up cycle.
  • Render-deploy swap noise: every push triggers ~30-60s of 502s + "connection forcibly closed" errors on the tail. Benign swap artifact. If 502s persist past ~2 minutes, check the Render API for the deploy status.
  • Address-by-name protocol in dev chat: default to silent unless a message names you (ck / aick / samai) or operationally concerns you. Side-chat between two agents doesn't need a third weighing in. Saves Sam's attention budget for what actually matters.
