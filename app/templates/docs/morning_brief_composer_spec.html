What this page is

Phase 1 Block 6 spec for the morning-brief composer. samai authored; aick implements against this doc rather than from memory. Sister to anomaly_service_spec (the rule library + Signal table) and anomaly_rules (the 32 rule definitions). The composer is the Claude-mediated turn that reads yesterday's Signal rows + today's lookahead + per-audience context and emits a structured daily brief delivered by email + posted to the in-app /partner/briefs surface.

Author / reviewer separation per Sam's directive (Phase 0): samai writes this spec, aick implements it in Phase 1 Block 6 code, samai reviews aick's implementation against this spec. The dataclasses, prompt templates, structured-output schema, and worked examples below are the contract.

1. Composer responsibilities

  1. Pull yesterday's unresolved Signal rows for the requesting audience role + store scope.
  2. Pull today's lookahead (signals projected to fire in the next 12h based on rule-state + historical pattern).
  3. Pull yesterday's calibration stats (rules that fired >3× per store, rules that fired 0 times in the last 14 days).
  4. Call Claude with a structured prompt that maps these inputs into a composed brief.
  5. Validate the response against the structured-output schema; on malformed output, retry once with a clarifying nudge, then fall back to a templated deterministic brief.
  6. Persist the rendered brief to a MorningBrief table (one row per audience per day) and dispatch via email + in-app surface.

The composer is read-only against the Signal table during composition. It does NOT acknowledge or resolve signals as a side effect of brief generation; ack is a separate per-user action.

2. Inputs (the data the composer receives)

2.1 Audience context

@dataclass
class AudienceContext:
    role: str                  # "partner" | "gm" | "km" | "corporate_chef" | ...
    user_id: int
    user_name: str             # for the greeting line
    store_ids: list[str]       # the user's assigned stores (1+ entries)
    store_labels: dict[str, str]  # {"store_1": "Copperfield", "store_2": "Tomball", ...}
    permission_tags: set[str]  # the user's effective permission set (from permissions.py)
    timezone: str = "America/Chicago"
    brief_date: date           # the date the brief covers (yesterday wrt now)

permission_tags matters: a partner sees Frost Bank / legal signals, a GM does not. The composer filters input Signal rows through _user_has(tag) before they reach the prompt — the LLM never sees data it shouldn't.

2.2 Signal payload (per signal, after permission filter)

@dataclass
class SignalForBrief:
    rule_key: str             # e.g. "orders.no_driver_30min_before"
    severity: str             # "info" | "warn" | "alert"
    subject_label: str        # human-readable subject (e.g. "Order #65X-814 — Tomball")
    store_id: str | None      # store_1..store_4 or None for cross-store
    store_label: str | None   # "Copperfield" / "Tomball" / etc.
    trigger_at: datetime
    payload: dict             # rule-specific context (raw measurement values)
    action_text: str          # the one-line action verbatim from the rule definition
    status: str               # "open" | "acknowledged" | "resolved"
    acked_by: str | None      # name of the user who acknowledged, if any
    age_hours: float          # hours since trigger_at

2.3 Lookahead payload (per projected signal)

@dataclass
class LookaheadForBrief:
    rule_key: str
    severity_if_fires: str    # the rule's natural severity
    projected_trigger_at: datetime  # earliest time the rule could fire
    confidence: float         # 0.0-1.0, how confident we are it will actually fire
    subject_label: str        # who/what the projection is about
    rationale: str            # one line of why we're projecting it

2.4 Calibration stats

@dataclass
class CalibrationStats:
    noisy_rules: list[tuple[str, int]]   # [(rule_key, fires_per_day_avg), ...]
    silent_rules: list[str]              # rules with 0 fires in last 14 days
    downgraded_yesterday: list[tuple[str, str, str]]
        # [(rule_key, original_severity, downgraded_to), ...]

3. Output (the structured brief)

The composer returns this exact JSON shape. Tested against the schema before render; on mismatch, retry once then fall back to deterministic template. 

@dataclass
class BriefSection:
    section_kind: str   # "alerts" | "warns" | "lookahead" | "info_aggregate" | "calibration"
    heading: str        # human-readable section header
    intro: str | None   # 1-2 sentence framing (or None)
    items: list[BriefItem]

@dataclass
class BriefItem:
    severity: str            # repeated for renderer; matches input signal severity
    subject_label: str       # echoed
    rule_key: str            # echoed
    one_line: str            # the headline the composer wrote (LLM-generated)
    action: str              # the action_text, possibly re-phrased for tone
    store_label: str | None
    link_path: str           # the surface page the user should click to
    badge: str | None        # optional emoji/tag the renderer applies

@dataclass
class MorningBrief:
    brief_id: str            # uuid, also used as the email Message-ID
    audience_role: str
    audience_user_id: int
    brief_date: date
    greeting: str            # 1 line, addresses the user by first name
    headline: str            # 1-2 sentences; the "if you read nothing else" summary
    sections: list[BriefSection]   # ordered: alerts -> warns -> lookahead -> info_aggregate -> calibration
    closing: str             # 1 line, friendly sign-off
    composed_at: datetime
    composer_model: str      # "claude-haiku-4-5-..." or "claude-opus-4-5-..." (whichever ran)
    fallback_used: bool      # True if the deterministic template ran

4. Model routing

Per Sam's anomaly-service-spec section 13 + reaffirmed in chat 16:54: simple threshold rules → no LLM, pure SQL; narrative summarization of individual signals → Haiku; the daily roll-up brief → Opus. The composer is the daily roll-up, so it defaults to Opus.

Use caseModelWhy
Per-signal one_line headline rewrite (in-line composer subcall, when a rule's raw payload would read poorly verbatim) claude-haiku-4-5 Fast + cheap; just rephrasing 1 sentence from structured data
Daily morning brief composition (the main call) claude-opus-4-5 Narrative quality matters; partner reads this every day; Opus handles audience-shaping and tone selection more reliably
Fallback (Claude unavailable or schema-mismatch retry exhausted) None (deterministic template) Brief still goes out; flagged with fallback_used=True so we can debug post-hoc

Set the model id from a single source of truth at the top of app/services/brief_composer.py so it's easy to bump when Anthropic ships a new minor version. Do not scatter model strings across the file.

5. Prompt templates

5.1 System message (cached)

Static across all audiences for a single brief_date; cached for the 4-hour window during which all 6-10 per-audience briefs are composed. Use Anthropic's prompt caching (cache_control: ephemeral on the system block). Expected hit rate: 5-9 cache hits per cache miss. 

You are the Cenas Kitchen morning brief composer. Once a day (7:30am CT)
you read a structured list of operational anomalies from the prior 24h
plus a lookahead for today, and you compose a per-audience brief that is
short, specific, and prioritized.

WHO READS THE BRIEF: a non-technical partner / GM / KM / FOH manager.
Most read it on a phone over morning coffee. They need to know the 1-3
things that matter, in priority order, and what to do about each.

VOICE: warm, specific, direct. Use the user's first name once in the
greeting and never again. Never use "I" or "we" — you're the brief, not
a colleague. Skip filler. Skip apologies. Skip emoji.

STRUCTURE (return JSON exactly matching the MorningBrief schema):
1. greeting — one line, "Good morning, ." or weekend variant.
2. headline — 1-2 sentences naming the most important thing for today.
   If nothing is urgent, say so directly ("Quiet overnight. Two things
   to keep an eye on today.").
3. sections, in order:
   a. alerts (red, immediate action needed)
   b. warns (yellow, needs attention soon)
   c. lookahead (today's projected alerts/warns based on history)
   d. info_aggregate (a single 1-line count of info-severity signals)
   e. calibration (only if any rules were downgraded or are silent
      enough to mention; otherwise omit entirely — empty section is
      not allowed)
4. closing — one short line.

PRIORITY RULES:
- Within alerts/warns: most recent first, then highest customer-facing
  impact (Google review > order pickup > labor > vendor).
- Skip any signal where status='acknowledged' or 'resolved'.
- Skip any signal more than 24h old at brief_date midnight.
- If an item has no recipient action because the user lacks the
  capability (e.g. a driver-bid signal sent to a partner who doesn't
  reassign drivers), keep it but make the action_text "FYI — no action
  required from you."

LOOKAHEAD RULES:
- Include lookahead items only when projected_trigger_at is within
  brief_date + 12h.
- Re-phrase the rationale as forward-looking: "Expected by 11am if no
  driver is assigned" rather than "Probability 0.7 of firing".
- Lookahead items never carry an action other than the rule's standard
  action.

INFO AGGREGATE:
- Single line: "N info signals in the last 24h. ".
- Never enumerate info items individually; if a partner wants details,
  they click through to /partner/anomalies.

CALIBRATION:
- Include only if at least one downgraded rule fired yesterday OR at
  least one rule has been silent >14d.
- Format: "Tuning notes: rule X was downgraded from alert to warn
  yesterday because it fired 8 times in one day. Rules Y and Z have not
  fired in 14 days — consider retiring." Two lines max.

LENGTH BUDGET:
- Total brief body (excluding greeting + closing): aim 80-180 words
  for a typical day. Hard cap 300. If you hit the cap, drop the
  info_aggregate and calibration sections first, then the lookahead.
- Each item's one_line: aim 12-22 words. Hard cap 35.

DO NOT:
- Restate the schema or your role.
- Invent signals not in the input list.
- Combine signals into "X had a rough day" narratives — keep them
  itemized.
- Speak in second-person plural ("you all"). Singular only.

OUTPUT FORMAT: a single JSON object matching the MorningBrief schema,
no leading/trailing prose, no markdown fences. The serving code parses
with json.loads() and validates against the dataclass.

5.2 User message (per call, not cached)

Includes the audience context, signals, lookahead, calibration. Order matters — signals first because that's what drives the brief. 

AUDIENCE
{audience_context_json}   # one AudienceContext

SIGNALS (last 24h, after permission filter, sorted severity desc, then trigger_at desc)
{signals_json}            # list[SignalForBrief]

LOOKAHEAD (next 12h projected, after permission filter)
{lookahead_json}          # list[LookaheadForBrief]

CALIBRATION (yesterday's tuning observations, partner-only)
{calibration_json}        # CalibrationStats; empty list for non-partner audiences

Compose the brief for {audience_user_name} ({role}, stores: {store_labels})
covering {brief_date}. Return only the JSON object.

All four _json blocks are serialized via Python's json.dumps(..., indent=2, default=str) so datetimes come out as ISO strings the model can parse. Total user-message tokens for a typical day: 1.5k-4k.

6. Per-audience variants

Same prompt + schema; the difference is the input data the composer receives. Audience filtering happens UPSTREAM:

  • partner: gets every signal (no permission filter blocks anything). Calibration section enabled.
  • gm: store-scoped signals for their assigned store(s). No frostbank.*, no legal.*, no developer.*. Calibration section omitted.
  • km / assistant_km: store-scoped, BOH-focused. No sales.* beyond sales.view_today-tagged signals.
  • foh_manager: store-scoped, FOH-focused. No kitchen prep-behind signals.
  • corporate_chef: cross-store kitchen.
  • prep_manager: cross-store prep.
  • expo / driver: no morning brief (audience too narrow to be worth the cost; alerts reach them via push instead).

All audience filters resolve through _user_has(tag) from permissions.py — the composer never hard-codes a role → tag-list mapping.

8. Error handling

  1. Schema-mismatch on first call. Re-call once with a short clarifying user message: "The previous response did not parse as a valid MorningBrief JSON. The error was: {error_msg}. Return only the JSON object, no leading or trailing prose, no markdown fences." If second call also fails, fall through to deterministic template.
  2. Anthropic API error (timeout, 500, rate limit): retry once with exponential backoff (1s, then 5s). On second failure, fall through to deterministic template.
  3. Deterministic template. Renders a plain itemized list of the input signals + lookahead, no LLM-rewrite. Brief still goes out so the partner doesn't lose their morning signal; fallback_used=True is logged.
  4. No signals + no lookahead. Skip the LLM call entirely; render a short canned "All quiet" brief with no sections beyond closing. Saves cost on the dead-quiet days.

9. Caching + cost

Use Anthropic prompt caching. The system message in §5.1 (~2400 tokens) is cached with cache_control: ephemeral; the cache survives 5 minutes, well within the window for composing 6-10 audience briefs back-to-back. Cache hits cost ~1/10 of the system tokens.

Per-brief budget (typical day):

  • System tokens: ~2400 (cached after first call)
  • User tokens: ~1500-4000 (signals + lookahead serialization)
  • Output tokens: ~600-1100 (the JSON brief body)

Opus pricing (2026 rates assumed): roughly $0.05-$0.10 per brief. With 10 audiences per day × 30 days = ~$15-30/month for the composer line item. Cheap enough that we don't second-guess running it.

Cost is logged per call to a BriefCompositionCost table (brief_id, model, input_tokens, output_tokens, cache_read_tokens, cost_usd, composed_at). The /partner/developer/llm-costs page reads this for a weekly rollup.

10. Testing

  1. Golden examples (offline). 4-6 frozen input-to-output pairs covering: typical partner morning, quiet morning, alert-heavy morning, GM-scoped morning, KM-scoped morning, fallback-triggered morning. Stored in tests/fixtures/morning_brief/; the test asserts the schema parses + section ordering + headline length is within bounds. NOT an exact-string match (LLM output varies).
  2. Schema fuzz. The validator function parse_brief(json_string) -> MorningBrief is tested against 30+ malformed inputs (missing keys, wrong types, empty sections, extra prose around the JSON). Must raise BriefSchemaError with a clear message on each.
  3. Permission-filter unit tests. Build a fake AudienceContext with a manager role + a synthetic signal carrying legal.view as its surface tag; assert that the signal is filtered out before the prompt is built.
  4. Cache-hit smoke test. Call the composer twice back-to-back; assert the second call's response metadata reports cache_read_tokens > 0 (Anthropic's API surfaces this in the usage block).
  5. Fallback path test. Mock the Anthropic client to raise on every call; assert the brief still renders via the deterministic template, fallback_used=True, and the partner still receives the email.

11. Delivery surfaces

  • Email. SMTP via the existing SiteGround setup; partner receives at sam@cenaskitchen.com, GMs at their assigned addresses. HTML body renders the brief through a Jinja template (the in-app surface re-uses the same partial). Subject line: "Cenas brief — {brief_date} — {alert_count} alerts / {warn_count} warns".
  • In-app surface. /partner/briefs lists the last 30 days of briefs; clicking opens the rendered brief in-page with ack buttons next to each item (which call the same acknowledge-signal endpoint ck builds for the per-page anomaly cards in Phase 1 Block 3).
  • Push. Out of scope here; Phase 2 if needed. Alert-severity items already get push from their own rule path.

When aick posts "Block 6 complete", samai runs this 14-item check:

  3. Dataclasses in types.py have the field names and types from §2-3.

  9. Fallback renderer produces a valid MorningBrief with fallback_used=True when the Anthropic client raises.
  10. No-signals-no-lookahead path skips the LLM call entirely (assert via mock: Anthropic client invoke count = 0).
  11. Cost log row written per call to BriefCompositionCost with non-null tokens fields.
  12. /partner/briefs renders the last 30 days; clicking a brief opens its detail with ack buttons.
  13. Email dispatch fires the same SMTP path the rest of the app uses; subject-line format matches §11.
