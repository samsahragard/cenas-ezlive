Page Guide — Cenas Kitchen

Generated 2026-05-21 by dck; refreshed 2026-05-23 by aick (added /sam/combined, the four tabbed bottom-nav dashboards — Today, Manager, Catering, Operations — and updated the Vendors entry to reflect its tabbed-dashboard rework). For every page in the app, this guide answers four questions — (1) the page's function and why it was built, (2) what it connects to, (3) how it works and where its data comes from, (4) what it gives the user — with reasoning and technical specs. Built from the live Flask route handlers and Jinja templates. A companion to the Site Map doc, which lists the same pages without the deep detail.

1. Store, Kitchen, Schedule & Core Pages

Scope: every user-facing page defined in app/web/store_routes.py — the per-store URL-prefix blueprint (store_bp, prefix /<store_slug>). The <store> segment is one of dos (Tomball), uno (Copperfield), corporate (both, GM view), or partner (both, owners-only, second-factor gated via auth.partner_login). Most routes here are thin wrappers that set g.current_store / g.current_location then delegate to a shared view; the analysis names the real data source behind each.

Store Home Dashboard — /<store>/

1. Function & why it exists. The per-store manager dashboard — the landing page for a store context. It surfaces today's catering deliveries, upcoming-delivery groups, KPI tiles (Tomball/Copperfield delivery counts, heads), and an "attention" list of today's orders missing a customer name. It exists so a manager opening the site sees the day's agenda first instead of bare navigation.

2. Connected to. Reached by clicking a store on the store picker (/); links out to every sidebar section (Orders, Ezcater, Schedule, Kitchen, Vendors, manager pages). The retired /review route redirects here. It is the home target of the per-store gate redirects.

3. How it works — data & refresh. home() delegates to ezcater_routes.home(), which queries the Order model directly via get_db(), filtered by delivery_date and origin_store_id (Tomball = store_2/store_4, Copperfield = store_1/store_3). "Today" is computed in Central Time (UTC-5) so the day doesn't flip after 7pm. Data refreshes on each page load; orders themselves arrive via the ezCater webhook into the Order table.

4. What it gives the user. A manager sees, at a glance, how many deliveries are due today per store, who is driving them, total headcount, and which orders have data problems (no customer) that the kitchen needs fixed — turning the app's front door into an actionable shift briefing.

Today Dashboard — /<store>/today (also ?tab=<key>)

1. Function & why it exists. Tabbed shell that gathers the per-store "today" surfaces (Dashboard, Notifications, Task Reports, Cena) into one bottom-nav landing. Each tab embeds the real underlying page inline in an iframe. Per Sam 2026-05-21 #8:03 — the goal is one entry-point per major section instead of forcing managers to drill through the sidebar. Tab gating mirrors the Today sidebar group: Task Reports only renders for viewers holding team_reports.view; the Cena tab only for the Sam-chat user. Dashboard + Notifications always render.

2. Connected to. The bottom-nav Today tab and the sidebar Today group both open this URL. Each tab iframe-embeds: Dashboard → Store Home Dashboard; Notifications → /<store>/partner/notifications; Task Reports → /<store>/partner/team-reports/; Cena → /sam/chat (Sam-only). Deep-links via ?tab=<key>.

3. How it works — data & refresh. today_dashboard() is a thin shell: builds the visible tab list (key, label, url) applying the same gates the sidebar uses, picks the active tab from ?tab= (defaulting to Dashboard), renders the wrapper template. No DB session is opened here — the iframed child pages run their own queries when the browser loads them. Refresh happens at the child-page level.

4. What it gives the user. A single bottom-nav stop that surfaces every "today" view a manager needs (live agenda, alert inbox, weekly reports, Sam's Cena chat where applicable) without separate sidebar trips — same data, one click.

4. What it gives the user. A manager opens one URL and can flip between every operational log type as tabs — daily log, handoffs, goals, audits, incident reports, counseling, attendance, training, maintenance — without leaving the page.

Sales Report — /<store>/sales (also /<store>/reports/sales and /reports/sales/<channel>)

1. Function & why it exists. The multi-channel sales report. Per Sam's spec, /uno/sales lands directly on the real report (not a card grid), with channel multi-select (Toast in-store, Toast Online, DoorDash, Uber, ezCater, etc.) and Today/This Week/Last Week period pills. It exists to give per-store revenue visibility across every ordering channel from one share-friendly URL.

2. Connected to. Reached from the sidebar Sales group and its sub-nav (each channel chip is a ?channels= link). Sibling of the Labor and Performance reports. The Sales landing and the /reports/sales/<channel> wrapper both render this same view.

3. How it works — data & refresh. sales_landing() sets g.location_override then delegates to reports.third_party_sales(), which calls toast_reports.third_party_sales_report(start, end, location, channels=...) — i.e. the Toast API is the data source. Channel/period state lives in the URL query string (channels=, period=, start/end). Data is fetched per page load / per Run. Note: the ezCater channel returns a "coming in Phase 3" notice because per-item order pricing isn't captured yet.

4. What it gives the user. A manager or owner can compare revenue by channel and time window for a single store, seeing where sales come from (dine-in vs delivery apps vs catering) without logging into Toast directly.

Labor Report — /<store>/labor (also /<store>/reports/labor and /reports/labor/<which>)

1. Function & why it exists. The labor-cost report, landing directly on the report with Today/Week/Last-Week and BOH/FOH/All pills. It exists so managers can watch labor cost against revenue, with management-pay data redacted unless viewed under /partner/.

2. Connected to. Reached from the sidebar Labor group and its Total/BOH/FOH sub-nav. Sibling of Sales and Performance. The Labor landing and /reports/labor/<which> wrapper render this same view.

3. How it works — data & refresh. labor_landing() sets g.location_override, then reports.labor() calls toast_reports.labor_report(start, end, location, role_filter=, redact_management=). Source is the Toast labor API. redact_management is true unless g.current_store == "partner". Fetched per page load; the _per_store_gate before_request hook hard-blocks Expo accounts from this URL (403).

4. What it gives the user. A view of labor spend split by back-of-house vs front-of-house for any window, so managers can spot overstaffing — and owners (partner view) see the full picture including management pay.

Produce Order Guide — /<store>/produce/

1. Function & why it exists. The daily produce order guide — a one-click ordering surface that shows each produce item already priced at the cheaper of the two vendors (Alvarado vs J. Luna). It exists so kitchen managers order produce fast at the best price without comparing vendor sheets by hand.

2. Connected to. Reached from the Vendors landing / sub-nav. Sibling of Produce Price History (/produce/orders). POST sub-routes (submit/confirm/cancel) under /produce/<subpath> 302 to the top-level /produce/ blueprint, which has its own confirmed/canceled templates.

3. How it works — data & refresh. produce_root() renders produce_order.index() inline (so the store sidebar context survives). index() calls load_winners() to compute the cheaper-vendor "winner" per item, plus get_managers(), get_locations(), and get_vendor_routing() (test vs live mode). Renders produce/order_guide.html. Cart submit POSTs JSON to /produce/submit, validated against the winners list. Pricing data updates when vendor price sheets are ingested.

4. What it gives the user. A kitchen manager picks quantities on a pre-priced grid and submits one order; the app has already chosen the cheapest vendor per line, so ordering is fast and cost-optimized.

Produce Orders & Price History — /<store>/produce/orders

1. Function & why it exists. A produce price-tracking report: latest pricing with the winning vendor per item, "biggest movers" (week-over-week price moves of >=5%), and a per-item price-history chart. It exists so the team can see how produce costs trend and catch sudden price spikes.

2. Connected to. Reached from the Vendors landing / sub-nav; sibling of the Produce Order Guide. Org-wide (vendor pricing isn't location-specific), but the per-store URL prefix is kept for sidebar symmetry.

3. How it works — data & refresh. produce_orders() delegates to reports.produce_orders(), which calls produce_history.latest_prices(), produce_history.biggest_movers(threshold_pct=...), produce_history.list_distinct_items(), and produce_history.history_for_item(name, size, days=). Renders reports_produce_orders.html with a Chart.js history graph. Query params: item, days (7–365, default 90), threshold (default 5.0). Updates as new vendor price data is recorded.

4. What it gives the user. Whoever buys produce can see which vendor is cheaper right now, which items just jumped in price, and the full price trend for any single item — supporting smarter purchasing and vendor negotiation.

Orders List — /<store>/orders

1. Function & why it exists. The per-location catering order list — today's plus upcoming ezCater orders grouped by delivery date. For partner/corporate (location = both) it shows a combined Tomball + Copperfield view so Ezcater→Orders is a functional landing for those users. It exists as the operational worklist for catering deliveries.

2. Connected to. Reached from the Ezcater landing / sub-nav. Each order links to the single-order view (/orders/view/<external_order_id>). Related to the Order Processor and the store home dashboard (which surfaces the same orders).

3. How it works — data & refresh. orders_list() branches on g.current_location: if "both" it calls orders_browse.list_orders_for_location() for each store, merges, runs group_orders_by_date() and rotated_dispatch_letters(); otherwise delegates to orders_browse.location_orders(location). Source is the Order table, populated by the ezCater webhook. Renders orders_by_store.html. Refreshes on page load.

4. What it gives the user. A catering manager sees every delivery for their store grouped by date, with dispatch-letter driver assignments — the day-to-day list for running catering fulfillment.

Order Processor — /<store>/orders/processor

1. Function & why it exists. The legacy PDF-order ingest surface — upload a PDF catering order to be parsed into the system. The webhook is now the primary intake path, so this is a fallback for orders that arrive as PDFs. It exists to keep a manual ingest route for non-webhook orders.

2. Connected to. Reached from the Ezcater landing / sub-nav ("Order Processor" card). Feeds the same Order table the Orders List and dashboard read from.

3. How it works — data & refresh. orders_processor() (GET+POST) renders ezcater_routes.orders() inline so the store sidebar context is preserved. The underlying view handles PDF upload and parsing into Order rows. Updates on manual submit/upload.

4. What it gives the user. Staff can still get a one-off or legacy PDF catering order into the system when it didn't come through the ezCater webhook, so no order is lost.

Driver Payroll List — /<store>/driver-tracking

1. Function & why it exists. A per-driver delivery-payroll roster. Lists every known ezCater driver for the store with their delivery count and pay total for the just-closed bi-weekly period. It exists to make driver pay reviewable per period — Sam's spec: show the period being paid out next, not the in-progress one.

2. Connected to. Reached from the Ezcater landing / sub-nav. Each driver name links to that driver's paycheck-history page (/driver-tracking/<known_id>). Related to Drivers Admin (which also shows ezCater identity).

3. How it works — data & refresh. driver_tracking() queries EzcaterKnownDriver scoped by ck_prefix (Tomball = 2, Copperfield = 1), matches each to a Driver by normalized phone, and uses ezcater_payroll.period_containing() / previous_period() / paycheck_for() to compute the closed period's totals. Renders driver_payroll_list.html. Refreshes on page load; pay rows derive from Order.ezcater_driver_name matches.

4. What it gives the user. Whoever runs payroll sees, per driver, how many deliveries and what total pay is owed for the period closing next — and can drill into any driver's full paycheck detail.

Driver Paycheck — /<store>/driver-tracking/<known_id> (also /<store>/driver-paycheck/by-driver/<driver_id>)

1. Function & why it exists. A single driver's bi-weekly paycheck history — the current period plus the previous five, each broken into one row per delivery (tracking status, extra miles, base, bonuses, total). It exists so a driver's pay is fully auditable line by line.

2. Connected to. Reached by clicking a name on the Driver Payroll list, or from the Payroll column on /partner/drivers. The by-driver variant is keyed on Driver.id so deactivated/deleted drivers still surface history.

3. How it works — data & refresh. Both routes render driver_paycheck.html. driver_paycheck() looks up EzcaterKnownDriver; driver_paycheck_by_driver() looks up Driver. Both call ezcater_payroll.paycheck_history(name, periods=6), which matches by driver name against Order.ezcater_driver_name. Store-scope guards 404 a cross-store peek (Tomball manager can't see a Copperfield driver). Refreshes on page load.

4. What it gives the user. A manager (or the driver) can verify exactly how a paycheck was built — every delivery, its miles and bonuses — across six pay periods, making pay disputes easy to resolve.

Drivers Admin — /<store>/drivers

1. Function & why it exists. The driver account-admin page: list drivers, reset passcodes, deactivate/reactivate, and edit driver fields (location/email/phone/address). Active/Inactive tabs with counts. It exists so managers can self-serve driver-account maintenance instead of needing a developer.

2. Connected to. Reached from the Ezcater landing / sub-nav. Driver names link to shift history (/drivers/<id>/shifts) and the Payroll column links to /driver-paycheck/by-driver/<id>. POST actions (/drivers/<id>/update, /reset, /toggle-active) return here.

3. How it works — data & refresh. drivers_admin() (gated by @requires_permission("drivers.admin")) queries the Driver model filtered by active status and location. It joins EzcaterKnownDriver by normalized phone to show a "Verified ezCater driver" badge and a "CK #X · Store" label, and pulls each driver's latest DriverShift. Renders driver_admin.html. Refreshes on page load / after each POST.

4. What it gives the user. A manager can add, fix, reset, or deactivate driver accounts for their store and instantly see which drivers are verified against the ezCater roster — keeping the driver list clean and self-managed.

Drivers Live Map — /<store>/drivers-live

1. Function & why it exists. A live GPS map showing a marker per driver currently on shift, auto-refreshing. It exists so a manager can see where catering drivers physically are in real time during a delivery rush.

2. Connected to. Reached from the Ezcater landing / sub-nav. Related to Driver Shift History/Playback (the recorded-track counterpart) and Drivers Admin.

3. How it works — data & refresh. drivers_live() renders drivers_live.html centered on the store (_STORE_CENTRES — Tomball 30.1118,-95.6230; Copperfield 29.8730,-95.6428). The page JS polls drivers_live_positions() (/drivers-live/positions.json) every 5000 ms (REFRESH_MS). That endpoint joins open DriverShift rows to Driver and returns each driver's most recent DriverLocation fix (lat/lng, accuracy, speed, heading); fixes older than 30 seconds are flagged stale.

4. What it gives the user. A manager watches drivers move on a map and can tell who is en route, who has stalled, and whose GPS has gone stale — useful for dispatch decisions and answering "where's my order" calls.

Driver Shift History — /<store>/drivers/<driver_id>/shifts

1. Function & why it exists. A list of one driver's recent shifts (up to 60), each with a summary: duration, GPS point count, distance traveled, and max speed. It exists so a manager can review a driver's recent activity and pick a shift to replay.

2. Connected to. Reached from Drivers Admin (clicking a driver). Each shift row links to the shift playback page (/drivers/<id>/shifts/<shift_id>).

3. How it works — data & refresh. driver_shifts() enforces store scope via _scoped_driver(), then queries DriverShift for the driver (newest 60). Per shift, _shift_summary() sums consecutive haversine hops over DriverLocation points for distance and finds max speed_mps. Renders driver_shifts.html. Refreshes on page load.

4. What it gives the user. A manager gets a recent-shift logbook per driver — how long each shift ran and how far the driver traveled — useful for verifying delivery activity and mileage.

Driver Shift Playback — /<store>/drivers/<driver_id>/shifts/<shift_id>

1. Function & why it exists. A map playback of a single recorded driver shift — the route the driver actually took, with the shift's summary stats. It exists so a manager can audit exactly where a driver went during one shift.

2. Connected to. Reached from Driver Shift History. The map track is fed by the sibling JSON endpoint /drivers/<id>/shifts/<shift_id>/track.json.

3. How it works — data & refresh. driver_shift_playback() validates the shift belongs to the scoped driver, builds _shift_summary(), and renders driver_playback.html. The page loads the full GPS path from driver_shift_track(), which returns every DriverLocation point (lat/lng, captured_at, accuracy, speed) for the shift, ordered chronologically. Static once loaded (it's a recorded shift).

4. What it gives the user. A manager can replay a completed delivery shift on a map to confirm a route, investigate a long delivery, or verify mileage claims.

Vendor Recent Orders — /<store>/vendors/<vendor>/recent-orders

1. Function & why it exists. A per-vendor recent-orders list for restaurant-supply vendors (Webstaurant, Performance Food, Restaurant Depot, Specs). It exists to track supply orders per vendor; the page is live even before its email parser is fed samples, so the navigation flow is intact.

2. Connected to. Reached from the Vendors section. Vendor chips switch between the four vendors. Sibling of the Produce pages under Vendors.

3. How it works — data & refresh. vendor_recent_orders() validates the vendor slug against _VENDOR_LABELS, then queries VendorRecentOrder filtered by vendor and store scope (store_scope = location or NULL), newest 100 by placed_at. Renders vendor_recent_orders.html. Rows are populated when vendor order-confirmation emails are parsed; renders an empty "no orders yet" table until then.

4. What it gives the user. A manager sees a running history of what was ordered from each supply vendor — once parsers are fed, this becomes the supply-order record per vendor.

Manager Section Page (list) — /<store>/manager/<page>

1. Function & why it exists. A shared text-heavy list view for the generic Manager-section pages (shift-handoff, supply-requests, daily-goals, staff-feedback, pre-shift-checklist, close-of-day-audit, recipe-page, interview, training, maintenance). Newest entries first, store-scoped. It exists as the v1 "approach A" surface so all 14 manager log types ship before per-page custom designs.

2. Connected to. Reached from the Manager sidebar section. Each row links to its detail view (/manager/<page>/<id>); a "new" link opens the create form. Four pages (daily-log, incident-reports, counseling, attendance) are special-cased to richer v3 views described separately below.

3. How it works — data & refresh. manager_page_list() resolves the slug to a SQLAlchemy model via _manager_model_for_slug() (all share ManagerLogMixin), gated by _manager_role_ok() (Expo/drivers denied). Queries the model store-scoped, newest 100, renders manager_log.html. Refreshes on page load / after a create.

4. What it gives the user. A manager gets a simple running log per topic (handoffs, supply requests, goals, etc.) for their store — a lightweight shared notebook before each page gets a purpose-built design.

Manager Section Entry Detail — /<store>/manager/<page>/<entry_id>

1. Function & why it exists. The read view for a single manager-log entry — its title, body, and tag. It exists so a logged manager note can be opened and read in full.

2. Connected to. Reached by clicking a row on the manager-section list. Links to the edit form (/manager/<page>/<id>/edit) for the author or partner/corporate.

3. How it works — data & refresh. manager_page_detail() loads the row by id from the slug's model, enforces store scope (404 on mismatch), and renders manager_log.html with entry set. Static read; updated only when the entry is edited.

4. What it gives the user. A manager can review one specific log entry in detail — the permanent record of a handoff, audit, or feedback note.

Manager Section Entry Form (new / edit) — /<store>/manager/<page>/new and /<store>/manager/<page>/<entry_id>/edit

1. Function & why it exists. The create/edit form for a generic manager-log entry (title, body, type tag). It exists so managers can add new entries or correct their own.

2. Connected to. "New" reached from the manager-section list; "edit" reached from an entry detail. Both POST back to manager_page_create / manager_page_update, returning to the list or detail.

3. How it works — data & refresh. manager_page_new() and manager_page_edit() render manager_log.html with form_mode "new"/"edit". Edit is restricted to the entry author or partner/corporate roles. Submitting writes a ManagerLogMixin row (store_scope, author_id stamped). The incident-reports slug is special-cased to a standalone v4 form (see below).

4. What it gives the user. A manager can quickly file a new note or fix a mistake in one they wrote, keeping each manager log accurate and current.

Daily Manager Log — /<store>/manager/daily-log

1. Function & why it exists. A v3 12-day, day-grouped manager log with structured entry fields (module/subject/issue/priority) and image attachments. It diverged from the shared shell because the daily log needs a calendar-windowed, photo-capable format. It exists as the running daily record of store issues and events.

2. Connected to. Reached from the Manager section. Entry images are served by /manager/daily-log/image/<image_id>. Entries can be flagged show_on_roster, surfacing them on roster views.

3. How it works — data & refresh. _render_daily_log_v3() queries DailyManagerLog for a 12-day reverse-chronological window (?date= sets the window end, default today), store-scoped, grouped by entry_date, rendered into manager_daily_log.html. Creating an entry (_create_daily_log_v3_entry()) writes a DailyManagerLog row plus DailyLogEntryImage rows; uploaded images (.png/.jpg/.jpeg/.webp/.gif, max 8 MB each) are saved to DAILY_LOG_IMAGE_DIR (default under CHAT_ATTACHMENTS_DIR/daily-log-images/<entry_id>/). Updates on manual submit.

4. What it gives the user. A manager keeps a structured, photo-backed daily diary of what happened in the store — categorized and prioritized — that the next manager and owners can review across the last 12 days.

Incident Reports — /<store>/manager/incident-reports
