What this page is

Phase 0 spec for the unified role-based permission system. samai wrote this for ck to lift directly into code in Phase 0 Block 4 (decorator + template helper + role-to-tag dict) and for aick to use as the source of truth when retiring ad-hoc role checks scattered across blueprints. Every claim made about a role's capabilities in this doc is meant to be the authoritative answer; if the app diverges, the app is wrong.

Scope: this is a spec, not the implementation. No code in app/ changes from this doc alone. ck's Block 4 turns the role-to-tag mapping below into a Python dict + a @requires_permission("x.y") decorator + a {% raw %}{% if has_permission('x.y') %}{% endraw %} Jinja helper + a per-template audit pass that migrates existing partner-only / expo-hidden / GM-vs-manager checks onto the decorator.

1. Role hierarchy

Ordered top-down, most authority first. A role inherits nothing implicitly — the role-to-tag mapping in §3 is the only source of permissions. The hierarchy here is a coarse organizational summary, not a parent-child permission inheritance graph.

  1. partner — owner-operator (Sam, Masood). Cross-store, cross-domain. Sees personal partner data (Frost Bank, legal matters, payroll detail). Wildcard permission “*” grants every tag below.
  2. corporate — non-partner corporate staff (e.g. office manager, controller). Cross-store, broad operational reach. Excluded from partner-personal scopes: no Frost Bank, no legal, no payroll detail beyond labor reports, no partner-scope AI.
  3. gm — General Manager. Owns one store primarily; may have read-only visibility into the other. Full operational permissions within their store: orders, labor, sales, drivers, produce, KDS, manager log, AI ops queries.
  4. km — Kitchen Manager. Single store. BOH-focused: labor (BOH only), prep, produce ordering + invoice verification, KDS, manager log (kitchen entries), recipe edits.
  5. assistant_km — Assistant Kitchen Manager. Subordinate to KM. Same store. Read access to most KM scopes; write access narrower (no produce ordering finalization, no recipe edits, no labor cost approvals).
  6. corporate_chef — cross-store kitchen oversight (Javier today). Multi-store: every store is in scope. Same capability shape as KM but applied to all stores. Read access to all-store aggregates.
  7. prep_manager — prep team lead (Gina today). Cross-store. Prep-scope: KDS prep view, recipe lookups, produce receiving (not ordering), prep manager log entries. Limited labor view (own + direct reports).
  8. foh_manager — Front-of-House Manager. Single store. FOH-focused: server performance, FOH labor, FOH manager log, customer-facing systems (Google reviews surface). No BOH labor, no kitchen-side produce, no recipe edits.
  9. expo — order expediter. Single store. Order-flow only: order view, mark-picked-up, mark-en-route, mark-delivered. No labor visibility, no sales totals, no driver admin, no manager log write. KDS view only.
  10. driver — delivery driver. Cross-store (drivers can bid on any store's orders). Driver portal only: /ez-market for Available / My queue / History tabs, /driver/logs, /my-profile. No back-office, no labor, no sales, no manager log.

Multi-store users (corporate, partner, corporate_chef, prep_manager, driver) work via a user_store_assignments mapping (or its equivalent) — one row per (user_id, store_id, role) pair. The decorator evaluates store-scoped permissions against the union of the user's assigned stores. See §4 for the mechanic and §6 for examples.

2. Permission tag namespace

Tags use dot-separated keys: <domain>.<action>. The domain is broad (labor, orders, legal, ai). The action is specific (view_own_wage, assign_driver, view, ask_claude_partner_scope). 60 tags total, grouped below.

Naming rules:

  • view_* = read access (page load, JSON endpoint, export).
  • view_own_* = read access scoped to the requesting user's own rows (own wage, own hours, own mailbox).
  • view_others_* = read access to other people's rows. Strictly stronger than view_own_*; granting view_others_* implies view_own_* at the decorator level (ck implements this by listing both tags in the role bundle — no implicit subsumption logic).
  • edit / add / upload / delete = write actions, each separable.
  • admin = umbrella write tag for a domain (drivers.admin covers reset_passcode + view_roster + edit driver profile etc.). Used sparingly; prefer fine-grained tags when ck or aick needs to gate a single page.
  • _partner_scope suffix marks an elevated variant of an otherwise generic tag (e.g. ai.ask_claude vs ai.ask_claude_partner_scope). The base tag is the public capability; the suffixed tag adds partner-personal context.

2.2 Tier rule for global-config tags (added 2026-05-13)

Rule: any tag that gates a global-config surface (system tuning, user management, developer / observability tools, cross-store admin) belongs to partner + corporate only. gm and below operate within store scope and never tune system-wide rules or manage all users.

This generalizes the existing pattern in §3 where developer.view_chat, developer.view_app_docs, access.approve_request, and access.deny_request are partner+corporate only and absent from gm / km / asst_km / corporate_chef / prep_manager / foh_manager / expo / driver. The rule is now spelled out so future global-config tags inherit by reference rather than relitigation.

Tags currently covered by the rule: developer.view_chat, developer.view_app_docs, access.approve_request, access.deny_request, anomaly.admin (added 2026-05-13 by ck for the /partner/anomalies/rules surface), access.team_admin (added 2026-05-13 by ck for the /partner/team Team UI). developer.view_render, access.impersonate, and frostbank.view stay partner-only via wildcard — partner-only is the stricter case of the same family (system internals + financial / impersonation surfaces stay above the corporate tier).

Future tags fitting this pattern (any new admin surface that crosses store boundaries or tunes the platform itself) should land in corporate's set by default. Partner-only stays reserved for system-internal surfaces (Render env, impersonation, partner-personal financials).

4. Per-store scope

Every tag in §2 has a scope column: own, store, or global.

  • own: the tag only grants access to rows where the requesting user is the owner (own wage, own hours, own bids, own mailbox, own transcripts). Scope evaluation happens inside the route handler against g.current_user.id; the decorator's job is just to confirm the tag is in the role's set.
  • store: the tag only grants access when the target row's store_id matches one of the user's assigned stores. The decorator takes an optional store_arg kwarg pointing to the route arg that carries the store identifier (e.g. @requires_permission("orders.view", store_arg="store_slug")). If absent, the decorator falls back to g.current_location and validates against the user's assignment set. Multi-store users (corporate, partner, corporate_chef, prep_manager) pass if any assignment matches; single-store users must match exactly.
  • global: no store filter applied. Granting the tag is sufficient. Legal, AI partner scope, Frost Bank, developer tools, access approval, and impersonation are global.

Special case — cross-store visibility for single-store managers. A GM may need read-only visibility into the other store for context (e.g. spot-checking the sister store's prep). Implement by issuing a second assignment row (user_id, other_store_id, role='gm', read_only=true); the decorator honors the read_only flag by stripping write tags from the assignment's effective set. This is per-assignment, not per-user — a GM with a read-only secondary assignment still has full write at their primary.

5. Denial behavior

5.1 Route decorator

Shape recommended for ck (Block 4): 

from functools import wraps
from flask import g, redirect, url_for, current_app

def requires_permission(tag: str, *, store_arg: str | None = None,
                        scope: str | None = None):
    """Gate a route on a permission tag.

    tag        - the permission key (e.g. "orders.assign_driver")
    store_arg  - route kwarg name carrying the store identifier; if set,
                 the user's assignment for that store is checked.
    scope      - extra qualifier the route handler interprets (e.g.
                 "own_bids" for /ez-market driver views). The decorator
                 doesn't enforce scope; it just makes it available via
                 g.permission_scope so the handler can filter rows.
    """
    def deco(fn):
        @wraps(fn)
        def wrapped(*args, **kwargs):
            user = getattr(g, "current_user", None)
            if user is None:
                return redirect(url_for("auth.partner_login"))
            if not _user_has(user, tag, kwargs.get(store_arg) if store_arg else None):
                return redirect(url_for("auth.access_denied", need=tag))
            g.permission_scope = scope
            return fn(*args, **kwargs)
        return wrapped
    return deco

_user_has(user, tag, store_id) resolves the user's effective permission set: wildcard "*" matches anything, otherwise the tag must be in the union of the user's assignment role sets. For store-scoped tags, the assignment's store_id must equal the target store_id (or the user must have a matching assignment with read_only=false for write tags). The impersonation case is handled inside _user_has by reading session.get("impersonating_user_id") and evaluating as that user instead.

5.2 Template helper

Jinja-side gate for hiding UI: 

{% raw %}{% if has_permission('orders.assign_driver') %}
  <button class="assign-driver">Assign Driver</button>
{% endif %}{% endraw %}

Register has_permission via app.jinja_env.globals['has_permission'] = .... Implementation delegates to the same _user_has used by the decorator, pulling the current user from g. Optional second arg for store override: has_permission('labor.view_others_wage', store='copperfield').

5.3 /access-denied page

Already exists (per Sam's spec). Behavior:

  • Query string: ?need=orders.assign_driver.
  • Page reads request.args.get('need') and displays: “You need permission 'orders.assign_driver' to access this page. Contact a partner to request access.” with a button that opens /request-access pre-filled with the needed permission.
  • Logs every denial to a new PermissionDenial table: user_id, tag, route, denied_at. Surfaces on /partner/developer/app/denials (a new doc/admin page) so partners can see what's getting blocked and decide if a role's tag set needs adjustment.

6. Edge cases

6.1 Javier (corporate chef, both stores)

Javier has two assignment rows: (javier, copperfield, corporate_chef) and (javier, tomball, corporate_chef). Effective permission set is the union of both (same set, since both rows have the same role). For any store-scoped tag check, either assignment matches, so the decorator passes regardless of which store the route is targeting. For cross-store reports (labor.view_all_stores, manager_log.read_all_stores), Javier has those globally because they're in corporate_chef's set in §3.1.

6.2 Gina (prep manager, both stores)

Same shape as Javier — two assignments, union of permissions. Notable divergence: prep_manager does not have produce.order (only produce.invoice_verify), so Gina can verify a delivery arrived correctly but cannot place a new produce order. KM owns ordering. Both Javier and Gina see manager_log.read_all_stores because their work crosses store boundaries; their manager_log.write defaults to their current page's store context (route kwarg).

6.3 Partner impersonating a manager (support / debug)

Partner clicks "Impersonate" on a user's admin row (gated by access.impersonate, partner-only). Server:

  1. Verifies caller has access.impersonate.
  2. Sets session['impersonating_user_id'] = target.id.
  3. Writes a row to ImpersonationSession: (actor_id, target_id, started_at, route). End time written on "Return to partner" click.
  4. Every subsequent request: _user_has evaluates permissions as the target user. Routes still see g.current_user as the target (so e.g. /my-profile shows the target's profile).
  5. UI: a sticky red banner at the top of every page — "Impersonating {target_name}. Return to partner." Banner is rendered from base_dashboard.html when session.get('impersonating_user_id') is set.
  6. Audit: every permission check during the impersonation window is logged with both actor_id (the partner) and effective_user_id (the target) in PermissionCheckLog. This is what protects against after-the-fact disputes about what was viewed under whose authority.

What impersonation does NOT do: it does not allow the partner to act with permissions they themselves don't have through the target. The check is “does the impersonated user have this tag,” but a route like access.impersonate still checks the actor's own permission set (not the target's). Concretely: a partner impersonating a driver cannot start a second impersonation; the driver doesn't have access.impersonate.

6.4 Driver self-service vs back-office driver actions

Drivers have orders.mark_picked_up and orders.mark_delivered but scoped to their own bids (scope argument own_bids in the decorator). Back-office staff (GM, expo, etc.) have the same tags without the scope — they can mark any order. The handler reads g.permission_scope and filters accordingly: own_bids ⇒ where driver_id = g.current_user.id; absent ⇒ no filter.

6.5 A user with no role assignment

Logged-in but no role row yet (mid-onboarding, or post-deny). Effective permission set is empty. Every decorator check fails, redirecting to /access-denied. The user can still hit /request-access and /my-profile (both explicitly exempt from the decorator). /my-profile renders the "Your access is pending" state.

8. What this spec doesn't decide (kicked to Phase 1+)

  • Time-based permissions — e.g. expo only has orders.mark_picked_up during their shift. Out of scope here; if needed, layer on top via a shift_active gate, don't contaminate the tag namespace.
  • Customer-facing scopes — corporate-order / <store>/corporate-order has its own gate (the Partner / Corporate g.current_store branch in corporate_order.py). Don't migrate this onto the decorator in Phase 0; let it stay as-is until we have a customer auth design.
  • Per-tag onboarding flows — e.g. a new foh_manager sees a 30-second tour the first time they hit /reports/server-performance. Tour gating is independent of permission gating.
  • Role hierarchy as code — this spec presents the order as documentation. We do NOT implement “partner inherits everything in corporate's set” via automatic union. Each role explicitly enumerates its tags. The hierarchy is a sorting aid for humans, not a logic shortcut.
  • Multi-tenant / franchisee future — if Cenas ever licenses the platform to a third party, the tag namespace stays the same; the multi-tenancy adds an org_id scope above store_id. Out of scope for Phase 0.
