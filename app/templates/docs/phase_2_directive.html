1C. Ribbon content router

A new module app/services/ribbon.py with:


def ribbon_items_for(page_slug, user, store_scope, category=None) -> list[RibbonItem]:

    """Returns ordered list of items relevant to (page_slug, user, store, category).

    If category is None, returns all categories. Otherwise filters to that category.

    

    Items pulled from: Task, Signal (anomaly engine), SalesInsight, ScheduledEvent.

    Items filtered by: user role + store_scope + page_slug relevance.

    Items rendered with: role-aware framing (see "Cross-cutting framing rule" below).

    Items sorted by: severity DESC, then deadline ASC.

    """

Register as Jinja global so templates can call {{ ribbon_items_for(...) }}.

Cross-cutting framing rule (Sam's "SPECS by Andres" example):

A single task or signal renders DIFFERENTLY depending on whose ribbon it appears on:

  • On the owner's to-do: action-required framing ("Place SPECS order by 5pm")
  • On the owner's category-specific ribbon (e.g., vendors page): informational framing ("Owner: you, due 5pm")
  • On a manager's vendors page: status framing ("SPECS order to be placed by Andres at 5pm")
  • On a manager's to-do: nothing — not their task unless escalated

Implement as a render_for(user) method on RibbonItem that returns the appropriate text + styling.

1D. X / Check controls on every ribbon item

Each ribbon item renders with two controls:

  • X — dismiss for current session. Writes to RibbonItemDismissal. Item disappears from this user's ribbon until they re-open the app (or until the next daily reset, samai's call which pattern is cleaner).
  • Check — mark complete. For tasks: sets completed_at, completed_by_user_id. For signals: triggers SignalAck. For sales insights: dismisses permanently (insights are intrinsically time-bound).

POST endpoints:

  • POST /partner/ribbon/dismiss/<item_type>/<item_id> — X handler
  • POST /partner/ribbon/check/<item_type>/<item_id> — Check handler

Both endpoints write audit rows. Both honor audience-eligibility-before-mutation (samai rule 1): user must be allowed to dismiss/check this specific item.

1E. Single-tier escalation engine

A cron job (every_5m bucket, reusing the anomaly engine's cron infrastructure) that scans Tasks:

  • For each task where deadline_at < now() AND completed_at IS NULL AND escalated_at IS NULL:
  • Find the owner's immediate manager (using existing role hierarchy)
  • Set escalated_to_user_id to that manager's User.id
  • Set escalated_at to now()
  • Write TaskAuditLog entry
  • Trigger SMS to the manager if Block 1I (SMS) is wired
  • The task now appears on the manager's ribbon
  • For each task where escalated_at + 24h < now() AND completed_at IS NULL:
  • Escalate one more tier up (to that manager's manager)
  • In Sam's case: Brittany missing follow-up escalates to partner tier

Auto-expiry rule for time-bound signals:

Same cron scans SalesInsight rows (weather, events) and expires them at end of their natural validity window (end of day for weather, end of event date for events). Auto-clear pattern mirrors the existing anomaly engine's _auto_clear from app/services/anomaly_engine.py.

3B. Action registry

app/services/agent_actions.py defines the set of operations the agent can perform. Each action is a registered function with:

  • name — canonical identifier
  • description — what it does (Claude reads this to decide when to invoke)
  • parameters — typed inputs the action needs
  • required_roles — which roles can invoke it
  • is_reversible — can it be undone, and how
  • confirm_required — must ck confirm with the user before executing
  • escalate_uncertain — if ck has even slight uncertainty about whether this is the right action or whether the user has authorization, escalate to Sam or Masood via Telegram BEFORE executing
  • handler — the actual function
  • audit_template — string template for the AgentActionLog entry

Initial action set (Sam-confirmed v1):

  1. toast.resync_employees — re-pull employees from Toast for a specific store. Roles: gm+. Confirm: yes (hits Toast API). Reversible: N/A (idempotent).
  2. drivers.recompute_scores — force a driver score recompute outside the nightly cron. Roles: partner, corporate, gm. Confirm: yes (could affect tier assignments). Reversible: yes (run again).

3C. Intent router via Claude

When a user sends a message, the system:

  1. Captures the message + the user's context (role, store, current page)
  2. Sends to Claude with the action registry as available tools (using Anthropic's tool-use API)
  3. Claude either:
  • Asks a clarifying question if intent is ambiguous
  • Invokes one or more actions
  • Answers conversationally without invoking actions (for Q&A use cases)
  1. System executes any invoked actions, returns results to Claude
  2. Claude composes a final response with the result

Routing logic:

  • Simple intent + simple lookup → Haiku 4.5 (fast, cheap)
  • Multi-step reasoning + multiple action chaining → Opus 4.7
  • Cost cap: $0.50/conversation, auto-escalate to a "this conversation is getting expensive, want to continue?" prompt

System prompt for the agent includes:

  • Who you are (ck, embedded agent inside the Cenas Kitchen app)
  • The user's role and store scope
  • The action registry (auto-injected via tool-use)
  • The system principles (archive-not-delete, audit-everything, no-pay-data-leakage, etc.)
  • The user's last 5 messages of conversational context

3E. Per-role permission gating on actions

Each action declares its required roles in its registration. The router refuses to invoke an action if the user's role isn't in the allowed set, AND tells Claude why so the response is informative ("I can't do that for you — that action requires GM or above; please ask Anna").

samai builds the role-vs-action matrix as a checklist surface (analogous to her existing permission_system spec): each action × each role = explicit allow/deny.

3F. Failure handling + reversibility

For actions that can fail (Toast API down, database write fails, integration timeout):

  • Action returns structured error
  • Claude composes user-facing explanation
  • Failure logged to AgentActionLog with action_result.status=failed

For actions that can be reversed (task reassignment, status change):

  • is_reversible=True flag in registration
  • User can ask "undo that" and the agent invokes the inverse action
  • AgentActionLog tracks the reversal

For actions that cannot be reversed (e.g., sending an SMS, posting a manager log entry):

  • is_reversible=False
  • Claude confirms before executing ("I'm about to send Andres an SMS reminder about the SPECS order — confirm?")
  • The confirmation pattern is decided per-action in the registration

  • Reversible where possible
