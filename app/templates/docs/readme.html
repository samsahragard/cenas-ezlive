What is this site?

app.cenaskitchen.com is the all-in-one operations dashboard for Cenas Kitchen, a two-location Tex-Mex restaurant in Houston (Tomball + Copperfield, branded "DOS MAS" and "UNO MAS" respectively). It consolidates catering orders, produce ordering, scheduling, labor cost, server performance, and third-party sales data into one navigable site — with per-store scoping and an owner-only Partner area for sensitive labor and financial data.

Who uses it?

  • Cenas Kitchen managers — daily catering ops, produce orders, schedule lookups.
  • Sam (owner) — full visibility, including management labor cost, the Partner area, and the developer chat.
  • Drivers — sign in to the Driver Portal to see assignments and (future) GPS tracking.
  • Sam + Masood (owners) — Partner area only, for management labor breakdown and the Developer Chat with this Claude / future AI agents.

What does each section do?

See the Features doc for a per-page breakdown. Quick overview:

  • Today → Dashboard: KPI strip + today's deliveries + needs-attention items.
  • Operations → Vendors → Produce → Orders: weekly Alvarado vs J. Luna price tracking + per-item history charts.
  • Operations → Ezcater: catering order processor, review queue, driver tracking, driver portal.
  • Operations → Schedule: BOH / FOH / All Roster + Weekly schedule view from Sling.
  • Insights → Performance: per-server tip percentage + service timing from Toast.
  • Insights → Sales: third-party sales by channel (Toast in-store, Online, DoorDash, etc.).
  • Insights → Labor: BOH / FOH labor cost as % of net sales.
  • Developer → Chat (Partner only): persistent thread for Sam + AI agents.
  • Developer → App (Partner only): these docs.

How is data flowing in?

See the Data Sources doc. In summary:

  • Toast POS — orders, employees, jobs, time entries, scheduled shifts. Pulled on demand by report routes; cached on disk.
  • Sling — schedules + roster (both locations cleanly; Toast's API only mirrors one location).
  • EzCater Partner API — incoming order webhooks → auto-assign courier → ingest into the kitchen DB → Telegram notify.
  • Produce vendors — Alvarado / J. Luna email weekly price sheets to orders@cenaskitchen.com; an IMAP poller parses them and writes to the DB.

  2. This Claude (or a successor) makes the code change, smoke-tests locally.
  3. Sam approves; commit + push to main.
  4. Render auto-builds + deploys (~3-5 min). New code goes live on the same URL.
