Toast menu depth diagnosis — Uber Eats sync rejection

Sam forwarded a 16-GUID list from a Uber Eats Global Technical Support email saying their integration was failing menu sync with 400 bad_request: the menu has exceeded the maximum allowed levels 15. Initial assumption was that the GUIDs identified modifier items in Toast that needed renaming or removal — that turned out to be wrong.

Diagnosis

  • Pulled both restaurants' full menu payloads via ToastClient._http_get(/menus/v2/menus, restaurant_guid) — 13 MB Copperfield, similar Tomball. None of the 16 GUIDs matched any of the ~17,000 GUIDs in the Toast catalog.
  • Confirmed the GUIDs were UUID-v5 (deterministic SHA-1 hash). Tried a brute-force reverse: hashed every Toast GUID + name with the four standard namespaces (DNS, URL, OID, X500) and 11 prefix conventions Uber-style integrations typically use. ~1.6 M hash candidates. Zero matches — Uber's hashing recipe is internal.
  • Sam then forwarded the full Uber payload that had been pushed (menu_payload_Cenas Kitchen.txt, 227 K lines, statusCode 200, Date Mon 02:22 UTC). Its max nesting depth was 7, so this was the last successful publish — the rejection was on a separate incremental update Toast was retrying.
  • Walked Toast's own modifierGroupReferences + modifierOptionReferences tables (the published menu stores groups and options keyed by numeric refs, dereferenced from menu items via numeric IDs) and found 262,950 paths >=13 deep across the menu, with the deepest hitting 19 levels.
  • Universal cascade pattern at the bottom of every offending chain: Add A Tostada → *^Add Tostada (no meat) → 371No Mixed Cheese On Tostada → 38b1Add Jalapenos → Choose your option → Pickled / Fresh / Pickled+Fresh Jalapenos. Six unique modifier groups at L16+ caused all 76 over-cap items.

  • 11 items at depth 17 in the Traditional menu (depth-19 reachable through alternate sub-paths but the canonical chain caps at 17): Plato Guzman, Plato Salinas, Plato Vaquero, Plato Suarez, Combo Special, Laredo Combo, Pollo Campero, Plato Catrina, Pancho Villa (tuesday special), Veggie Suarez, VEGGIE LOS CABOS.
  • ~50 items at depth 17 across the enchilada family, plus duplicates in DD-COMBINATION-, Apps & Enchiladas, and other parallel menus.

The actual leak path (Sam's pushback was right)

Sam pushed back on my "edit the nested modGroup" advice with: "the parent menu is not visible to partners, this shouldn't matter." He was correct. Verified by enumerating every item that references the bad ModGroups:

  • The 8 top-level "Would you like to remove or add anything to your <item>?" ModGroups (Plato Guzman, Plato Salinas, Plato Vaquero, Plato Suarez, Laredo Combo, Pollo Campero, Pancho Villa, VEGGIE LOS CABOS) are each referenced by exactly ONE item, in the Traditional menu. Both menu and item visibility = ['POS', 'KIOSK']. Those are not leaking.
  • The two deep ModGroups (a3432882-... and f1b1a250-...) are also referenced by Steam Tampiquena in the Grill and favorites🔥 menu. The menu itself is ['POS', 'KIOSK'] — but Steam Tampiquena's item-level visibility override is ['ORDERING_PARTNERS', 'TOAST_ONLINE_ORDERING', 'POS', 'KIOSK']. That single override is the entire leak path: Toast respects the more-permissive setting, pushes Steam Tampiquena to Uber, Uber walks its modGroups, finds the deep cascade, rejects.

Recommended one-click fix: in Toast Web → menu Grill and favorites🔥 → item Steam Tampiquena → uncheck the partner visibility on the item. Steam Tampiquena will still serve Uber via its DD-CENAS FAVORITES- copy (which is at depth 15 and clean). Sam was reviewing this at session close.

ezCater order + driver lookup mechanics

Sam asked who took yesterday's order 7TH-22P and (after a pivot) wanted the same lookup for tomorrow's 1H57T9. Working from scripts/ezcater_api_ingest.py's gql_pull(delivery_id, token) helper.

The orderNumber-without-dash trick

  • The Partner GraphQL endpoint https://api.ezcater.com/graphql accepts order(id: ID!). Initial attempts with the user-facing "7TH-22P" returned 403: You are not authorized to access this data — the 403 is misleading; the actual problem is ezCater stores the order number internally without the dash.
  • Switching to order(id: "7TH22P") (dash removed) returned the full record: uuid, orderNumber, deliveryId, caterer, event (handoff time + address + contact), customer.
  • Same trick worked for tomorrow's order: order(id: "1H57T9") returned uuid 282a5490-9b09-47a2-a48a-c5545815a014, deliveryId 0f639a42-45f0-46a9-b775-b7d7142c2ea6, MARKETPLACE source, lifecycle.orderIsCurrently = "accepted".
  • Generalization: any user-facing ezCater order code can be looked up via the existing gql_pull helper by passing orderNumber.replace("-", "") as the id. Unlocks the entire Order/Event/Customer/Caterer subtree for any order Sam knows by name, without needing the deliveryId in advance.

Why driver name is NOT accessible via Partner API

Introspected the full Partner GraphQL schema:

  • Order type has 12 fields total: caterer, catererCart, deliveryId, event, isTaxExempt, lifecycle, orderCustomer, orderNumber, orderSourceType, taxableAddress, totals, uuid. No driver, courier, dispatch, delivery, fulfillment, or assignment field.
  • Delivery type exists but has only one field: id: ID!. Stub.
  • OrderLifecycle exposes a single string field: orderIsCurrently (returns "accepted" / "completed" / etc.).
  • No Query.* field name contains "deliv", "courier", "track", "driver", or "dispatch".
  • Schema does have CourierAssignPayload / CourierUnassignPayload types — but only as mutation outputs, not queryable. ezCater accepts courier assignment from the caterer side via these mutations (which the existing ezcater_webhook.py uses for Sam-or-Masood auto-assign), but doesn't expose the inverse query.

Where driver info DOES live

  • The public delivery tracker at https://delivery-management.ezcater.com/delivery_tracking/v1/delivery/<tracking_uuid> returns drivers[].currentLocation + currentStatus.key during the active delivery window. This is what app/services/ezcater_live_tracker.py:poll_one() already polls into Order.ezcater_driver_lat/lng.
  • The tracking UUID becomes useful only after the courier hits "Start" in their ezCater driver app (per ck's existing comment in ezcater_live_tracker.py). Before Start, the tracker returns {"status": "expired"}. After delivery completes, same.
  • For yesterday's 7TH-22P the tracker is "expired" because the delivery has long completed. For tomorrow's 1H57T9 the tracker is "expired" because the courier hasn't started the trip yet.

Practical implication

Sam's "who took this delivery?" question can only be answered for orders whose delivery is currently in flight. For Marketplace orders specifically (3rd-party courier dispatched by ezCater), the assignment lands 30-60 minutes before pickup; until then nobody knows. For Direct-source orders on Cenas's own drivers, the data already lives in the local Order.assigned_driver column and can be looked up directly.
