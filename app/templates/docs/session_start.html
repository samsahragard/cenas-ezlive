What this page is

Start-of-session routine for every Claude on the project (aick, ck, samai) and a quick reference for Sam on what to expect when he kicks off a new session. Mirror of Session Closeout — closeout writes the handoff log + archives the transcript; session-start reads it back. Each Claude runs these six steps before engaging on new work so context (yesterday's handoffs, active protocols, open threads) is loaded before Sam's first directive lands.

1) Sam pastes a bootstrap prompt

Sam restarts each Claude with a tailored bootstrap message naming the agent, the working directory, the three handoff docs to read, the chat-tail recipe, and the multi-Claude operating model (push-orchestrator, Masood gate, Render-deploy-noise tolerance). The gist by role:

  • aick — Windows desktop on AiCk, working dir C:\Users\sam\Desktop\cenas-kitchen-claude, push-orchestrator for ck + samai (only Claude with wincredman GitHub creds).
  • ck — Mini PC, SSH'd via Tailscale alias aick, commits locally and pings aick in chat with the SHA to push.
  • samai — Sam's laptop, SSH'd to AiCk, runs the chat tail on AiCk via the laptop session; same push-on-behalf as ck.

2) Read the three prior-session handoffs

At minimum:

/partner/developer/app/aick-session-<most-recent-date>
/partner/developer/app/ck-session-<most-recent-date>
/partner/developer/app/samai-session-<most-recent-date>

Each has an "Open threads for the next <name>" section — surface those after step 5 (chat post) so Sam knows what's queued. Read the other two Claudes' as well, not just your own — cross-stream context (who shipped what, who's waiting on a decision, who flagged a bug for another agent) matters.

If you're cold (first time on the project), also read README + Architecture.

3) Arm the chat tail under Monitor

Canonical recipe (from cenas_chat_tail.md memory on AiCk):

Monitor(
  command='cd "/c/Users/sam/Desktop/cenas-kitchen-claude" && \
           ./.venv/Scripts/python.exe scripts/chat_tail.py --interval 6 \
           2>&1 | grep -E --line-buffered "^\[|ERROR"',
  description="new Developer Chat messages from app.cenaskitchen.com",
  persistent=true,
  timeout_ms=3600000,
)

Why Monitor not background Bash: background Bash only fires on process completion. Monitor emits a notification per stdout line, so new chat messages wake the Claude reactively. The grep filter scopes notifications to chat lines ([time] author: body) plus ERROR so a tail crash doesn't go unnoticed.

ck/samai SCP scripts/chat_tail.py off AiCk before arming — it's the canonical copy and holds the UTF-8 stdout fix for Windows cp1252 (which would otherwise crash on the → char).

Posting from the same tail script: python scripts/chat_tail.py --author <name> --post "...". For multi-line bodies use the post_from_file.py helper (PowerShell argparse drops newlines + reinterprets leading dashes as flags).

4) Confirm active protocols are loaded

All five are memory-backed on each Claude so they survive restart. A quick mental check that each is in context before engaging:

  • "confirm" trigger — Sam writes the bare word "confirm" → reply with the literal I confirm. (no preamble, no trailing question).
  • "QS:" prefix — Sam asks a QS question → aick → ck → samai answer in that order. Later agents must differ on overlap (devil's advocate / different angle).
  • Masood approval gate — Masood's questions auto-answer; any change request escalates to Sam first ("Sam, masood asked for X — OK to proceed?").
  • Address-by-name silence — only reply in chat when Sam (or Masood, on questions) addresses you by name, unless you have something material to flag.
  • Speed > tokens — for building/diagnostic work, parallelize tool calls and read whole files; don't artificially compress. Routine chatter still follows the address-by-name silence rule.

5) Post "online" to chat with a real question

<name> online — handoffs (<dates>) read, repo clean at <sha>,
chat tail armed under Monitor. Push-orchestrator standing for ck/samai.
What's first?

End with a real question, not "standing by" (per cenas_explicit_questions.md — "standing by" reads as resolved, not a request). Sam scans for the three online posts as the go-signal that all agents are loaded.

6) Stand by for Sam's first directive

Don't pre-emptively start on yesterday's open threads — Sam may have re-prioritized overnight. Wait until he names a pickup by your handle. Render deploy noise (~30–60s of 502s + connection-closed errors after every push) is benign swap traffic; don't over-investigate. If 502s persist > 2 min, check the Render API: 

curl -s -H "Authorization: Bearer $(cat ~/.openclaw/.secrets/render_api_key.txt)" \
  "https://api.render.com/v1/services/srv-d7ue6ivlk1mc73aaka20/deploys?limit=2"

Checklist for Sam (before sending the first directive)

  • ☐ All three Claudes posted an "online" message with today's date
  • ☐ Each ended the post with a real question (not "standing by")
  • ☐ The chat tail is reactive — your next message wakes them within ~6s
  • ☐ git log on AiCk shows yesterday's three handoff commits at HEAD
  • ☐ Render shows the latest deploy as live (no stale 502s)
