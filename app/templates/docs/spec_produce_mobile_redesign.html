Spec — Produce Order Guide mobile production implementation

Spec authored by dck (2026-05-18) per Sam #2660 (initial directive, cena relayed), Sam #2637 (photo direction), Sam #2663 (photo source confirm), cena #2638 (style spec), cena #2666 (path lock), cena #2670 (full-catalog discipline reinforcement), and Sam #2677 (v1.2 batch approval). Mockup reference: produce_mobile_redesign.html v1.2 (live at 884f737). Implementation by ck; samai canonical-three-gate on ship.

1 · Scope

Production implementation against the v1.2 mockup, targeting the live /<store>/produce route (per-store produce order guide). In-place redesign — NOT a two-commit + behavior-parity cutover (that pattern applies to delivery-channel switches like ribbon→notifications, not in-place responsive redesigns of an existing live page).

1.1 · What changes

  • Below 640px: table replaced with vertical card list. Each card has product photo (right column, 96×96) + text content (left column: name + price up top, vendor badge + size + date meta middle, qty stepper + line subtotal bottom).
  • Above 640px: existing 6-col table preserved BUT overflow fixed: overflow-x: auto + sticky ITEM column + right-edge gradient fade + QTY column min-width 120px.
  • Sticky bottom bar (narrow widths, gated on cart-has-items): compact vendor chips + Place Order. Reuses existing .sticky-cart DOM — just re-pinned bottom on narrow + visibility gate.
  • Stepper component: 44×44pt tap targets (a11y minimum WCAG 2.5.5), wraps existing .qty-input. refreshCart() JS unchanged.
  • Per-row line subtotal: NEW visible per-card readout of $price × $qty. Live-updates on stepper change.
  • Product photos: NEW right-column 96×96 thumbnail per card. Sourced from Sam's reference catalogs (chat attachments 105/106). Match by photo content not by source label. See §4 below.

3 · Template strategy — single source, CSS-toggled visual modes

Recommended pattern: render BOTH the card list AND the table in the template; CSS toggles visibility per breakpoint via @media (max-width: 640px). Both consume the same {% for it in col_items %} data loop — no Python branching, no risk of "cards show qty 3, table shows qty 0" desync.

Alternative (JS-driven swap on resize) is more fragile and not recommended. The existing 2-col .item-grid split (left_items / right_items) only matters above ~900px; below 640px the card list is single-column showing all items in a flat scroll.

Existing refreshCart() JS wires unchanged — stepper +/− buttons fire the same input event listeners on .qty-input. New code: stepper button click handlers that increment/decrement the input value, then trigger the input event so refreshCart() recomputes.

4 · Photo handoff — full-catalog pass

v1.2 shipped 4 sample photos (avocado / tomato-roma / onion-yellow / cilantro) under app/static/img/produce/<slug>.jpg per cena #2666 path lock. The remaining ~110 catalog items need ck to complete the pass using dck's slicer.

4.1 · Slicer tool

scripts/produce_image_slicer.py (884f737). Four functions:

  • slice_catalog(image_path, out_dir) — splits a catalog grid into per-cell PNGs. Returns list of dicts {row, col, x0, y0, x1, y1, path}. Handles non-uniform grid (top 3 rows = 7-col, bottom 4 rows = 9-col).
  • extract_photo_square(cell_path, out_path) — skips top 25% label band, centre-square-crops the photo, resamples to 240×240 JPG (2× retina for 96×96 display).
  • batch_extract({slug: (image_key, row, col)}, cells_by_image, out_dir) — convenience for slug-mapped extraction.
  • Two underlying detect primitives (detect_row_cells, detect_col_cells_in_strip) if needed separately.

4.2 · Source catalog discipline — MATCH BY PHOTO CONTENT NOT BY LABEL

Critical (cena #2670 reinforcement): The source vendor catalog has label/photo mismatches in some middle-row cells. Examples already identified:

  • ref105 row 3 c1: source label says "Roma Tomatoes" but the actual photo is CILANTRO. Slug assigned to cilantro per Sam #2651 "find the correct fruit for the correct item."
  • ref105 row 3 c3: source label says "Cilantro (Bulk)" but the actual photo is TOMATILLOS.
  • ref106 row 4 c1: source label says "Oranges (88ct)" but the photo is TURMERIC.
Assume more mismatches exist in middle rows. Visually verify every cell before slug-assigning. If ambiguous on a cell, flag it back to dck rather than guess.

4.3 · Pass workflow

  1. Pull ProduceItem list from DB (canonical_name + slug column). Slug derivation: canonical_name.lower().replace(' ', '-').replace('(', '').replace(')', '').replace('.', '') or equivalent — confirm existing slug convention in model.
  2. Run slice_catalog on both ref images (105 + 106) into a working directory.
  3. For each ProduceItem, visually scan the cell grid to find the matching photo (by content, not by label). Record {slug → (image_key, row, col)} mapping.
  4. Run batch_extract with the mapping → writes per-slug JPGs to app/static/img/produce/.
  5. Update order_guide.html template <img> tag to use the photo_url helper.
  6. Items in DB with no matching catalog photo: leave the .pcard-photo-placeholder div as fallback (CSS handles graceful degradation). Future catalog updates from Sam fill those in.

4.4 · Img markup pattern (locked per cena #2664 swap-time discipline)

<img class="pcard-photo"
     loading="lazy"
     src="{{ url_for('static', filename='img/produce/' + it.slug + '.jpg') }}"
     alt="{{ it.canonical_name }}">

loading="lazy" — mandatory, per cena #2664 (30+ produce items × 96×96 on mobile data shouldn't all eager-load). alt="{{ canonical_name }}" — NOT a generic "produce photo" label; screen-reader parity and a11y. Cheap to do right at swap time, expensive to retrofit.
