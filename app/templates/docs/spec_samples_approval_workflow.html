Spec — Samples page approval workflow

Spec authored by dck (2026-05-17) per cena #2547 Item 2. UI lives on the Samples page per-card (not embedded in static mockup HTML — see §1.1 for rationale). Implementation by ck; samai canonical-three-gate on ship; Playwright tests appended to PLAYWRIGHT_BACKLOG.md.

1 · Purpose

Tighten the approval loop on dck mockups. Instead of Sam relaying feedback through dev chat (lossy, async, fragments across messages), he marks each mockup directly on the Samples page — checkmark to approve, X to reject, plus a notes box + image-attachment slot for correction signals. State persists server-side and is visible to the team.

1.1 · Where the approval UI lives

Per-card footer on the Samples page, NOT embedded in the static mockup HTML itself.

  • Static mockups live in app/static/mockups/ as bare HTML files; they have no Flask request context, no CSRF token, no auth state. Embedding the form in them would mean each mockup ships its own JS POST + duplicates the form per mockup — fragile, hard to keep consistent.
  • The Samples page (app/templates/developer_samples.html) is server-rendered with full session context. Adding the approval form once at the card level handles every current and future mockup with zero per-mockup overhead.
  • Sam's experience: he clicks the "Open mockup" button to view the mockup in a new tab, then returns to the Samples page and marks his decision on that card. The mockup itself stays clean (pure design demonstration).

If Sam wants the approval UI literally inside each mockup HTML (e.g., for cases where he wants to mark while looking at the design), that's a v2 consideration — flag as future scope in §13.

2 · Data model

Two new tables (one parent, one child). Cleanest at this size.

2.1 · sample_approvals

columntypenotes
idint PK auto
sample_slugvarchar(64) UNIQUE NOT NULLmatches DEV_SAMPLES[i]["slug"] in app/web/dev_samples.py
statusvarchar(16) NOT NULL DEFAULT 'pending''pending' | 'approved' | 'rejected'

One row per sample_slug — latest state only. If Sam changes approval (approve → reject or vice versa), the row updates in place. No history table for v1; if audit becomes needed, add sample_approval_history later.

2.3 · Migration

New alembic migration: migrations/versions/<next>_sample_approvals.py. Two CREATE TABLE statements + indexes on sample_slug + sample_approval_id.

3 · Routes

methodpathpurpose
POST/partner/developer/samples/<slug>/approveSet status='approved', optionally update notes + attachments. Multipart-form, accepts notes + files[].
POST/partner/developer/samples/<slug>/rejectSet status='rejected', same payload shape.
POST/partner/developer/samples/<slug>/notesUpdate notes only (no status change). Same payload, no status update.
POST/partner/developer/samples/<slug>/attachments/<att_id>/deleteDelete a single attachment.

Auth + permissions:

  • Write surfaces (POST routes) gated to Sam only — user_id == SAM_CHAT_USER_ID at the route entry. Masood per Charter §7A may also be granted, ck's discretion + consistency with other Sam-only endpoints. Recommend Sam-only-by-default + Masood added explicitly if Cena directs.
  • Read (the rendered Samples page with approval state visible) gated to developer.view_app_docs permission — same gate as the Samples page itself. Everyone partner-tier sees the approval status + notes + attachments.

4 · Backend integration

In app/web/developer_chat.py's Samples-page view function, join the approval state into the render context: 

def samples_page():
    # ... existing context setup ...
    from app.models import SampleApproval, SampleApprovalAttachment
    db = next(get_db())
    try:
        approvals_by_slug = {
            a.sample_slug: a for a in db.query(SampleApproval).all()
        }
        # Eager-load attachments to avoid N+1
        atts_by_approval_id = {}
        for a in db.query(SampleApprovalAttachment).all():
            atts_by_approval_id.setdefault(a.sample_approval_id, []).append(a)

        # Enrich each sample with its approval state
        enriched = []
        for s in DEV_SAMPLES:
            ap = approvals_by_slug.get(s["slug"])
            enriched.append({
                **s,
                "approval_status": ap.status if ap else "pending",
                "approval_notes": ap.notes if ap else None,
                "approval_attachments": atts_by_approval_id.get(ap.id, []) if ap else [],
                "approval_marked_at": ap.updated_at if ap else None,
            })

        return render_template(
            "developer_samples.html",
            samples=enriched,
            is_sam=(g.current_user and g.current_user.id == SAM_CHAT_USER_ID),
            # ... existing kwargs ...
        )
    finally:
        db.close()

5 · UI design (per-card approval footer)

Below the existing .dvs-card-foot (Open mockup button + secondary link), add a new .dvs-approval section with a top border separator.

5.1 · Status badge (always visible)

Single inline pill showing current state. Color-keyed:

  • Pending: bg rgba(236, 227, 208, 0.08), color var(--ck-text-soft), label "PENDING APPROVAL"
  • Approved: bg rgba(156, 203, 118, 0.14), color var(--ck-status-open) (#9CCB76), label "APPROVED" + checkmark icon ti ti-check
  • Rejected: bg rgba(226, 100, 92, 0.18), color var(--ck-status-warn) (#E2645C), label "REJECTED" + X icon ti ti-x

Status pill: 10px / 700 / uppercase / letter-spacing 0.06em / padding 3px 10px / radius 4px. Matches the existing .dvs-tag visual family.

Two large action buttons + notes box + attachment slot. Hidden entirely for non-Sam viewers — they see status + notes + attachments read-only.

5.3 · Read-only display for non-Sam viewers

When is_sam is False, render only:

  • Status badge (per §5.1)
  • If notes exist: a read-only text block (no textarea), Markdown-rendered, var(--ck-text-muted)
  • If attachments exist: a chip strip with click-to-view-full (opens image in new tab)
  • "Marked by <user> · <updated_at>" footer line (12px, soft) — gives team context

7 · JS behavior

  • Approve button click: POST to /partner/developer/samples/<slug>/approve with the form data (notes + attachments). On success, swap .is-active class to Approve button, update status pill. No page reload.
  • Reject button click: same as Approve but POST to /reject endpoint.
  • Save notes button: POST to /notes endpoint (no status change). On success, brief inline confirmation ("Saved" for 1.5s).
  • Attach zone click: opens the hidden file input.
  • Drag-drop on zone: add .is-drag-over class on dragenter, remove on dragleave/drop, push dropped files into the form's file input.
  • Attachment chip X click: POST to /attachments/<att_id>/delete, on success remove the chip.
  • All POSTs use fetch with FormData, include CSRF token via existing app pattern.

8 · State transitions

All transitions allowed. Sam can flip approve ↔ reject ↔ pending at will. Each transition updates updated_at and bumps marked_by_user_id. v1 keeps only the latest state; no history table.

9 · Mobile responsive

  • At ≤720px: .dvs-approval-row wraps; buttons stack on their own row, save-btn loses margin-left: auto and goes full-width.
  • Notes textarea full-width on all viewports.
  • Attach zone padding shrinks to 10px.

10 · Accessibility

  • Approve/Reject buttons get aria-pressed="true" when in .is-active state.
  • Status pill uses semantic color + icon + text together (not color-only signal).
  • File input keeps its native <input type="file"> for keyboard-accessibility; the drop zone is the click-target visual but the underlying input remains tabbable.
  • Notes textarea: aria-label if no visible label; placeholder alone isn't sufficient.
