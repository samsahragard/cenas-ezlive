Why these choices

  • SQLite over Postgres: single-tenant, low traffic, all reads are sub-50ms. Postgres adds ops overhead with no tangible upside at this scale.
  • Server-rendered over SPA: kitchen managers + drivers are not power users; reliable, no-build, no-cache-invalidation-headache pages are easier to maintain than a React SPA.
  • Vanilla JS over a framework: total client-side JS is <500 lines across the whole site. A framework would be more code than the JS itself.
  • Chart.js from CDN over self-hosted: Chart.js is the only client-side chart we use; loading it from a CDN avoids a build step.
  • In-process token caches over Redis: Render runs one or two gunicorn workers. Token caches are ephemeral by design and a small re-fetch on cold-start is cheap.

Code style

  • Type hints where helpful (especially service layer); no full mypy enforcement
  • Docstrings explain why, not what — the code already tells you what
  • One blueprint per top-level URL space; no monolithic views.py
  • Service layer is pure Python (no Flask imports) so it's easy to unit-test or call from a CLI script
  • Templates extend base_dashboard.html via Jinja's extends tag to avoid duplicating the sidebar/topbar shell
