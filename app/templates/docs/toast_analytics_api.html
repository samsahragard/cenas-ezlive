What this page is

Reference + recommendation on the Toast Analytics API Sam subscribed to on 2026-05-12 for the Copperfield location only. This is a different Toast API from the ws-api.toasttab.com Web/Partner API we already use in app/services/toast_client.py — different credentials, different host (api.toasttab.com via the /era/v1 path), different access-type, different rate limits, different shape of data.

Source docs: Analytics API Access overview, Analytics API overview, rate-limiting, plus the per-endpoint pages quoted inline below.

Authentication flow (inferred from credentials shape)

The credentials Sam was given are in this shape (values redacted): 

{
  "userAccessType": "TOAST_MACHINE_CLIENT",
  "userId":         "ssr-...",
  "clientId":       "...",
  "clientSecret":   "...",
  "apiAccessUrl":   "https://api.toasttab.com/..."
}

Token-acquisition flow follows the standard Toast OAuth client-credentials grant the existing toast_client.py already implements — same body shape (clientId + clientSecret + userAccessType), different host + different access type. Toast's own auth-doc page returns 403 to anonymous WebFetch, but the pattern is the same. Token TTL on the existing API is ~24h; expect similar.

Use the returned bearer in Authorization: Bearer <token> on every /era/v1/... call. Don't reuse the token from toast_client.py — the access type is different and the scope of the analytics token won't cover the standard endpoints (and vice-versa).

Rate limits (per docs, 2026-05-12)

Endpoint familyLimit
POST /era/v1/metrics + POST /era/v1/menu (custom range)10 / hour
POST /era/v1/metrics/{day|week}10 / minute, 60 / hour
POST /era/v1/metrics/{month|year}10 / hour
POST /era/v1/check/{timeRange}5 / minute, 60 / day
POST /era/v1/payout/payments/{timeRange}5 / minute, 60 / day
All GET (retrieval) endpoints5 / second, 30 / minute

The 60-per-day cap on check + payout-payments is the tightest constraint. For Copperfield it means we can't poll those endpoints from a live UI on every page load — they'd burn the daily budget in minutes. Cache the results on disk same way toast_client.py already does for the Web API.

Does it have value vs what we already have?

Short answer: yes — modest value, not a replacement. Specifics:

  • Worth wiring up:
    • POST /era/v1/metrics for aggregated daily / weekly / monthly sales — replaces our local channel-detection + summation logic. Less code, fewer edge cases, and the dashboard donut would only need a single call instead of scanning every order in the date range.
    • POST /era/v1/payout/* — new capability, nothing equivalent in the current code. Lets us build a "deposits this week" tile or a payout-vs-sales reconciliation view. Useful for Sam.
    • GET /era/v1/restaurants-information — handy as a health-check / restaurant-discovery endpoint.
  • Not worth replacing:
    • Server-performance report (per-server tip %) — our current code uses the standard Web API's per-order tip details which the analytics /era/v1/check endpoint also exposes, BUT the 5/min + 60/day limit makes it impractical for a live UI refresh. Keep the Web API version.
    • Labor cost report — our current code already pulls /labor/v1/timeEntries + classifies BOH/FOH locally. The analytics /era/v1/labor endpoint adds groupBy conveniences but only EMPLOYEE or JOB (not both at once), and our existing classifier handles BOH/FOH split. Not enough lift to migrate.
  • Cost vs value: Sam is paying the RMS Pro tier regardless. The analytics-API access is bundled — no marginal API cost. The "higher cost" Sam mentioned is the Pro subscription itself, not a per-call fee. Since Pro is sunk, using the API just recoups some of that spend.

Recommended integration plan (when Sam greenlights)

  1. Add four env vars on Render (Copperfield-only): TOAST_ANALYTICS_CLIENT_ID, TOAST_ANALYTICS_CLIENT_SECRET, TOAST_ANALYTICS_USER_ID, TOAST_ANALYTICS_API_HOST (default https://api.toasttab.com).
  2. New module app/services/toast_analytics_client.py mirroring the shape of toast_client.py: in-process token cache, disk cache for report-request results, two-step async helper (create_report → poll get_report by GUID).
  3. Wire /era/v1/payout into a new /reports/payouts view first — it's pure addition, no existing code to disturb.
  4. Then layer /era/v1/metrics behind a feature flag in third_party_sales_report so we can A/B compare the pre-aggregated numbers against our current per-order summation. Switch fully if numbers reconcile for a week.
  5. Hard cap analytics-API calls in code with a per-day counter to stay inside the 60/day budget on check + payout-payments.

Open items / cautions

  • Copperfield-only for now. If Sam wants Tomball on the same stack, Tomball needs to be upgraded to RMS Pro first.
  • Auth-doc page was not fetchable anonymously (403 from apiAnalyticsAccessOAuth.html + the apiAnalyticsAuthentication.html alias). Confirm the exact auth body shape on first live call before promoting beyond a test endpoint.
  • The async POST→GUID→GET pattern means our existing "fetch sales for today" code can't be a drop-in replacement. Plan for a small background worker (Flask CLI or daemon thread) that fires the POSTs and writes the results to disk cache.
  • Rate-limit headroom is tight on check + payout-payments. Don't put these behind a user-triggered button without caching.
