What is the Toast API and what can it do?

Toast is a restaurant management platform — POS, payments, kitchen ops, payroll, online ordering, loyalty, gift cards, reporting. Its developer portal exposes a suite of REST APIs that let third-party integrations and custom apps read/write Toast restaurant data.

Integration types

  • Standard API access — read-only credentials any developer can self-generate. No partnership required. Good for analytics, dashboards, exporting your own restaurant's data. Limitations: read-only, no sandbox, no webhooks against arbitrary restaurants.
  • Custom integration — direct integration developer; can use the full API surface (read + write + webhooks) within Toast's developer program.
  • Integration partner — sanctioned third-party partner; can be discovered in the Toast Partner Connect marketplace; can request full access scopes including write APIs.
  • Analytics API developer — for data warehousing / BI integrations; uses the dedicated Analytics API for bulk report retrieval.

What the API surface covers (19 documented API categories)

  • Authentication — OAuth 2 client-credentials login; returns a bearer JWT (accessToken) used on every other call.
  • Orders — Create, read, update orders / checks / selections / payments. The main "point of sale" surface.
  • Menus / Menus V2 / Menus V3 — Read the published menu hierarchy (groups, items, modifier groups, options, prices) for a restaurant.
  • Configuration — Read configuration entities: dining options, service charges, sales categories, payout methods, etc.
  • Restaurants — Restaurant location info, hours, etc.
  • Labor — Employees, jobs, shifts, time entries, breaks.
  • Cash Management — Cash drawer entries, deposits, payouts.
  • Credit cards — Card payment data (transaction-level).
  • Stock — Inventory quantities + 86 / availability flags on menu items.
  • Kitchen — Kitchen prep stations, ticket routing.
  • Packaging configuration — Take-out packaging options.
  • Order management configuration — Dining option / fulfillment configuration.
  • Partners — Partner connection management for sanctioned integrations.
  • Restaurant availability service — Whether a restaurant is currently accepting orders.
  • Analytics — Asynchronous bulk-report requests (large-scale analytics / data warehouse loads). This is what aick wired up for Toast Analytics on Tomball + Copperfield.
  • Gift cards integration spec — Specification third-party gift-card processors must implement to plug into Toast.
  • Loyalty integration spec — Specification third-party loyalty programs must implement.
  • Tender integration spec — Specification for custom payment tender integrations.

How to use the APIs (in one paragraph)

  1. Get credentials. Standard API access: self-service in the Toast back-office at toasttab.com/restaurants/admin/integrations/api-access. Custom / Partner: Toast's integrations team issues a clientId + clientSecret after you join the developer program.
  2. Authenticate. POST https://ws-api.toasttab.com/authentication/v1/authentication/login with {clientId, clientSecret, userAccessType: 'TOAST_MACHINE_CLIENT'} → returns an accessToken JWT, valid for the number of seconds in expiresIn.
  3. Call any other API with Authorization: Bearer <accessToken> plus the Toast-Restaurant-External-ID: <restaurantGuid> header — every business-data request is restaurant-scoped.
  4. Listen for webhooks. Toast can push order / payment / employee events to your endpoint. Subscriptions are configured via the dev portal.
  5. Scopes are explicit. Each operation requires specific OAuth scopes (e.g. orders.read, cashManagement:read). Standard API access gets a fixed read-only scope set; partners can request additional scopes.
