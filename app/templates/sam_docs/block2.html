2I — Recipe Page Shipped

What it is

The manager-facing recipe reference. Each store has a recipe catalogue: dish, ingredients, portions, prep notes, equipment, last-modified. Per-store; recipes are scoped to the store the manager is logged in to.

PathPurpose

Permission gate

  • The real permission key is kds.edit_recipes (permissions.py:82/182/226) — held by km, corporate_chef, corporate (and partner via wildcard). Correction: the previously-documented recipes.view / recipes.write keys do NOT exist in ROLE_PERMISSIONS.
  • Resolved (aick 2026-05-29): the /recipes* routes (store_routes.py:3480+ — index/new/create/detail) carry NO route-level @requires_permission. Edit is NOT gated by kds.edit_recipes at the route; it relies on the global site gate (auth required) + the page's "everyone except expo" audience model + per-store store_scope. (kds.edit_recipes exists in ROLE_PERMISSIONS but is not applied as a decorator here — a soft spot if route-level enforcement is desired; left FLAGGED to Sam.)

Data model

  • Recipe table — id, store, name, category, portions, prep_minutes, notes, created_at, updated_at.
  • RecipeIngredient table — recipe_id, ingredient_name, quantity, unit, notes (one row per ingredient).
  • RecipeStep table — recipe_id, order, text (one row per prep step).

Seed

Initial recipe set seeds on first deploy via the seed entry point. Bulk-insert is supported for migrating an external recipe list; per-store seeds preserve the store scope.
