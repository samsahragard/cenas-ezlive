2I — Recipe Page Shipped

What it is

The manager-facing recipe reference. Each store has a recipe catalogue: dish, ingredients, portions, prep notes, equipment, last-modified. Per-store; recipes are scoped to the store the manager is logged in to.

Permission gate

  • Route gate (verified): all /<store>/recipes* routes (store_routes.py:3480/3517/3528/3562/3598/3634) gate on _manager_role_ok() (abort(403) at :3482/3519/3530…) — a role denylist (_MANAGER_ROLES_DENIED, :1214), so every role except driver, corporate-driver, and expo may view AND create/edit recipes. There is no @requires_permission decorator on these routes.
  • The KDS permission key kds.edit_recipes (permissions.py:82/182/226, held by km / corporate_chef / corporate + partner via wildcard) exists but does not gate these routes — it governs the KDS surface. The previously-documented recipes.view / recipes.write keys do NOT exist in ROLE_PERMISSIONS.
  • aick assessment (2026-05-29): CONFIRMED the broad gate is current behavior - recipe create/edit is gated by _manager_role_ok() (the _MANAGER_ROLES_DENIED denylist), so every role except driver / corporate-driver / expo may edit recipes; the 4-role kds.edit_recipes model is NOT enforced at these routes. Whether to TIGHTEN recipe editing to kds.edit_recipes holders is a product/security decision → FLAG → Sam.

Data model

  • Recipe table — id, store, name, category, portions, prep_minutes, notes, created_at, updated_at.
  • RecipeIngredient table — recipe_id, ingredient_name, quantity, unit, notes (one row per ingredient).
  • RecipeStep table — recipe_id, order, text (one row per prep step).

Seed

Initial recipe set seeds on first deploy via the seed entry point. Bulk-insert is supported for migrating an external recipe list; per-store seeds preserve the store scope.

Manager dashboard Shipped
