Employee Performance

How the employee performance system works end to end: where the data lives, what reaches the deployed app, and the safety model that keeps restaurant sales and raw identifiers off every employee-facing surface. Later sections cover the database schema, the ranking engine, the all-employee rollout, the employee UX, the per-employee data-mart, and the recurring refresh.

Overview

The system gives each employee a private, role-aware view of their own work performance — hours, earnings, tips, and where they rank against same-store peers — computed from Toast labor data and surfaced inside the app.

End to end, the flow is: pull per-shift labor data from the Toast LABOR API into a local performance database; compute normalized per-period metrics and three peer-ranking systems on that local box; sanitize the result down to an employee-safe projection; and push only that projection to the deployed app, which serves each signed-in employee a view scoped strictly to their own data.

  • Periods: every figure is computed for four windows — today, week (Saturday-start, matching the app), month (month-to-date), and last30 (trailing 30 days).
  • Ranking: three systems — effective hourly ($/hr), tip percent (a ratio), and a combined percentile blend — plus a sales-clean tips-per-hour board, each computed within per-store, role-split cohorts.
  • Role awareness: tipped and non-tipped (back-of-house) employees see coherent, role-appropriate dashboards; non-tipped surfaces never carry tip metrics at all.
  • Privacy: small cohorts are gated, peer rows carry only whitelisted metrics, and an employee only ever sees data linked to their own confirmed Toast identity.

Data architecture

The system spans two machines with a deliberate split between where data is computed and where it is served.

Source of truth: the local performance database

The authoritative store is a SQLite database on the Mini_IT13 box: C:\Users\sam\cena-perfdb\perf.sqlite (schema version 1, defined in C:\Users\sam\cena-perfdb\schema_v1.sql). It is not on the hosting platform and not in the app repository. All Toast pulls, scoring, and ranking happen here. The compute modules live in two parallel trees: the deployed copies the scheduler runs (C:\Users\sam\cena-perfdb\) and the pilot/extractor copies (C:\Users\sam\_schedv2_wt\perfdb\, e.g. pilot_phase5.py and toast_perf_refresh.py).

The deployed app reads only a sanitized snapshot

The hosted app never queries the local box live and never calls Toast on a per-employee-page request — the local machine may be asleep, and per-request Toast calls historically overloaded the host. Instead, the app serves every employee page from app-side cache tables (perf_period_cache, perf_shift_cache, perf_rank_cache in app/models.py) that hold the last-good sanitized snapshot. The employee-facing routes live in app/web/employee_auth.py and are scoped strictly to the signed-in employee's session. The app + Toast client worktree of record is C:\Users\sam\_schedv2_wt.

The derived per-employee data-mart

A second local artifact, the per-employee data-mart under C:\Users\sam\cena-perfdb\datamart (datamart.sqlite plus one folder per employee under employees\<cena_employee_id>\), mirrors the same sanitized surface as a deterministic per-employee QA artifact. It introduces no new employee-visible endpoint and no new push — the only employee surface remains the existing projection.

Deploy topology

In short: pulls and compute run on Mini_IT13; only a sanitized projection ever reaches the deployed app. The refresh is driven by session-independent Windows Scheduled Tasks on Mini_IT13 (not host-side cron, which is what caused the historical overload), so it runs whether or not anyone is logged in.

Safety model

There are exactly two outflows from the local box, and only these two. Both are re-validated by the server on the receiving side, so the local sanitization is defense-in-depth rather than the sole guarantee.

  1. The sanitized projection push (outbound): POST /cron/perf-push (defined in app/web/driver_system.py) carries the per-period rows, per-shift rows, and the sanitized rank blob. The receiver is token-gated (an X-Cron-Token header matched against CRON_TOKEN, else 403) and re-validates every payload before storing anything: a whole-body sales-wall regex rejects any body containing a sales-ish term (HTTP 422), and a peer-row field whitelist rejects any peer row carrying a field outside the allowed metric set (422, fail-closed).
  2. The read-only profile/schedule export pull (inbound): GET /cron/employee-datamart-export feeds only the data-mart's profile and schedule domains. It is read-only, fail-closed, and field-whitelisted at the source — it excludes passcode hashes, contact PII, Toast IDs, pay, attribution, and manager free-text — with the consuming mappers re-whitelisting on consume.

Everything else — restaurant sales dollars, Toast GUIDs, hourly rate, and raw scoring inputs — stays on the local box and is structurally unable to leave.

The sales-wall

Restaurant sales dollars are sanitized by construction, not merely filtered at runtime. In the local schema, sales live physically isolated in a dedicated internal-only table and are used only as the denominator for the tip-percent ratio; every employee-facing payload is assembled from the sanitized period and shift data only, so sales has no structural path into an employee payload. On top of that, a source-side guard refuses to emit any payload containing a sales term, and the receiver's whole-body sales-wall independently rejects one. Only the tip-percent ratio ever leaves — never the underlying dollars.

Deterministic pipeline

The compute and data-mart build are deterministic Python with no model/LLM in the runtime path. The data-mart orchestrator enforces hard guardrails — a write-jail limiting output to the data-mart directories, a forbidden-domain deny-list, names-only secret loading (secrets are referenced only by env-var name, e.g. CRON_TOKEN and DATAMART_EXPORT_TOKEN, never by value), read-only sources, and an eligible-set gate — and a validator checks the on-disk output against the contract on every run, failing closed on any drift.

Rankings & metrics

How an employee's normalized performance is scored and ranked. All ranking math runs on the CK-local performance database (perf.sqlite); only a sanitized current-rank blob is ever projected to the employee-visible caches. Restaurant sales dollars live exclusively in the internal perf_internal table and never reach any ranking output that an employee can see.

The four ranking systems

Ranks are computed for all four periods (Today / Week / Month / Last 30) by the ranking engine (perfdb/pilot_phase5.py). Three systems form the core scoreboard, plus a sales-clean fourth tile:

SystemFormulaNotes
effective_hourly(base_pay + tips) / hours ($/hr)Total take per hour. base_pay is employee earnings (rate×hours, OT up-weighted 1.5×), not sales.
tip_percenttips / eligible_sales (ratio)The denominator eligible_sales is INTERNAL — only the resulting ratio ever leaves the CK box.
combinedaverage of the effective_hourly percentile and the tip_percent percentileA blend of two percentile ranks, NOT a mix of dollars and percent.
tips_per_hourtips / hours ($/hr)Sales-clean third tile — no sales denominator anywhere in it.

Per-store, role-split cohorts

Employees are ranked only against comparable peers. A cohort key has the form store:<store>|role:<tipped|nontipped>|metric:<...>.

  • effective_hourly is role-split. Tipped employees rank within the same-store TIPPED cohort; non-tipped / back-of-house employees rank within the same-store NON-TIPPED cohort. The two are never mixed on one board, which avoids a misleading cross-role comparison.
  • tip_percent, combined, and tips_per_hour are tipped-cohort only (a non-tipped employee never qualifies for them).
  • Role is a stable property, not a per-window guess. An employee is tipped when the minimum hourly wage across ALL their shifts at a store is at or below TIPPED_WAGE_MAX ($5.00) — e.g. a $2.13 server. It is computed over all shifts, not the period window, so a tipped server who worked little in a short window is not misclassified.

Privacy & fairness gate (MIN_COHORT)

A cohort with fewer than MIN_COHORT = 4 qualifying peers is GATED: rank and percentile are null, no peer rows are emitted, and the surface shows a "cohort too small" state instead. This is both a privacy guard (you cannot back out a peer's figures from a 2-person board) and a fairness guard (ranks are not meaningful below a minimum cohort size). Leaderboards gate independently per metric — a gated tip-percent board cannot ride out on an ungated effective-hourly board.

Qualifiers (separate from the cohort gate): the Today period needs at least TODAY_MIN_HOURS = 2.0 logged hours; Week / Month / Last 30 need at least LONG_MIN_HOURS = 4.0 hours, and the tip metrics additionally require at least one completed tipped shift. Percentiles are mid-rank within the cohort (outlier-robust); ties break by higher value, then more qualifying hours, then fewer review-flagged shifts.

The FLAG-2 rule — back-of-house never see tip metrics

This is a server-side role gate, enforced where the rank output is built, not a UI hide. A non-tipped employee's sanitized rank output does NOT carry tip_percent, combined, or tips_per_hour objects at all — not even a null placeholder. The employee surface reads is_tipped = false and renders a clean "Not eligible" state rather than empty tip holes. A tipped-but-unqualified period simply omits those keys too. Peer rows on any board carry only whitelisted metric values — never a peer's base_pay, tip dollars, sales, Toast GUID, attribution, or review flags.

The own-card SCORE follows the same split: tipped employees are scored on the combined percentile, non-tipped on the effective-hourly percentile (no tips), banded into encouraging labels (Excellent / Strong / Good / Steady / Building). A gated or no-data period yields "not available" — no fabricated number.

Employee-visible surface

What a signed-in employee sees, and how it is served. Every read is scoped strictly to the session's own employee id (zero cross-employee data) and is served ONLY from the sanitized app-side caches — never live Toast and never the CK box live (which may be asleep). A performance panel appears only for an employee with a partner-confirmed Cena↔Toast link; with no link the panel is hidden.

The performance center — reconciling clock-in/out to totals

The performance center is the day-by-day breakdown that lets an employee see exactly how their period totals were built. Each row reconciles a shift's clock-in / clock-out to the hours, base pay, and (for tipped roles) tips that rolled up into the period summary, so the headline numbers on the dashboard are explainable down to the individual shift. A shift flagged as a likely missed-punch / auto clock-out carries a review reason the employee can expand. A genuinely undeclared tip is shown as "not declared" rather than coerced to a misleading $0.00.

Employee UX

The employee pages share the house dark/gold theme — warm gold accents (#E4B340 / #FFD970 / #d4af37), cream text, dark surfaces, and gold hairline borders — consistent with the rest of the employee dashboard.

  • Period pills (Today / Week / Month / Last 30) drive the summary cards — total pay (base + tips), hours, and, for tipped roles, tips and tips-per-hour.
  • Shift list — each shift shows its time range and base pay; tipped roles also show tips, with a review chip on any flagged shift.
  • Leaderboard rows — per-metric boards with the signed-in user's row highlighted as "you"; a gated cohort renders a "too small" state, and a non-tipped role renders "Not eligible" for the tip boards.
  • Metric drilldowns — tapping a card opens the per-metric day-by-day detail, de-bulking the breakdown for that one metric.
  • Performance / Roster / Service tabs — the Performance tab carries the ranking surface above; the Roster view is a sanitized, read-only roster (no peer pay, tips, or sales); the Service tab carries employee-visible service metrics.
  • Role-aware rendering — when is_tipped is false the tip cards, tip shift spans, and tip leaderboards are not rendered at all, matching the server-side FLAG-2 omission so the page is coherent with no empty tip placeholders. Board labels also adapt — a tipped board reads "Effective hourly (with tips)" while a back-of-house board reads "Hourly rate".

Per-employee data-mart

An INTERNAL CK-local mirror/QA artifact of the same sanitized surface employees already see. It introduces no new employee-visible endpoint and no new push — the only employee-facing surface remains the existing /cron/perf-push projection. Owner: CK (Mini_IT13). Deterministic Python only.

What it is

Rooted at C:\Users\sam\cena-perfdb\datamart, the mart has two outputs built from read-only sources by the orchestrator datamart_build.py:

  • datamart.sqlite — a single SQLite file of 9 dm_ tables (DDL in schema.sql), every table keyed by cena_employee_id (the Cena surrogate integer). No Toast GUID and no name is a key anywhere in the mart.
  • 88 per-employee folders under C:\Users\sam\cena-perfdb\employees\<cena_employee_id>\ — one folder per linked employee, each holding a FIXED 8-file template (identical keys/shape for every employee; only the values differ).

The 9 dm_ tables (schema.sql)

Eight are SANITIZED (may feed the projection) and one is INTERNAL (the only sales-bearing table). Per corrected scope (aick #3334), no incident / counseling / manager-attendance / wage table is built — those domains lack an employee-id key, so a name-match on high-sensitivity manager free-text would be unsafe; the orchestrator's _assert_schema_scope refuses to apply a schema that declares a dropped-domain token outside a negation comment.

The G5 projection-isolation wall

internal.json / dm_internal_sales is the ONLY place restaurant-sales dollars live in the mart, and it is structurally unreachable from any employee surface — not merely runtime-filtered:

  • datamart_build._merge_into_projection returns early for dm_internal_sales, so it is never folded into projection.json.
  • _assemble_template never reads dm_internal_sales as an input, so sales has no structural path into any per-employee folder.
  • Only the tip% ratio ever leaves CK, and it leaves via dm_rank's already-sanitized blob — never from internal.json.
  • Every employee-visible write is gated by boundary.assert_sanitized (a whole-body sales-term regex), and the validator additionally asserts the internal-only keys (eligible_sales / sales_basis / shifts_with_sales) never appear in any visible file.

Read-only profile/schedule source

The mart's profile and schedule domains are fed by a single read-only, fail-closed endpoint: GET /cron/employee-datamart-export. It is field-whitelisted at the source — it excludes passcode_hash, phone, email, address, toast_id, pay, attribution, and any manager free-text — and returns only the profile and schedule domains. The consumer (extractors/app_export.py) bulk-fetches once per process using a transient bearer (env var DATAMART_EXPORT_TOKEN); if the token is absent the build degrades to an offline sample and still passes. The mappers re-whitelist on consume (defense in depth). All other domains are sourced from the read-only perf store.

Auto-enroll of new employees

roster_enroll.py is the reconcile-first front half of a build run. The live roster is the UNION of every id-keyed active signal in the read-only perf store (the employee dimension's active rows plus the distinct employee ids present in the period and shift tables — because the dimension table can be empty while the period/shift tables carry the real population). That union is intersected with the export's profile.active when live, then eligible-gated by eligible_set_88.json, then collapsed to one logical employee per id. Any active, eligible employee lacking a folder gets the full 8-file set written (every value at its MISSING state); a folder missing any contract file is completed on the next run. The operation is idempotent on re-run.

Build modes & guardrails

The orchestrator datamart_build.py runs in three modes: --self-test (no network/writes; asserts every guardrail), --dry-run (sample data, throwaway mart + folders, runs the validator), and --run (the local build: reconcile → read-only extractors → mappers → datamart.sqlite + folders → validator). Hard guardrails G1–G9 are enforced in code: write-jail to datamart/ + employees/ only (G1); no driver/catering/GE4-EFR/ezcater domains (G2); no plaintext secrets (G3); employee-visible output limited to the existing sanitized push (G4); dm_internal_sales kept separate and never joined (G5); deterministic Python only, no LLM at runtime (G6); no scheduler / token-persist / prod-push from this orchestrator (G7); read-only sources (G8); and eligible-set gating (G9). boundary.py re-exports the orchestrator's tested guards as a single source of truth, and validator.py encodes the template contract once and checks both the on-disk folders and the live builder output to catch contract↔code drift, failing closed.

The rank step runs on a temporary copy of perf.sqlite so the canonical file is never opened writable (verified byte-identical before and after a run).

Recurring refresh (scheduled tasks)

The CK-local performance database (C:\Users\sam\cena-perfdb\perf.sqlite on Mini_IT13) is refreshed from Toast on a recurring cadence by three Windows Scheduled Tasks — not by a Render cron. This is deliberate: Task Scheduler launches the work on the machine that owns the database, so the deployed app is never asked to call Toast per employee-page request, and the historical ~15-min Render 502 pattern from in-app polling does not recur. Only the nightly task pushes the employee-visible projection; the two intraday tasks refresh CK-local state only.

No delete, so week / month / last30 are preserved. Light Toast pull (~1 window × 2 stores); each store's today window is primed fresh once per run so per-employee reads hit a fresh cache (true ~1-min freshness at roughly one fetch per store, not per employee). \Cena\DataMart-T2-Windows Hourly (top of the hour) Full pull of all four windows (today / week / month / last30) + full rank recompute. CK-local only — no push.
\Cena\DataMart-T3-Nightly Once after close, 03:30 local Full historical re-pull + recompute, then the sanitized production push to the employee-visible receiver. This is the only task that pushes.

Dispatcher — one entrypoint

Every task invokes a single dispatcher with the tier as an argument:

cena-perfdb\run_tier.py --tier today|windows|nightly [--no-push] [--self-test]
  • Absolute paths only; it chdirs to the writable perf-db directory because Task Scheduler launches the action with a non-writable working directory.
  • Each tier does: refresh perf.sqlite from Toast at the right scope → recompute ranks → rebuild the data-mart (runs datamart_build.py --run as a subprocess that inherits the environment).
  • --tier nightly --no-push runs the full pull + recompute + rebuild without the production push (useful for a manual dry refresh).
  • A failed Toast source is logged and the run exits non-zero, so Task Scheduler records a non-zero LastTaskResult; the last-good perf.sqlite and per-employee folders are retained (no partial clobber).

Run-as principal (DPAPI vault access)

The refresh tokens live in the Windows Credential Manager DPAPI-encrypted and keyed to the owning user, so they are readable only by a process running as that user. The scheduled tasks must therefore run as the interactive Mini_IT13 user that owns the credentials. A practical consequence: the tasks fire while that user is logged on; if the box is logged off, DPAPI cannot decrypt and the refresh pauses (see troubleshooting). Loading is non-interactive (no prompt) — the dispatcher pulls the tokens into the process environment at runtime by name; values are never logged or written to disk.

Tokens (by name)

Two tokens, both stored in the Windows Credential Manager (generic credentials), referenced here by name only:

  • CRON_TOKEN — bearer for the sanitized production push (the perf-push receiver).
  • DATAMART_EXPORT_TOKEN — bearer for the read-only profile/schedule export pull that feeds the data-mart.

Read precedence at runtime is: existing environment → Credential Manager → absent. The export pull degrades gracefully — if DATAMART_EXPORT_TOKEN is absent the build falls back to an offline sample and still passes; the push, by contrast, simply does not authenticate without CRON_TOKEN.

Toast LABOR API budget

The perf pull uses the direct Toast LABOR API (/labor/v1/timeEntries), not the tighter Analytics API. Documented limits are 20 requests/second and 10,000 requests / 15 minutes, with no per-call billing for own-restaurant data — generous relative to this workload:

  • The today tier primes the window fresh at roughly 2 calls per run (~1 window × 2 stores). Over a 16-hour operating window that is on the order of ~1,920 calls/day — far under the 10k/15-min ceiling.
  • Designed handling for 429 / 5xx: exponential backoff with jitter (1s, 2s, 4s, 8s, cap ~30s, ~5 retries), then log-and-skip the source for that run rather than crash (the prior good snapshot is retained). Per-run jitter also spreads the minute-aligned today-tier fires.

Logs

Secret-scrubbed run log at cena-perfdb\datamart\run_tier.log (the dispatcher's per-run record); the data-mart build emits datamart.log and the standalone refresh runner emits perf_refresh.log. A secret-scrub log filter strips token-like values from all three.

Operations & troubleshooting

Rollback / disabling the refresh

To stop the recurring refresh entirely, delete the three scheduled tasks:

schtasks /Delete /TN "\Cena\DataMart-T1-TodayDelta" /F
schtasks /Delete /TN "\Cena\DataMart-T2-Windows"    /F
schtasks /Delete /TN "\Cena\DataMart-T3-Nightly"    /F

Deleting the tasks halts all CK-local refresh and the nightly push; the deployed app keeps serving the last-good sanitized snapshot it already holds, so the employee surface stays up (just frozen until the tasks are restored). The dispatcher and the database files are untouched by the delete — re-register the tasks to resume.

Troubleshooting

  1. Stale employee data. Open Task Scheduler → \Cena\ and check each task's LastRunTime / LastTaskResult. A non-zero LastTaskResult means the run exited on a failed source — read cena-perfdb\datamart\run_tier.log for the scrubbed error. The last-good snapshot is retained, so the symptom is "not updating," not "data lost." A manual catch-up run is run_tier.py --tier windows (CK-local) or --tier nightly (also pushes).
  2. Token rotation. After rotating CRON_TOKEN or DATAMART_EXPORT_TOKEN, re-store the new value in the Windows Credential Manager under the same target name (so the next scheduled run picks it up by name), and for CRON_TOKEN trigger a Render redeploy so the receiver validates against the new value. Confirm by watching the next nightly push (or a manual --tier nightly) land without a 403.
  3. Box logged off. The tasks run as the interactive user for DPAPI vault access, so a logged-off (or locked-without-stored-credentials) machine cannot decrypt the tokens and the refresh pauses — intraday freshness stops and the nightly push is skipped until the user is logged back on. Log on, then run a manual --tier windows to catch up CK-local state.
