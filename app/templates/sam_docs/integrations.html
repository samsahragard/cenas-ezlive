EZCater

Catering order intake from EZCater. One ingest path (webhook) + a driver-assignment surface that drives the partnerportal.ezcater.com UI from our side:

  • Webhook — EZCater posts inbound order events to /ezcater/webhook (signature-verified). The webhook handler ingests the order into the catering tables, picks the default in-house driver from the store map (Masood CK #1 for Copperfield, Sam CK #2 for Tomball), and queues a DriverAssignmentJob row that the AiCk gateway picks up to drive the partnerportal Chrome flow (Sam #900, 2026-05-25).
  • Ingest is webhook-only (samai 2026-05-28 correction) — there is NO EZCater IMAP backup in the current code. The only IMAP poller is produce_ingest.py (PRODUCE orders to orders@cenaskitchen.com, gated by PRODUCE_INGEST_ENABLED) — a separate pipeline, NOT EZCater. EZCater orders arrive solely via the signature-verified webhook above.

Driver assignment (Selenium UI flow)

The dropdown on each catering Ez Orders card AND the new-order auto-assign both run through the same gateway path. The path:

  1. Render route (/<store>/catering/assign_driver or the webhook) creates a DriverAssignmentJob row and POSTs a wake to AiCk's cena_gateway.py:/jobs/driver-assign over Tailscale via CENA_PROXY=socks5h://localhost:1055 (httpx, not urllib — urllib bypasses the proxy and never arrives).
  2. The gateway spawns a Selenium thread that loads cookies from ~/.openclaw/.secrets/ezcater_session.json into a dedicated Chrome profile (C:\Users\sam\ezcater_assigner_chrome_profile), navigates to the order page, clicks Unassign driver if needed, clicks Assign in-house driver, types the new driver name, clicks the matching label, submits, and re-reads the DOM to verify.
  3. The gateway POSTs the result back to Render's /catering/assign_driver/result callback (X-Cena-Token gated). On status='completed', Render writes the new driver name into Order.ezcater_driver_name; on '__no_driver__' unhook, it clears the field.
  4. Failover hop (Sam #900, 2026-05-25) — if aick's Selenium fails (auth, popup-stuck, RuntimeError, etc., but NOT verification_mismatch which is an ezCater-side state rejection), the gateway forwards the same job payload to CENA2_GATEWAY_URL=http://100.73.38.82:8766 on Mini_IT13. ck's Playwright gateway runs the same flow + posts the callback with gateway_processed='ck' instead of 'aick'. End-to-end proven in production 2026-05-25 (8 consecutive failovers caught cleanly during a stale-module-cache outage on aick).

NOTE (samai 2026-05-28): per the Architecture audit, the driver-assign PRIMARY is now pwck (Mini_IT13 :9000, /pwck/run); aick Selenium + ck Playwright are now fallback, and the cena2:8766 peer referenced above is currently DOWN. EZCater rules

  • Idempotent upsert — every order is keyed on the EZCater order id; re-ingest is a no-op if the row hasn't changed.
  • Status transitions — the EZCater pipeline statuses map to internal statuses (pending → confirmed → fulfilled → delivered).
  • Driver assignment is UI-mediated — both auto-assign on new orders AND manager dropdown re-assigns go through the partnerportal Chrome flow. The direct courierAssign GraphQL API is only used as a last-resort restoration path (e.g. when ezCater UI silently rejects a same-driver re-pick) — per Sam standing rule, agents do not use the API except for in-house Masood/Sam restoration when the UI path cannot complete.
  • Same-driver re-assigns are ezCater-rejected via UI — re-picking the driver who's already on the order surfaces the red "There was an error assigning the driver" dialog. The Selenium flow detects this and raises _AssignmentMismatch so the dropdown reports failure cleanly to the manager.
  • Cancellation — vendor cancellations come through the webhook and mark the row cancelled; manual cancellations require the catering dashboard surface.

Toast

POS data flows through two distinct Toast surfaces:

Toast API (standard)

  • Sales rollups, menu, order history.
  • OAuth client (server-to-server); access token refresh is managed by the Toast client wrapper.
  • Refreshed by the sales-insights cron every ~30 minutes.

Toast Analytics API

  • Higher-resolution analytics: sales, labor, menu mix.
  • Separate OAuth credentials from the standard API (analytics has its own client_id + client_secret).
  • Pre-aggregated rollups land in the SQLite analytics tables (the app DB is SQLite at /var/data/cenas_kitchen.db, NOT Postgres — see Architecture) for fast dashboard reads.

Toast rules

  • Per-store mapping — Toast's restaurant id maps to the internal store slug. Mapping lives in the toast_store_map table.
  • Time windows — every analytics query carries an explicit start/end; never an unbounded scan.
  • Idempotent writes — rollup writes are keyed on (store, period, metric); a re-run overwrites cleanly.
  • Nested modifier depth ≤ 15 (Uber Eats limit) — Uber Eats rejects items where the menu→category→item→(mg→mo)* chain exceeds 15 levels. Run _count_deep_nested_modifiers.py on AiCk to scan; the Menus v2 API uses REFERENCE-based structure (top-level modifierGroupReferences + modifierOptionReferences dicts) — naive inline walkers will return 0. The 2026-05-25 sweep identified 22 items in 1DD-COMBINATION- / 1DD-ENCHILADAS- with depth 16-17; Sam cut the shared Choose your option.., mg to bring all 22 back under the limit. Re-run the scan after any DD-menu edit.

Anomaly engine

A rules-driven engine that compares incoming Toast-derived metrics against expected baselines and emits anomaly events.

Anomaly service

  • Runs as a cron job at configurable cadence (daily + weekly anomaly briefs).
  • Loads the rule registry on startup; each rule is a typed Python class with evaluate(window, baseline) -> [AnomalyEvent].
  • Emits anomaly events to the anomaly table; downstream the morning-brief composer reads them.

Anomaly rules

The rule registry contains rules covering revenue, labor, menu mix, catering pipeline, vendor status, and outages. Rule classes:

  • Threshold rules — metric outside an absolute band.
  • Trend rules — metric outside its trailing rolling baseline.
  • Composite rules — multiple metrics together (e.g. revenue down + labor up = margin compression).
  • Absence rules — expected event missing (e.g. catering rollup didn't run).

Rules are added by writing the class + registering it in the rule registry. Each rule has a unique key, a severity, and a description that surfaces in the brief.

Morning brief composer

The composer reads the most recent anomaly batch, renders a per-store summary, and emits the daily morning brief. The brief lands in the dashboard's notifications surface and (when enabled) goes out via email to the on-call recipient.

Brief calibration

A calibration runbook covers how to tune the anomaly rules without bouncing the production engine:

  • Dry-run mode — run a rule against historical windows and inspect the output without writing the anomaly table.
  • Per-rule enable/disable — env-driven; flip a noisy rule off without a deploy.
  • Baseline windows — adjust the rolling baseline window for trend rules; defaults to 30 days.

Sentry

Error monitoring for the Flask app. Configured via SENTRY_DSN env var; integration is automatic for Flask + SQLAlchemy + the cron entry points. Critical anomalies in the anomaly engine optionally trigger a Sentry message for visibility.
