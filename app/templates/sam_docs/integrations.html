Driver assignment (Selenium UI flow)

The dropdown on each catering Ez Orders card AND the new-order auto-assign both run through the same gateway path. The path:

  1. Render route (/<store>/catering/assign_driver or the webhook) creates a DriverAssignmentJob row and POSTs a wake to AiCk's cena_gateway.py:/jobs/driver-assign over Tailscale via CENA_PROXY=socks5h://localhost:1055 (httpx, not urllib — urllib bypasses the proxy and never arrives).
  2. The gateway spawns a Selenium thread that loads cookies from C:\Users\sam\cena-secrets\ezcater_session.json into a dedicated Chrome profile (C:\Users\sam\ezcater_assigner_chrome_profile), navigates to the order page, clicks Unassign driver if needed, clicks Assign in-house driver, types the new driver name, clicks the matching label, submits, and re-reads the DOM to verify.
  3. The gateway POSTs the result back to Render's /catering/assign_driver/result callback (X-Cena-Token gated). On status='completed', Render writes the new driver name into Order.ezcater_driver_name; on '__no_driver__' unhook, it clears the field.
  4. Failover hop (Sam #900, 2026-05-25) — if aick's Selenium fails (auth, popup-stuck, RuntimeError, etc., but NOT verification_mismatch which is an ezCater-side state rejection), the gateway forwards the same job payload to CENA2_GATEWAY_URL=http://100.73.38.82:8766 on Mini_IT13. ck's Playwright gateway runs the same flow + posts the callback with gateway_processed='ck' instead of 'aick'. End-to-end proven in production 2026-05-25 (8 consecutive failovers caught cleanly during a stale-module-cache outage on aick).

EZCater rules

  • Idempotent upsert — every order is keyed on the EZCater order id; re-ingest is a no-op if the row hasn't changed.
  • Status transitions — the EZCater pipeline statuses map to internal statuses (pending → confirmed → fulfilled → delivered).
  • Driver assignment is UI-mediated — both auto-assign on new orders AND manager dropdown re-assigns go through the partnerportal Chrome flow. The direct courierAssign GraphQL API is only used as a last-resort restoration path (e.g. when ezCater UI silently rejects a same-driver re-pick) — per Sam standing rule, agents do not use the API except for in-house Masood/Sam restoration when the UI path cannot complete.
  • Same-driver re-assigns are ezCater-rejected via UI — re-picking the driver who's already on the order surfaces the red "There was an error assigning the driver" dialog. The Selenium flow detects this and raises _AssignmentMismatch so the dropdown reports failure cleanly to the manager.
  • Cancellation — vendor cancellations come through the webhook and mark the row cancelled; manual cancellations require the catering dashboard surface.

Toast

POS data flows through two distinct Toast surfaces:

Toast API (standard)

  • Sales rollups, menu, order history.
  • OAuth client (server-to-server); access token refresh is managed by the Toast client wrapper.
  • Refreshed by the sales-insights cron every ~30 minutes.

Toast Analytics API

  • Higher-resolution analytics: sales, labor, menu mix.
  • Separate OAuth credentials from the standard API (analytics has its own client_id + client_secret).
  • Pre-aggregated rollups land in the SQLite analytics tables (the app DB is SQLite at /var/data/cenas_kitchen.db, NOT Postgres — see Architecture) for fast dashboard reads.

Toast rules

  • Per-store mapping — Toast's restaurant id maps to the internal store slug. Mapping lives in the toast_store_map table.
  • Time windows — every analytics query carries an explicit start/end; never an unbounded scan.
  • Idempotent writes — rollup writes are keyed on (store, period, metric); a re-run overwrites cleanly.
  • Nested modifier depth ≤ 15 (Uber Eats limit) — Uber Eats rejects items where the menu→category→item→(mg→mo)* chain exceeds 15 levels. Run _count_deep_nested_modifiers.py on AiCk to scan; the Menus v2 API uses REFERENCE-based structure (top-level modifierGroupReferences + modifierOptionReferences dicts) — naive inline walkers will return 0. The 2026-05-25 sweep identified 22 items in 1DD-COMBINATION- / 1DD-ENCHILADAS- with depth 16-17; Sam cut the shared Choose your option.., mg to bring all 22 back under the limit. Re-run the scan after any DD-menu edit.

Anomaly engine

A rules-driven engine (app/services/anomaly_engine.py) that runs registered rules against the live DB and emits Signal rows for the dashboards + morning brief. Correction (ckai 2026-05-28): the page previously described a class-based evaluate(window, baseline) engine comparing Toast metrics to baselines — that is not the implementation.

Anomaly service

  • Rules are functions, not classes — each is a function decorated with @anomaly_rule(name=, bucket=, severity=, surfaces=, audience_roles=, action_text=, llm_route=) taking (db: Session) and returning list[SignalDraft]. The decorator registers it into REGISTRY (name -> RuleSpec) at import; app/__init__.py imports the rule modules so the registry is populated on boot.
  • Cadence is bucket-driven — each rule declares a bucket: every_5m / every_15m / hourly / daily / weekly / on_write. The entrypoint run_bucket(bucket) is invoked by the cron route /cron/anomaly-eval and runs every rule in that bucket in one transaction.
  • Emits to the Signal table (not an "anomaly table"). _upsert_signal dedupes open rows on (rule_name, subject_id, store_id); reversible rules (REVERSIBLE_RULES) auto-clear via _auto_clear when the condition stops firing. A new alert-severity Signal fires a Telegram alert (_maybe_telegram_alert).

Anomaly rules

~32 rules are registered (so the /partner/anomalies/rules admin + the brief see the full set), grouped by domain in app/services/anomaly_rules.py: orders/delivery, vendor/produce, sales, labor, attendance, server-perf, kitchen, customer, system. Most are stubs returning [] pending their data source (Toast Analytics, KDS, vendor-quotes, Sling, Google Reviews) — the docstring tags each Implemented vs Stub.

  • Live rules (~5): orders.no_driver_30min_before, orders.late_delivery, orders.driver_no_show, orders.ezcater_rejection_rate, orders.high_pickup_lag. system.deploy_failed is coded but no-ops without RENDER_API_KEY.
  • Stubs occupy a REGISTRY slot (so admin + brief treat them as known) and go live by swapping the body for a real query — no engine/cron/admin change needed.

Per-rule attributes: a unique name (key), a default severity (info/warn/alert), target surfaces, audience_roles, an action_text (what surfaces in the brief), and an llm_route (sql/haiku/opus).

Morning brief composer

The composer (app/services/brief_composer.py) builds one brief per role-audience (partner / gm / km / foh_manager / corporate / corporate_chef / prep_manager), not just per-store. compose_brief gathers yesterday's open Signal rows (permission-filtered to the audience) + wins + lookahead + calibration, then composes with Anthropic claude-opus-4-5 (cached system prompt) and falls back to a deterministic template (fallback_used=True) if the LLM is unavailable or returns bad JSON. Each brief persists as a MorningBrief row. The cron route POST /cron/anomaly-brief runs compose_all_briefs once daily and dispatches by email — dry-run unless BRIEF_EMAIL_DISPATCH=1.

Brief calibration

Rules are tuned without a deploy via the RuleOverride table (set at /partner/anomalies/rules), which the engine consults at run start:

  • Severity override — RuleOverride.severity_override (store-scoped wins over global) re-grades a rule's output; downgrading to info is how a noisy rule is effectively quieted. There is no separate enable/disable flag.
  • Threshold edits — RuleOverride.threshold (JSON) holds per-rule threshold params.
  • Calibration surfacing — the brief's partner-only calibration section flags noisy rules (>6 fires/day) and rules silent >14d (gather_calibration).

Correction (ckai 2026-05-28): the prior "dry-run mode / env-driven enable-disable / 30-day trend baseline" bullets were aspirational — tuning is DB-driven via RuleOverride, and no trend/baseline rules are live yet.
