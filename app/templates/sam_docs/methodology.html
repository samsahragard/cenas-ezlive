Methodology

Engineering standards for the Cenas Kitchen codebase. The rules below are non-negotiable gates before a change ships. Last audited 2026-05-28 (samai, docs full-audit pass).

The six standing rules

Every commit that touches user-facing state, a database mutation, or a partner doc must pass each rule that applies. These are not guidelines — they are the gate.

  1. Audience-eligibility-before-mutation — the audience check (who can perform this action) runs BEFORE any database mutation. A 403 leaves zero rows.
  2. Verify-against-prod-not-local — deploy-verify hits production state, not local state. Must check something BUILD-SPECIFIC: a route or asset unique to the new commit. A build-agnostic response (a partner-gate 302) proves the service is up, NOT which commit it is serving.
  3. Deploy-success-verify — every commit gets a real deploy-verify pass after the deploy lands. BLOCKED-PENDING-DEPLOY is a valid state during build-pipeline outages; never claim "deploy-verified" against a stale build.
  4. Doc-registration-checks-both-route-AND-nav — a new documented surface requires the route AND the nav entry. Both surfaces, both updated together.
  5. Hit-new-doc-URL-once-before-trusting-it — render-verify the URL returns 200 with expected content before claiming the surface is live.
  6. Amendments-against-git-show-HEAD — spec doc amendments are authored against the committed tree (git show HEAD:<path>), NEVER a local pre-commit copy. The tree is the source of truth.

Three-gate deploy verification rubric

Rule 3 (deploy-success-verify) is operationalized as three independent gates. A commit is "deploy-verified" only when ALL THREE pass on the same SHA, and the verification is reported with the gate-by-gate breakdown.

  1. Gate 1 — Local pytest: the full suite passes on the working tree. Catches schema mismatches, fixture drift, and obvious regressions before the change ever leaves the dev machine. Run from the repo's .venv: .\.venv\Scripts\python.exe -m pytest -q (or platform equivalent).
  2. Gate 2 — CI: the GitHub Actions tests.yml workflow lands completed / success for the SHA. CI's environment differs from local in subtle ways (no .env, hermetic fixtures); a green CI proves the change still passes in a clean room.

CI is therefore red whenever such a test exists: a stable structural pattern, not flakiness.   3. Gate 3 — Build-specific deploy probe: a probe that confirms the COMMIT is live — not just "the service is up". A partner-gate 302 redirect proves the service is up, NOT which commit it is serving.

Verifying a deploy in practice

  1. Push the commit to origin/main.
  2. Wait for Render's auto-deploy. Typical: 2–3 minutes from push to live.
  3. Confirm the deploy is live — two methods, by key access:
    • Render-API status-poll (authoritative; key-holders) — poll GET https://api.render.com/v1/services/<service-id>/deploys?limit=5, match the deploy whose commit.id == target SHA, wait until status == "live" (terminal set also incl. build_failed/update_failed/canceled/pre_deploy_failed/deactivated). aick runs this as the Render key-holder on Mini_IT12 (script cena/_deploy_watch_<sha>.py); observed this session: bb5f636 live in 216s, 465a7b0 153s, d50574f 153s. (Whether a Render API key is in the canonical agent secret-set is being reconciled on the Pass page — aick has one on Mini_IT12; ckai/samai do not, hence the URL-poll fallback.)
    • URL-poll (key-less fallback; agents without the Render key) — the swap is silent: curl -sI https://app.cenaskitchen.com/login serves the old build (and brief 502s) until the new build goes live. Poll until non-502 and the swap settles. Used by samai (badge deploys this session).
  4. Hit the build-specific probe — a 200/302 alone is NOT enough: the old build serves 200s right up to the swap. Fetch a route whose body carries content unique to the new commit and assert that exact markup. For a gated route, authenticate first (site + partner password) then fetch the page and check the new string; a bare curl -sI 302 to /keypad-login only proves the route exists, not which build serves it.
  5. Three-gate report: explicitly call out Gate 1 / Gate 2 / Gate 3 status when reporting "deploy verified".

Workflow norms

  • Pull-rebase before commit. Stage by name — git add file1 file2 with explicit filenames, never -A. The working tree often has scratch files from prior debugging; -A sweeps them in.
  • Coordinate shared-file edits in advance for high-traffic files (base_dashboard.html, store_routes.py, sidebar.html, etc.). One coordinator at a time avoids overwrites.
  • Render-deploy swap noise — every push triggers ~30–60s of 502s + "connection forcibly closed" errors. Benign swap artifact. If 502s persist past ~2 minutes, check the Render API for the deploy status.
  • Hash protection on user-typed secrets — passwords (keypad PINs, partner gate, manager pins) are stored hashed, never plaintext. Migrations that touch hash columns must remain idempotent.

Build-lane variations (multi-agent lanes)

The three-gate rubric above governs a single agent shipping a change solo. Some build lanes run a different gate model — documented here so methodology reflects reality:

  • Peer cross-review + Sam single sign-off — for multi-agent coordinated efforts (e.g. the docs full-audit pass, the Schedules V2 build), each agent's work is cross-reviewed by a named peer (read the diff + spot-check one claim against cited evidence) instead of samai gating every page. The reviewer posts "cross-review PASS — <unit>" or flags an issue; Sam is pulled in once at the end to review the consolidated FLAGGED list, not per unit.
  • Flag-and-proceed — in a parallel lane, a claim that can't be verified from the owning agent's access is posted as a cross-domain ask to the right peer; the agent keeps working and folds the answer in when it lands. No answer in ~15 min → leave it FLAGGED and move on. A flagged-honest artifact beats a stalled one.

Adding a documented surface

  1. Write the page (template under app/templates/<area>/).
  2. Register the route (in the appropriate blueprint).
  3. Register the nav entry (the relevant sidebar / tab strip).
  4. Gate the surface (audience eligibility per rule 1).
  5. Verify the URL renders 200 with the expected content (rule 5).

Current operator-docs surface: /sam/docs renders its pages as Jinja includes from app/templates/sam_docs/ (see sam_chat.py → sam_docs_page); for a docs page, "register the nav entry" means wiring it into that surface's index/tab strip.
