Operations

Day-to-day runbook for opening a session, closing a session, and the recurring manager / operator procedures.

Session start — opening checklist

When the dev session begins (whether a code change cycle or an ops triage), the first responsibilities are environment health + state awareness.

  1. Repo state: git fetch && git status. Confirm working tree is clean or known-dirty (scratch files only).
  2. Origin/main HEAD: confirm you are at the latest origin/main SHA. Pull-rebase if not.
  3. Render web service: hit /produce/healthz — expect a 200 (the public liveness probe; on the auth exempt-list in app/web/auth.py). NOTE: bare /healthz is NOT exempt and 302s to keypad-login — use /produce/healthz.
  4. Render cron health: glance at the Render dashboard for any failing cron. A red cron usually indicates a Toast / Sling / EZCater API outage or a credential rotation.
  5. External vendors: if a flow is reported broken, confirm vendor side first (Toast Analytics status, EZCater; Sling is being replaced by Schedules V2 — see Phase 2 Roadmap).
  6. Outstanding tasks: review the working punch list before introducing new work.

Session closeout — leaving the system safe

  1. Working-tree audit: git status. Stage only files you authored; never git add -A.
  2. Stage-by-name discipline: git add path/file1 path/file2 — explicit. Scratch files (plan.md, debug dumps, data dirs) stay out.
  3. Commit with a clear subject + the why in the body. Never amend a previous commit; create a new one.
  4. Three-gate verify: per the Methodology tab — Gate 1 pytest, Gate 2 CI, Gate 3 deploy probe. NOTE: Gate 2 (CI / tests.yml) is currently KNOWN-RED on every main push — deploys ship on Gate 1 + Gate 3 only (see Methodology).
  5. Operational notes: leave a short closing note for the next session if anything is mid-flight (which task, which branch, which gate is blocked).

Common procedures

Adding a manager / GM to a store

  1. Create the user via the access-request flow (manager keypad-login admin surface).
  2. Assign role gm + store affiliation.
  3. The manager keypad-login pin is set via the env var rotation procedure.

Rotating a vendor / API credential

  1. Update the secret at its source: the Render env var and/or the on-disk secret file (app/runtime lane C:\Users\sam\cena-secrets\ or Cena gateway lane C:\Users\sam\cena\.secrets\ on AiCk; mirror to Mini_IT13 if it is used there too). Then record the change in samai's canonical credential index on his Desktop. NOTE: 1Password is RETIRED (Sam, 2026-05-28) — it is no longer a credential source; see the Pass tab.
  2. If Render env: trigger a redeploy to pick up the new value.
  3. Hit a cron that consumes the credential to confirm the rotation worked.

Investigating a failing cron

  1. Render dashboard → cron service → recent runs → failed run logs.
  2. Common causes: external-API outage, credential rotation not yet propagated, schema mismatch from a recent migration.
  3. Once root cause is identified, fix and trigger a manual run to confirm.

Responding to a partner-side 502 spike

  1. 30–60s of 502s immediately after a push is normal Render swap; ignore.
  2. Persistent 502s (>2 min) — check the Render web-service deploy status; if "build failed", revert the offending commit.
  3. If the deploy is "live" but 502s continue, check Sentry for the most-recent error and the application logs.

Triaging a stuck driver-assignment job

  1. Look up the job in the DB: SELECT * FROM driver_assignment_jobs WHERE order_id=<X> ORDER BY created_at DESC LIMIT 5.
  2. If status='pending' for >2 min: the AiCk gateway probably isn't running or the Render→AiCk Tailscale hop is broken (check that Render outbound uses CENA_PROXY=socks5h://localhost:1055 with httpx — plain urllib silently bypasses the proxy + never arrives).
  3. If status='failed' with error_message starting verification_mismatch: ezCater rejected on its side (same-driver re-pick or driver doesn't match the order's store). Not a bug — surfaces back to the manager via the dropdown's error chip.
  4. If error_message starts auth_expired: ezCater session cookies dead. Sam logs in to partnerportal.ezcater.com via Cenas Edge; aick re-extracts cookies via DPAPI into C:\Users\sam\cena-secrets\ezcater_session.json (the old ~/.openclaw/.secrets/ lane is archived/gone — see Pass tab).
  5. If gateway_processed='ck': aick selenium failed + ck's Playwright took over via the failover hop. Expected behavior; no action unless ck also reports failure.
  6. Repeat-failure storm with no modal-match log lines on aick: likely the stale-module-cache bug (Chrome import returned None at gateway-startup). Bounce cena_gateway.py on AiCk to force a fresh import.

Releasing a new mobile (Capacitor) build

  1. Edit anything under mobile/ (typically mobile/package.json for a plugin/version bump, or root app/templates/ changes for WebView UX).
  2. Commit + push to main (or run the workflow manually via workflow_dispatch). GitHub Actions .github/workflows/mobile-android.yml runs two jobs.
  3. build job: builds the debug APK, uploads it as a workflow artifact, and (on push to main) publishes it to the android-debug-latest GitHub release so /driver/app.apk has a stable no-login sideload URL for drivers.
  4. release-aab job (push to main or manual dispatch, only when the keystore secrets are configured): builds the signed AAB, signing with the GitHub Secrets keystore (ANDROID_UPLOAD_KEYSTORE_B64 + ANDROID_KEYSTORE_PASSWORD / ANDROID_KEY_ALIAS / ANDROID_KEY_PASSWORD), targets SDK 35, bumps versionCode to the Actions run number, and uploads the signed AAB as a workflow artifact (60-day retention). There is NO Android Publisher API step.
  5. Play Console upload is currently MANUAL: download the signed AAB artifact from the Actions run and upload it to the Play Console internal-testing track. (scripts/upload_aab.py + the play_publisher.json service account exist but are NOT wired into CI — confirmed @aick #1567; see Pass tab.)
  6. Sam updates the app on his phone via the Play Store internal-testing flow once the AAB is uploaded; verify the new build behaves as expected.
  7. Promote to production track in Play Console when stable. PWA cache is not relevant — Capacitor doesn't cache server.url content the way browsers cache PWAs.

Toast nested-modifier >15 (Uber Eats rejection)

  1. Run C:\Users\sam\_count_deep_nested_modifiers.py on AiCk to scan both restaurants' menus. Threshold is depth >12 in my-walker terms (= Uber level >15).
  2. For each flagged item, the dumped trail (saved to C:\Users\sam\_deep_modifier_items.json) shows the exact mg→mo chain that exceeds.
  3. In Toast web UI, navigate the item's modifier tree + delete the shared "Choose your option..," / "Add Jalapenos" group at the bottom of the chain (often shared across all flagged items in a category).
  4. Re-run the scanner to verify 0 violations before re-publishing the DD-prefixed menus to Uber.
