Permissions

Role taxonomy and the audience-gate system that fronts every mutating action in the app. Source of truth: app/services/permissions.py. Last audited 2026-05-28 (samai, docs full-audit pass).

Role taxonomy

Every authenticated session resolves to exactly one role. Role is the source of permission; permissions are NEVER granted individually outside the role-to-permission map (ROLE_PERMISSIONS in app/services/permissions.py).

Trust hierarchy (top-down visibility)

Roles sit in a top-down trust model (Sam-confirmed, directive #1546). Visibility flows downward: a tier sees its own surfaces and everything below it, never above — lower tiers cannot see upper-tier surfaces even to know they exist.

  1. partner — owner tier, wildcard *.
  2. corporate + corporate_chef — all-stores corporate tier.
  3. gm, km, assistant_km, foh_manager — store-management tier (same access level as one another, store-scoped).
  4. expo — kitchen line / orders + KDS.
  5. driver — route-scoped, own bids / payout only.
  6. floor staff — cook / server / busser / host / bartender (minimal set).

The 5 floor-staff roles are shown at the bottom, but their explicit tier placement is likewise pending Sam's confirmation. (Directive "PLACE-ME ROLES.")

How a request is gated

  1. Site gate (before_request) — one global gate in app/web/auth.py intercepts every path not on the EXEMPT_PREFIXES list. A request passes if it carries a keypad session (session.user_id) OR the legacy shared-password session (session.auth_ok, set by /login with EZLIVE_PASSWORD); otherwise it 302s to /keypad-login?next=<path>.
  2. Partner second-factor — /partner/* additionally requires session.partner_auth_ok (the PARTNER_PASSWORD gate at /partner-login), so co-managers can be in the site without seeing Partner data.
  3. Per-route permission check — the route's @requires_permission("<tag>") decorator runs _user_has(g.current_user, tag, store_id). On denial it 302s to /access-denied?need=<tag>&next=<path>. Enforcement is controlled by PERMISSION_ENFORCE — default ENFORCING since 2026-05-13; set PERMISSION_ENFORCE=0 for the dark-launch log-and-permit fallback (used during role-schema migrations).
  4. Per-row store/scope filter — store-scoped roles are confined to their User.store_scope (CSV) at the query layer; multi-store roles (partner / corporate / corporate_chef / prep_manager / driver) have store_scope = None = all stores. Cross-store reads never leak.

Boundary verification (spot-checked at HEAD 012f74e, samai 2026-05-28): the per-route gate genuinely 302s lower tiers to /access-denied (not just a hidden menu) — confirmed against ROLE_PERMISSIONS:

  • anomaly.admin — required by the rules-admin route (anomaly_routes.py:227); granted only in the corporate set (permissions.py:100) + the partner wildcard. So gm/expo/driver hitting it 302 to /access-denied?need=anomaly.admin.
  • developer.view_chat — required by the dev-chat routes (developer_chat.py:129 et al.); granted in corporate (permissions.py:83) + partner only. Store-tier and below are blocked at the route, not merely in nav.
  • labor.view_all_stores vs labor.view_store_summary — the all-stores key sits in corporate (permissions.py:66) + corporate_chef (permissions.py:210); store-scoped roles get only the summary key, enforcing the cross-store read boundary at the query layer.

Permission catalogue

Permissions are keyed strings of the form <area>.<action>, mapped many-to-many to roles via ROLE_PERMISSIONS. Adding a permission means adding the key + role mappings in code; never inline-grant by user-id. partner holds the wildcard * and short-circuits every check.

Representative live keys (verbatim from app/services/permissions.py):

  • orders.assign_driver / orders.unassign_driver / orders.mark_delivered — order & driver actions.
  • drivers.admin / drivers.reset_passcode / drivers.view_roster — driver administration (opened to all non-driver roles, Sam #1063).
  • manager_log.write / manager_log.read_own_store / manager_log.read_all_stores — manager daily-log.
  • produce.order / produce.invoice_verify — produce ordering & receiving.
  • labor.view_store_summary vs labor.view_all_stores — store-scope vs all-stores split.
  • team_reports.view vs team_reports.view_all_stores — team-reports tab (gm store-scoped) vs cross-store (corporate+).
  • kds.view_kitchen_display / kds.edit_recipes — kitchen display & recipes.
  • anomaly.admin / access.team_admin — partner/corporate-tier config surfaces.
  • ai.ask_claude / ai.ask_claude_personal / developer.view_chat — AI & dev-chat.

The full live list is enumerated in app/services/permissions.py (the source of truth); the Operations tab covers day-to-day permission additions. Scheduling keys (schedule.* / shift.* / time_off.* / availability.* / announcement.* / message.*) are NOT yet present as of HEAD — pending the scheduling build.

Audience-eligibility-before-mutation

This is the strictest rule. Every mutating action must check audience BEFORE the database mutation:

  @bp.route("/some/mutator", methods=["POST"])
  def some_mutator():
      # 1. Audience gate first
      if not has_permission("area.action"):
          return redirect(url_for("auth.access_denied", need="area.action"))
      # 2. Validate payload
      payload = request.get_json() or {}
      ...
      # 3. THEN mutate
      with SessionLocal() as db:
          ...

A failure of step 1 must leave zero rows. Audience checks AFTER a mutation are treated as a vulnerability, not a style issue. (has_permission() is the in-handler check; requires_permission() is the decorator form — both in app/services/permissions.py.)

Operator-only (Sam) surfaces

The /sam/* family — /sam/chat, /sam/combined, /sam/agents, /sam/pass, /sam/docs, /sam/automation, /sam/cena — is gated by is_sam_chat_user() (app/web/sam_chat.py): a direct match between the authenticated user's id and the single SAM_CHAT_USER_ID env var, NOT via the role-to-permission map. The capability cannot be role-inherited; only the one configured user id has access, and when SAM_CHAT_USER_ID is unset the surface is closed for everyone (safe-closed dormant).

Public / unauthenticated

Only paths on EXEMPT_PREFIXES (app/web/auth.py) bypass the session gate — each carries its own auth where it mutates:

  • Auth surfaces — /login, /logout, /keypad-login, /keypad-logout, /change-passcode, /partner-login. (/access-denied is NOT exempt — it requires a session.)
  • /produce/healthz — public liveness probe (NOT /healthz).
  • Static & public pages — /static/, /favicon.ico, /install, /privacy, /driver/app.apk, /request-access.
  • Signed / token-gated machine endpoints — /ezcater/webhook (HMAC), /orders/ingest* (token), /cron/* (CRON_TOKEN), /sam/cena/* (X-Cena-Token), /docck/* (bearer), /produce/admin/* (bearer), /produce/confirm|cancel/* (random order-id), /ez-manage/pending-count.json (returns 401 JSON, not 302).
