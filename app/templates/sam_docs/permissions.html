Permissions

Role taxonomy and the audience-gate system that fronts every mutating action in the app.

Role taxonomy

Every authenticated session resolves to exactly one role. Role is the source of permission; permissions are NEVER granted individually outside the role-to-permission map.

RoleAudienceLogin path
partnerInternal staff with partner-dashboard access.Site password + Partner password gate.
corporateCorporate / payroll role.Site password + Corporate password gate.
gmGeneral manager (per store).Manager keypad-login.
expoExpo / kitchen line role.Manager keypad-login.
driverDelivery driver.Driver login flow.
operatorSystem owner; /sam/* surfaces.SAM_CHAT_USER_ID env match against the authenticated user.

How a request is gated

  1. Site-password gate (EZLIVE_PASSWORD) — intercepts all paths that aren't on the exempt list (/healthz, the keypad-login itself, static assets). Returns a 302 to the gate if not present.
  2. Keypad-login gate — paths under /partner/*, /<store>/*, /sam/* require an authenticated user session. Unauthenticated requests redirect to /keypad-login?next=<path>.
  3. Per-route permission check — once authenticated, the route's @requires_permission("<cap>") decorator (or an inline gate) runs. A denial returns a 302 to /access-denied?need=<cap>.
  4. Per-row audience filter — for list views and reports, the rows themselves are filtered by the user's store / role scope. Cross-store reads never leak.

Permission catalogue

Permissions are keyed strings of the form <area>.<action>. They map many-to-many to roles via ROLE_PERMISSIONS. Adding a new permission means adding the key in code + the role mappings; never inline grant by user-id.

Indicative permission keys:

  • team_reports.view — see the team-reports surface.
  • manager.daily_log.write — log into the manager daily-log surface for a store.
  • vendor.recent_orders.view — vendor recent-orders dashboard.
  • catering.dashboard.view — catering dashboard.
  • operations.dashboard.view — operations dashboard.
  • recipes.view / recipes.write — recipe page R/W.
  • driver.live_tracking.view — driver live-tracking map.

The full live list is enumerated in app/services/permissions.py (the source of truth); the Operations tab covers day-to-day permission additions.

Audience-eligibility-before-mutation

This is the strictest rule. Every mutating action must check audience BEFORE the database mutation:

  @bp.route("/some/mutator", methods=["POST"])
  def some_mutator():
      # 1. Audience gate first
      if not has_permission("area.action"):
          return redirect(url_for("auth.access_denied", need="area.action"))
      # 2. Validate payload
      payload = request.get_json() or {}
      ...
      # 3. THEN mutate
      with SessionLocal() as db:
          ...

A failure of step 1 must leave zero rows. Audience checks AFTER a mutation are treated as a vulnerability, not a style issue.

Operator-only (Sam) surfaces

The /sam/* family — /sam/chat, /sam/combined, /sam/agents, /sam/pass, /sam/docs — is gated on a direct match between the authenticated user's id and the SAM_CHAT_USER_ID env var, NOT via the role-to-permission map. This is intentional: the capability cannot be role-inherited; only the explicitly configured user id has access. When SAM_CHAT_USER_ID is unset, the surface is closed for everyone (safe-closed dormant).

Public / unauthenticated

Only the following paths bypass authentication:

  • /healthz — liveness probe.
  • /keypad-login, /access-denied — the login + denial surfaces themselves.
  • Static assets (CSS, fonts, images).
  • Signed webhook endpoints (e.g. EZCater inbound), gated by signature verification instead of session auth.
