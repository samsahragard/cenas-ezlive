The current build-cycle master direction. What is being built and why.

Phase 2 in one sentence

Phase 2 turns the Cenas Kitchen app from a reporting surface into an operating system — manager-actionable tools, an operator-side AI assistant, and the integrations that feed both.

Block 1 — Foundation (shipped)

  • Preconditions: Role Taxonomy spec; ScheduledEvent spec.
  • 1A — Task System: tasks model, escalation policy, audit log.
  • 1B — Ribbon Component: the shared in-page ribbon control.

  • 1H — Pay Masking: privacy controls on driver pay surfaces.

Block 1 is closed and verified; amend only via formal spec amendment citing commit SHA.

Block 2 — Manager Tools (partial)

  • 2I — Recipe Page Shipped: manager-facing recipe reference per store.
  • 2A–2H, 2J: spec slots open; work begins as the operator opens them.

Block 3 — Agent Surface (future)

The in-app assistant for managers — distinct from the operator-only chat. Block 3 opens after Blocks 1 and 2 close. No live specs yet; the agent_surface is currently used internally for testing tool wiring.

Cross-block primitives

  • Permission system: role-to-permission map gates every mutating action (see Permissions tab).
  • Tab-strip dashboards: Today / Manager / Catering / Operations / Vendors share the tabbed-iframe pattern so each tab embeds the real working page.
  • Per-store scoping: every store-scoped route is prefixed /<store>/; the store-blueprint resolves the store at request time.
  • Deploy gates: every commit runs the three-gate verification (see Methodology tab).

Open work — current focus

  • Operator-side AI assistant: /sam/chat + the AI gateway feeding the operator surface.
  • Cena page tab suite: Cena chat / Cena + Dev / Agents / Page Info / Pass / Docs (this surface).
  • Vendor email-parse pipeline: ingest WebstaurantStore equipment and warranty emails.
  • Cross-host message queue: the secondary peer host wires into the primary for resilient delivery.
  • Anomaly engine extensions: additional rules for the daily / weekly brief.
