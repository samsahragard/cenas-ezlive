Documentation for the Cenas Kitchen system. Project-level reference — operations, architecture, integrations, and engineering standards.

What this is

The Cenas Kitchen platform is the partner-facing dashboard and operations system for the restaurant. It runs at app.cenaskitchen.com and integrates with Toast (POS), EZCater (catering intake), and several vendor channels. Scheduling currently uses Sling, which is being replaced by an internal scheduling system (see Phase 2 Roadmap). This Docs surface is the single index of how the system is built and how it operates.

If you're new, read these six first

  1. Architecture — the high-level system picture: services, data flow, where things live.
  2. Methodology — the engineering standards: how changes ship, how they are verified, what counts as "done".
  3. Permissions — the role + permission model gating every surface of the app.
  4. Operations — the day-to-day runbook: opening, closing, anomalies, manager workflows.
  5. Phase 2 Roadmap — what is currently being built and why.
  6. Cena Surface — the operator-only AI assistant: what it does, what it touches.

Status conventions

  • Active — work currently in progress; subject to change.
  • Shipped — code shipped and verified; amend only via formal spec amendment.
  • Reference — long-lived operational doc; updated as state changes.
  • Draft — spec authored, implementation not yet started.

Editing protocol

This Docs surface is read-only at runtime. Content updates ship via git commit + Render auto-deploy, never via an in-app edit form. Secrets (tokens, passwords, GUIDs) are never recorded here — they are referenced by env-var name or secrets-file path only. Credential locations live on the Pass tab, not here.
