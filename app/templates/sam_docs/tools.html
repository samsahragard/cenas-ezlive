ezCater Driver Re-Assignment

Status: Live in production. Verified 10/10 by aick (primary) and 10/10 by ck (backup) on 2026-05-24.

Purpose

When a manager picks a new driver from the dropdown on the catering Ez Orders page, the system unhooks ezCater's default driver (Sam #2 / Masood #1) and assigns the chosen driver, then verifies the change actually landed by re-reading the order page.

Trigger

Manager picks a driver from the per-order dropdown on /<store>/catering Ez Orders page.

How it works (plain English)

You pick a driver. The dropdown grays out and shows "assigning [name]...". The system opens a real Chrome browser somewhere on your network, drives the ezCater website like a person would, unhooks the current driver, assigns your pick, then re-reads the page to confirm it worked. Green check if it landed, red X if not.

By default this runs on aick (Mini_IT12). If aick fails twice with a fresh browser, ck (Mini_IT13) takes over. The two run completely different software stacks (Selenium+Edge vs Playwright+Chromium), so a problem on one usually doesn't affect the other.

Architecture (three tiers)

[Tier 1: Frontend]
Manager's browser -> catering Ez Orders page -> dropdown change
        v POST /<store>/catering/assign_driver

[Tier 2: Control Plane - Render web service]
Creates DriverAssignmentJob row (status=pending)
HTTP-pings AiCk via SOCKS5 proxy through Tailscale
        v POST http://100.108.119.19:8765/jobs/driver-assign

[Tier 3: Execution Plane - AiCk gateway]
Spawns daemon thread -> opens Chrome -> drives ezCater UI
Verifies via DOM re-read after fresh page load
        v POST https://app.cenaskitchen.com/catering/assign_driver/result

[Back to Control Plane]
Flips job status to completed/failed
Frontend polling sees the update -> flashes check or X

If aick's execution fails twice with fresh browser contexts, it forwards the job to ck's gateway at the same /jobs/driver-assign endpoint shape. ck runs its own independent flow on Mini_IT13.

Two independent implementations (primary + backup)

The same flow is built two different ways on two different machines, so a failure on one stack doesn't take the whole tool down.

Primary — aick on Mini_IT12

  • Browser engine: Selenium WebDriver + Microsoft Edge
  • Verification method: DOM re-read of the "Assigned Driver" field after fresh page reload
  • Cycle time: ~23–25 seconds end-to-end
  • Code location: app/services/ezcater_driver_assigner.py + cena_gateway.py /jobs/driver-assign
  • Profile dir: C:/Users/sam/ezcater_assigner_chrome_profile

Backup — ck on Mini_IT13

  • Browser engine: Playwright + Chromium
  • Verification method: Reads the Apollo GraphQL cache embedded in the page HTML (DeliveryDriverAssigned:<uuid>:{fullName:...}) — faster and more deterministic than DOM scraping
  • Cycle time: ~17 seconds end-to-end
  • Code location: C:/Users/sam/ck-ezcater/ck_assigner.py
  • Profile dir: Playwright's per-context isolated storage

Why two stacks matter

  • If Edge updates and breaks Selenium, ck still works.
  • If Chromium gets a quirk that breaks Playwright, aick still works.
  • If both happen at once, both stacks need fixing — but that's extremely unlikely simultaneously.
  • The two are intentionally different. Don't "harmonize" them.

The 12-step flow (same logic, both stacks)

For any driver re-assignment, regardless of which machine runs it:

  1. App-side unhook — clear our DB's ezcater_driver_name for the order. Best-effort ezCater API call to free the portal field.
  2. Open order page — drive browser with authed ezCater session cookies to partnerportal.ezcater.com/orders/<numeric_id>. Resolve numeric ID from external order ID via cached lookup.
  3. Auth check — if the URL redirects to a sign-in page, session is expired. Stop everything and post URGENT alert to dev chat.
  4. Dismiss popups — scan for [role=dialog] elements with close/dismiss/got-it/ok/no-thanks affordances. Try to close them within 3 seconds. If they won't dismiss, trigger fresh-context retry.
  5. UI unassign — if page still shows an assigned driver, click "Unassign driver" button.
  6. UI confirm — if "Yes, unassign" modal appears, click it. (Also handles variants: "Unassign", "Confirm", "Yes", "OK".)
  7. Open pick-driver modal — click "Assign in-house driver" button. (Also handles variants: "Assign driver", "Change driver".)
  8. Type driver name — find the search input in the modal, type the new driver's name. Wait ~1.2s for client-side filter to apply.
  9. Click matching name — iterate visible labels, normalize each (strip CK #N / CK#N / Ck#N kitchen prefixes via regex), tokenize, case-fold, check for subset match against the target driver name. First match wins. JS-click fallback if standard click fails.
  10. Submit — find the modal's "Assign driver" submit button, click it.
  11. Error dialog watch — wait 3 seconds. If a dialog appears with text matching "error assigning the driver", mark as verification_mismatch (this catches ezCater's silent cross-kitchen rejection where they show an error modal but the page state doesn't change).
  12. Verify — fresh GET on the order page (don't trust the modal closure). Wait for document ready. Either re-read the "Assigned Driver" DOM field (aick) OR parse the Apollo cache JSON embedded in the HTML (ck). Compare to the picked driver name using the same normalizer from step 9. Match = success. Mismatch = failure with the actual driver name surfaced as the error.

Concurrency rules

  • Same order, two assignments within 5 seconds: server-side rejection with HTTP 409. The frontend dropdown lock is the primary defense, but server-side enforcement catches double-clicks, race conditions, and any UI bug.
  • Different orders, multiple assignments: run in parallel. Each gateway thread is independent. No shared state beyond the Chrome profile directory (which is fine — each context opens its own browser session).
  • Recommendation if scaling to 5+ concurrent: use per-thread profile dirs to avoid any shared-cookie weirdness. Currently the system has been tested up to 10 sequential cycles, not yet stressed with high parallel load.

Security posture

  • Token gate: X-Cena-Token shared secret, both directions on every HTTP hop. Rotate by updating cena_token.txt on AiCk + bouncing gateway + updating CENA_GATEWAY_TOKEN env on Render.
  • Callback exemption: /catering/assign_driver/result is in EXEMPT_PREFIXES in auth.py (own token check inside handler) so the gateway's callback bypasses the keypad auth. Other two endpoints inherit @requires_store_access (Sam session OR partner_auth_ok).
  • Selenium isolation: Profile directory reused intentionally across retries (cookies persist). Wipe via rm -rf <profile_dir> between major deploys if a cookie state goes weird.
  • No secrets in dev chat: ezCater password, Anthropic API key, GitHub PAT, and Render API key are all on disk in .secrets/ directories — never pasted into dev chat or committed to the repo.

Known limitations

  1. Same-kitchen only. ezCater blocks cross-kitchen assignments (CK#1 driver on CK#2 order or vice versa). System catches this as verification_mismatch. Manager must pick from the order's own kitchen roster.
  2. Driver must exist in ezCater first. This tool doesn't add drivers — it only reassigns existing ones. If a name isn't on the ezCater roster, the modal won't filter to it, and the assignment fails with driver_name_not_found.
  3. Session expiry is manual. When ezCater cookies expire (every few weeks), Sam must log in manually via Chrome on AiCk to refresh the cookie jar. There's no auto-refresh because ezCater requires SSO+MFA.
  4. Concurrent load not yet stress-tested. Verified up to 10 sequential cycles per machine. Untested at high parallelism. If scaling, use per-thread Chrome profile directories.

Three root causes that were debugged during build

Worth knowing for whoever maintains this — these were not obvious and burned real time to find:

  1. The type-and-click bug. Initial code typed the driver name in the modal search box but never clicked the filtered result. ezCater needs BOTH actions: type to filter, then click the matching label. Type alone doesn't select.
  2. The silent cross-kitchen rejection. ezCater shows an error dialog when you try to assign a CK#1 driver to a CK#2 order (or vice versa), but the modal closes normally. Without explicit dialog text detection, the system thought the assignment succeeded and marked it verification_mismatch on the DOM re-read. Step 11 (error dialog watch) now catches this explicitly.
  3. The Render-can't-reach-AiCk bug (worst — took 1 hour). Two compounding issues:
    • from selenium import webdriver at module top level. Render doesn't install Selenium, so the module's import failed silently and dispatch_assignment_job was never callable. Jobs sat pending forever. Fix: lazy try/except around Selenium imports.
    • Render's outbound to AiCk's 100.108.x.x Tailscale IP requires routing through Render's userspace tailscaled SOCKS5 proxy at CENA_PROXY=socks5h://localhost:1055. Plain urllib.request ignores the proxy and the packet drops silently. Fix: switched to httpx.Client(proxy=CENA_PROXY).post(...) matching the pattern sam_chat.py already uses for the cena_sam_chat mirror.

Extension template — building the next ezCater UI automation

This same pattern works for any other ezCater UI action (changing delivery time, updating customer info, rescheduling, etc.). The five pieces:

  1. Job table. New $thing_jobs SQLAlchemy model + migration. Boot-time create_all backfill.
  2. Frontend trigger. Button / dropdown / form on the relevant dashboard page. POST + lock + poll status every 2s + flash green/red.
  3. Render-side dispatch. Three endpoints: POST /catering/$thing (creates job, 5s same-resource guard, dispatches), GET /catering/$thing/status (polling), POST /catering/$thing/result (callback from gateway, token-gated, in EXEMPT_PREFIXES).
  4. Gateway endpoint. /jobs/$thing on cena_gateway.py (aick) AND on ck's gateway. Both accept payload + spawn daemon thread + run inline flow.
  5. Inline flow. $thing_inline_flow in app/services/ezcater_$thing.py mirroring run_assignment_flow_inline:
    • Lazy Selenium/Playwright import
    • Same exception taxonomy (_AuthExpired, _PopupStuck, _AssignmentMismatch, generic)
    • Same retry policy (max 2 fresh-context, optional failover to ck)
    • Same observability (dev chat start/success/failure, logger per step)

After the 5 pieces:

  • Reuse CENA_GATEWAY_URL / CENA_GATEWAY_TOKEN / CENA_PROXY envs (no new envs needed).
  • Bounce cena_gateway.py on AiCk and ck's gateway to pick up the new endpoint.

Operator quick reference (for Sam)

  • Daily check: open catering dashboard. Look at Ez Orders cards. Every order should show a real driver name (not Sam #2 or Masood #1). If you see fallback names still, the cron didn't catch them — open the dropdown and pick manually.
  • When something looks stuck: if a dropdown has been spinning "assigning..." for more than 90 seconds, refresh the page. If it still looks stuck, check dev chat — there should be a failure message with the reason.
  • When ezCater login expires: you'll see "session expired" alerts in dev chat. Log into partnerportal.ezcater.com manually in Chrome on AiCk. Once logged in, post in dev chat "session refreshed" — aick will re-extract the cookies and resume.
  • To manually trigger a re-assignment outside the dropdown: tell aick or ck in dev chat: "reassign order [ID] from [current driver] to [new driver]." They run the same flow manually.
  • To pause the whole tool: tell aick "pause driver auto-assign." He'll stop the worker thread. Dropdown stops responding. Resume with "resume driver auto-assign."
